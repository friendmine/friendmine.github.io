# 第110章 大模型推理的关键瓶颈

## 先说结论

大模型推理常见的误解，是把问题简单理解成“算不动”。
而现实里，卡住系统的往往是带宽、显存、KV cache、通信、调度和单位服务成本共同形成的系统瓶颈。

推理系统真正难的，不只是完成一次前向计算，而是在大量真实请求中持续做到低延迟、高吞吐、可扩展、成本可控。

![插图 第110章 推理瓶颈四象限交通图]({{ site.baseurl }}/ai_ecosystem_book/images/chapter_110_true_bottlenecks_of_large_model_inference.png)

> 插图 第110章 推理瓶颈四象限交通图

第109章已经区分了训练集群和推理集群：训练在制造能力，推理在交付能力。
本章继续往下拆：推理集群在交付能力时，真正会被哪些瓶颈卡住。

---

## 1. “算力不足”当然存在，但它远远不是完整答案

当模型越来越大，最自然的解释是：算力不够。
这个判断有时成立，但过于粗糙。

现实推理系统里，很多问题并不是乘法次数本身造成的，而是来自更具体的环节：

- 参数要不断从显存读到计算单元
- KV cache 会随着上下文和并发增长
- decode 阶段通常难以把计算单元喂满
- 多卡切分会引入通信和同步
- 长短请求混跑会造成调度冲突
- batch 过小浪费吞吐，batch 过大又恶化时延

所以，“算不动”往往只是表面描述。
继续往下拆，常见问题其实是搬不动、放不下、排不好、同步太慢、单位成本过高。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/110_true_bottlenecks_of_large_model_inference_deep_revised__mermaid_01.png)

> 图110-1 为什么“算不动”往往只是表面描述
> 很多推理问题都会被先描述成算力不足，但继续往下拆，卡住系统的常常是带宽、容量、缓存和调度。这张图先把这种误判拆开。

这也是为什么推理优化不能只盯芯片峰值 FLOPs。
FLOPs 是上限的一部分，但推理服务的真实表现，还取决于数据怎样流动、缓存怎样管理、请求怎样排队、资源怎样复用。

---

## 2. Prefill 和 decode 是两种完全不同的压力

大模型推理通常可以拆成两个阶段：prefill 和 decode。

Prefill 阶段处理已有上下文。
它一次性读入 prompt，把上下文转成内部状态，并生成后续 decode 需要的 KV cache。
这个阶段计算量大，矩阵运算更密集，通常更容易把硬件计算单元用起来。

Decode 阶段逐 token 生成。
每生成一个新 token，都要读取模型参数、访问已有 KV cache、写入新的 KV cache，并继续下一步。
这个阶段每步计算相对小，但需要频繁访存，常常更容易受到带宽和缓存压力限制。

可以粗略理解为：

- prefill 更像一次大批量处理，偏计算密集
- decode 更像连续小步生成，偏访存和调度密集

这一区分非常关键。
很多系统在 prefill 阶段看起来性能不错，但到了 decode 阶段吞吐和延迟就开始变差。
原因不是模型突然变复杂，而是工作负载形态变了。

推理系统如果不区分 prefill 和 decode，就很容易用错误指标判断性能。
一个平台 prefill 快，不代表在线生成体验一定好。
一个平台 decode 稳，也不代表长 prompt 首 token 延迟一定低。

---

## 3. 带宽常常比 FLOPs 更早成为瓶颈

在大模型推理中，计算单元很强并不意味着系统一定快。
如果数据供不上，计算单元就会等待。

尤其在自回归 decode 阶段，每一步都需要：

- 读取模型参数
- 读取历史 KV cache
- 写入新的 KV cache
- 处理当前位置的计算
- 根据调度策略和其他请求共享资源

如果带宽跟不上，系统表现出来就像“算力不足”。
但本质上，是计算单元在等数据。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/110_true_bottlenecks_of_large_model_inference_deep_revised__mermaid_02.png)

> 图110-2 大模型推理的关键瓶颈图
> 推理性能并不是由 FLOPs 单独决定，而是被参数访存、KV cache、带宽、通信和调度一起塑形。很多时候，先卡住系统的不是算不动，而是搬不动。

一个最小直觉式可以写成：

`单步推理时间 ≈ 计算时间 + 参数读取时间 + KV cache 访问时间 + 通信时间 + 调度等待`

只要后面几项足够大，计算本身再快也无法直接转化为用户体验。

这也是 roofline 模型给我们的核心提醒：
系统性能不仅取决于峰值计算能力，还取决于算术强度和内存带宽。
如果一个负载每搬一字节数据只做很少计算，它就更容易成为 memory-bound。
许多 decode 工作负载正是这样。

---

## 4. KV cache 是长上下文和高并发背后的真实压力源

大模型推理不是只把参数放进显存就结束。
真正进入服务后，KV cache 会迅速成为核心资源。

KV cache 的作用，是保存历史 token 的 key/value 状态，避免每生成一个新 token 都重新计算全部上下文。
它提升了生成效率，但也把上下文长度、并发和显存压力紧紧绑在一起。

可以用一个粗略关系式帮助理解：

`KV cache 开销 ∝ 层数 × 序列长度 × batch × hidden size × 精度字节数`

严格实现会受到 head 数、GQA/MQA、分片方式和数据格式影响，但方向很清楚：
上下文越长、并发越高、模型越大，KV cache 压力越快膨胀。

一旦容量吃紧，后果不只是“模型放不下”：

- 并发数下降
- 长上下文被限制
- 请求被迫排队
- cache 被频繁迁移或释放
- offload 带来明显延迟
- 多租户场景更难隔离资源

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/110_true_bottlenecks_of_large_model_inference_deep_revised__mermaid_03.png)

> 图110-3 容量为什么直接改写服务质量
> 容量问题不只是“装不装得下模型”。一旦显存和缓存空间紧张，并发、上下文长度和数据搬运都会被一起拖慢，所以它会直接变成服务质量问题。

这就是为什么长上下文能力不能只看模型标称支持多少 token。
系统真正要回答的是：在真实并发、真实延迟目标和真实成本约束下，能稳定服务多少长上下文请求。

---

## 5. Batch 能提高吞吐，也会制造时延和资源冲突

推理服务常用 batch 提高硬件利用率。
把多个请求合并在一起，可以让矩阵运算更饱满，减少计算单元空转。

但 batch 不是越大越好。
它会带来几个问题：

- 请求需要等待凑批
- 长请求和短请求互相拖累
- 不同上下文长度导致 padding 或调度浪费
- KV cache 占用更大
- 首 token 延迟可能变差
- 尾延迟更难控制

在线推理不是离线批处理。
它既要吞吐，也要响应。
因此，推理系统常常需要 continuous batching、动态 batching、请求分组、优先级队列、抢占和回退策略。

这里的核心矛盾是：

- batch 小，延迟可能好，但硬件利用率低
- batch 大，吞吐可能高，但用户等待和资源占用上升

所以，调度不是服务层小问题。
它直接决定硬件能力能否变成可接受的用户体验。

---

## 6. 多卡推理不会自然线性变快

很多大模型必须跨卡甚至跨节点运行。
但把模型切到更多设备上，并不意味着性能会按比例提升。

多卡推理会引入新的成本：

- 张量并行带来的通信
- pipeline 阶段之间的等待
- KV cache 分布后的访问成本
- 请求在不同设备间协调
- 网络拓扑不匹配导致的额外延迟
- 小 batch decode 阶段的通信占比上升

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/110_true_bottlenecks_of_large_model_inference_deep_revised__mermaid_04.png)

> 图110-4 为什么多卡推理不会自然线性变快
> 很多系统一扩卡就会发现，新增硬件并没有按比例换来体验提升。原因往往不是卡不够强，而是通信、切分和协调开始吞掉扩展收益。

训练集群和推理集群在这里也有明显差异。
训练可以更强调大吞吐和长时间并行效率。
推理还要关心每个用户请求的端到端时延。
一次跨卡同步如果发生在热路径上，就可能直接影响用户感知。

因此，多卡推理的关键不是“能不能切”，而是“切完以后通信是否划算，延迟是否稳定，资源是否还能高效复用”。

这也自然引向第111章。
很多切分、layout、通信重叠和 kernel 选择问题，最后都会落到编译器、运行时和执行计划上。

---

## 7. 缓存策略决定系统是在复用资源，还是被资源拖住

推理系统里有多种缓存：

- KV cache
- prompt cache
- prefix cache
- embedding 或检索缓存
- 编译后 kernel 或执行计划缓存
- 热门模型和 adapter 缓存

缓存的价值，是避免重复工作。
但缓存也会占据宝贵资源。
如果没有策略，缓存会从加速器变成负担。

关键问题包括：

- 哪些 cache 值得保留
- 哪些 cache 应该尽快释放
- 长上下文请求是否会挤占短请求资源
- prefix 能否被多个请求复用
- 不同租户之间 cache 是否隔离
- cache 命中率是否真的覆盖了占用成本

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/110_true_bottlenecks_of_large_model_inference_deep_revised__mermaid_05.png)

> 图110-5 推理瓶颈的四类系统压力
> 推理体验不是由单一瓶颈决定的。显存决定能装多少上下文和并发，带宽决定数据能不能喂得上，通信决定多卡扩展是否划算，调度则决定这些资源是否被组织成了可用服务。

把四类压力说得更直接一点：

- `显存 / 容量压力` 决定模型、KV cache 和并发请求能不能同时被容纳
- `带宽压力` 决定参数和缓存能不能及时送到计算单元面前
- `通信压力` 决定多卡、多节点协作会不会把收益吃掉
- `调度压力` 决定长短请求、不同优先级和不同模型是否会互相拖累

这些压力会相互影响。
比如，为了提高吞吐增大 batch，可能让 KV cache 压力上升；
为了支持长上下文，可能降低并发；
为了跨卡放下模型，可能引入通信瓶颈。

推理优化的难点，正是这些目标不能孤立最优。

---

## 8. 首 token 延迟、吞吐和尾延迟经常互相拉扯

用户体验不是一个数字。
推理服务至少要同时看几类指标：

- TTFT：首 token 延迟
- TPOT：每个输出 token 的平均时间
- 吞吐：单位时间生成多少 token 或完成多少请求
- 尾延迟：P95、P99 等高分位延迟
- 成本：每次请求或每百万 token 的服务成本

这些指标之间常常存在张力。

为了提高吞吐，系统可能增加 batch。
但用户可能要等更久才开始生成。

为了降低首 token 延迟，系统可能减少凑批等待。
但硬件利用率可能下降，单位成本上升。

为了支持长上下文，系统要保留更多 KV cache。
但这会压缩并发能力。

为了降低尾延迟，系统可能需要预留更多冗余资源。
但这又会增加成本。

所以，推理系统不是单纯追求“最快”。
它是在不同服务目标之间做工程权衡。
成熟系统需要按业务优先级设计调度策略，而不是只追一个 benchmark 指标。

---

## 9. 大模型推理是一场全栈协调战

你无法只靠模型优化解决推理问题，也无法只靠更换更强芯片解决推理问题。
有效系统必须同时处理：

- 模型结构
- 数值格式和量化
- attention 与 KV cache 机制
- kernel 和编译器优化
- 显存管理
- 请求调度
- 多卡通信
- 服务等级和降级策略
- 端边云部署位置
- 单位服务成本

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/110_true_bottlenecks_of_large_model_inference_deep_revised__mermaid_06.png)

> 图110-6 自回归推理为什么像一场数据搬运战
> 在大模型推理里，计算不是孤立发生的。每个 token 生成都伴随参数、缓存、通信和调度成本，所以全栈系统协调才会比单点算力数字更重要。

很多推理瓶颈最终会变成系统选择题：

- 高价值请求是否优先
- 长上下文请求是否限流
- 低优先级任务是否降级到小模型
- 是否用 speculative decoding 换取更高吞吐
- 是否用量化牺牲一点精度换成本
- 是否把部分任务下沉到端侧或边缘

这些都不是单个算子问题，而是系统治理问题。
推理系统越接近真实业务，越需要把技术指标和服务策略放在一起看。

---

## 10. 工程现实：最难的常常是长上下文、混合负载和成本线

现实系统里，让团队头疼的往往不是单个干净 benchmark，而是混合负载。

常见痛点包括：

- 长上下文请求拉低整体吞吐
- 短请求被长请求阻塞
- 多卡扩展收益迅速下降
- 批处理提升吞吐却恶化时延
- KV cache 被少数请求长期占用
- 不同模型或 adapter 混部后资源碎片化
- 峰值流量导致大量冗余资源长期闲置
- 成本线压迫模型大小和上下文长度

这些问题很少在一张简单的性能表里完整出现，却决定线上服务能不能长期运行。

因此，推理优化如果只盯理论峰值，就很容易在真实环境里受挫。
真正的生产指标，往往是综合性的：
在给定成本和稳定性目标下，系统能服务多少真实用户、支持多长上下文、维持多低尾延迟。

这也是为什么第111章要讨论编译器。
只有当编译器、kernel、layout、内存规划、runtime 和服务调度共同工作，推理瓶颈才有可能被系统性压低。

## 11. 容易误判的地方

### 误解一：推理慢主要因为模型参数太多

参数规模重要，但参数访存、KV cache、带宽、通信和调度经常才是更具体的瓶颈。

### 误解二：只要换更强芯片，推理问题就能解决

更强芯片可以提高上限，但如果带宽、缓存、通信和服务组织没有跟上，收益会被迅速吃掉。

### 误解三：带宽和通信只是基础设施团队的问题

它们会反向影响模型设计、上下文长度、并发策略、部署位置和产品能力边界。

### 误解四：吞吐高就代表用户体验好

吞吐只是指标之一。
首 token 延迟、尾延迟、稳定性和单位成本同样决定真实体验。

## 12. 本章小结

1. 大模型推理的瓶颈常常不在纯 FLOPs，而在带宽、显存、KV cache、通信、调度和成本。
2. Prefill 和 decode 面对不同压力，不能用同一组指标粗暴评估。
3. KV cache 把上下文长度、并发、显存和延迟紧紧绑在一起。
4. 多卡推理不会自然线性变快，通信和协调成本会迅速显形。
5. 推理优化是一场全栈协调战，并自然引向下一章的编译器和执行计划问题。

核心判断：

> 大模型推理难的，不只是纸面上要算多少，而是现实里参数、缓存、请求和设备资源能不能流得动、放得下、排得顺、算得值。
