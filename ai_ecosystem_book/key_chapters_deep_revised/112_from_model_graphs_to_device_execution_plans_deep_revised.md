# 第112章 从模型图到设备执行计划

## 先说结论

模型图到设备执行之间，隔着一整条优化、lowering、划分、布局、内存规划和调度链。
模型跑得好不好，往往取决于这条转换链能否把高层意图稳定翻译成贴近硬件现实的执行计划。

从工程角度看，执行计划不是编译链末端的例行产物，而是模型定义之后的第二次系统设计。

![插图 第112章 排兵布阵图]({{ site.baseurl }}/ai_ecosystem_book/images/chapter_112_from_model_graphs_to_device_execution_plans.png)

> 插图 第112章 排兵布阵图

第111章讨论了 AI 编译器如何面向芯片。
本章继续往下走：编译器和运行时到底怎样把一个模型图，变成设备可以执行、可以调度、可以复用内存、可以跨设备协作的现实方案。

---

## 1. 模型图表达的是计算意图，不是设备执行方式

研究者或工程师写下模型结构时，表达的是高层计算意图。
例如注意力、MLP、归一化、卷积、残差连接、路由、采样和后处理。

但设备需要的不是“这个模型想做什么”，而是更具体的问题：

- 哪些算子要被融合
- 哪些张量需要保留
- 哪些中间结果可以复用内存
- 哪些 kernel 在哪个设备上执行
- 张量采用什么 layout
- 数据什么时候搬运
- 通信和计算能否重叠
- 动态 shape 如何处理
- 失败或资源不足时如何回退

模型图并不直接回答这些问题。
它只是执行计划的原材料。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/112_from_model_graphs_to_device_execution_plans_deep_revised__mermaid_01.png)

> 图112-1 模型图和设备执行之间并不是直接连通
> 研究者写下的是“想做什么”，设备需要的是“怎样排、怎样放、怎样搬、怎样同步”。这条转换链存在与否，往往决定模型能不能在平台上跑出来。

这也是为什么同一个模型，在不同框架、不同编译器、不同硬件和不同部署配置下，性能可能差出很多。
模型图相同，只说明数学结构相同；执行计划不同，现实成本就可能完全不同。

---

## 2. 第一层转换，是把模型图变成可优化的 IR

模型通常先来自 PyTorch、TensorFlow、JAX、ONNX 或其他前端表达。
这些表达适合研究和开发，但不一定适合直接做底层优化。

因此，系统通常会先把模型转换成中间表示，也就是 IR。
IR 的价值不只是“中间格式”，而是把模型变成可分析、可重写、可检查的结构对象。

这一层通常要处理：

- 算子依赖关系
- 张量 shape 和 dtype
- 常量与参数
- 控制流与动态分支
- 前端框架差异
- 目标硬件支持的算子集合

好的 IR 能让后续优化有共同语言。
没有这层，编译器就只能对不同框架、不同算子和不同设备做大量碎片化适配。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/112_from_model_graphs_to_device_execution_plans_deep_revised__mermaid_02.png)

> 图112-2 从模型图到设备执行计划的流程图
> 执行计划不是自动长出来的，而是要经过图编译、lowering、切分、映射、内存规划和调度等一连串决策之后，才能变成硬件可执行、且尽量高效的现实方案。

从出版角度看，这里要避免把 IR 写成枯燥术语。
它真正的意义是：把“模型表达”带进“系统优化世界”。

---

## 3. 图优化先在结构层面减掉浪费

执行转换通常会先进行图级优化。
因为很多浪费在高层结构里就已经存在，如果不先处理，后面会被一层层放大。

常见图优化包括：

- 算子融合
- 常量折叠
- 死节点消除
- 公共子表达式复用
- 冗余 reshape 或 transpose 消除
- layout 相关变换合并
- 推理阶段不必要训练节点删除

这些优化的目标，不是为了让图看起来更简洁，而是为了减少真实执行里的中间结果落地、重复搬运和无效 kernel 调用。

例如，多个小算子如果分别启动 kernel，中间结果反复写回显存，可能会被启动开销和内存带宽拖住。
融合之后，系统可以把一段计算组织成更紧凑的数据流。

不过，图优化也不是越激进越好。
过度融合可能导致 kernel 太大、寄存器压力变高、并行度下降，甚至让后续调度更困难。
因此，图优化不是单纯“合并越多越强”，而是在结构简化和硬件适配之间找平衡。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/112_from_model_graphs_to_device_execution_plans_deep_revised__mermaid_03.png)

> 图112-3 执行计划在分别解决哪四类问题
> 执行计划不是一个动作，而是在处理浪费、分工、摆放、时序和资源约束。若这些问题混在一起，后续调优会变成反复试错。

这一节的关键，是让读者理解：
图优化不是装饰性清理，而是在执行计划真正生成前，先把结构性浪费降下来。

---

## 4. Lowering 把高层算子落到硬件可执行的形式

模型图里的算子通常是高层语义。
例如 attention、layer normalization、matmul、convolution、softmax、top-k、采样策略和各种自定义算子。

设备并不直接理解这些高层概念。
编译器需要把它们 lowering 成更底层、更贴近硬件的数据流、kernel 或指令序列。

Lowering 过程会决定：

- 高层算子拆成哪些底层操作
- 哪些操作可以调用现成 kernel
- 哪些操作需要生成专用 kernel
- 哪些数据块适合 tiling
- 哪些循环要展开或重排
- 哪些内存访问要合并
- 哪些算子适合交给特定硬件单元

这一步是性能开始变真实的地方。
如果 lowering 不贴近硬件，后面再怎么调度，也只能在低质量基础上修补。

例如，同样是矩阵乘法，真正落到设备上时要关心 tile 大小、共享内存使用、向量化、访存连续性、tensor core 或矩阵单元的调用方式。
同样是 attention，落地时要关心 softmax、mask、KV cache、带宽、分块和数值稳定性。

Lowering 的本质，是把高层数学表达转成硬件擅长执行的形式。
这一步越成熟，纸面模型越有机会变成可交付性能。

---

## 5. 划分与 placement 决定哪里本地算，哪里跨设备合作

当模型大到单设备无法承载，或者系统需要多个加速器协同运行时，划分和 placement 就会成为核心问题。

它们要回答：

- 哪些层放在同一设备
- 哪些张量需要跨设备传输
- 哪些地方做数据并行
- 哪些地方做张量并行或流水并行
- 哪些中间结果必须同步
- 哪些操作可以异步推进
- 哪些设备适合承担哪类算子

划分得好，通信会被压缩在可接受范围内，设备能持续工作。
划分得差，系统会在设备之间不停转交数据，理论算力被等待和同步吞掉。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/112_from_model_graphs_to_device_execution_plans_deep_revised__mermaid_04.png)

> 图112-4 划分问题是在平衡本地算和跨设备合作
> 划分不是把图切开那么简单，它实质上是在决定哪里该在本地完成，哪里需要跨设备协作，而这会直接改写通信成本和扩展效率。

在训练集群里，这涉及数据并行、模型并行、流水并行和通信拓扑。
在推理系统里，这涉及模型分片、KV cache 放置、prefill/decode 分离、批处理策略和多租户隔离。
在端边云系统里，这还会进一步扩展成第113章要讨论的跨位置执行：哪些任务留在端侧，哪些交给边缘，哪些集中到云侧。

因此，placement 不是“把任务放到某个设备上”的简单动作。
它是在决定整个系统的分工结构。

---

## 6. Layout 决定数据路径是否顺畅

同一组张量，不同 layout 会带来非常不同的执行后果。
这不是表面格式差异，而是直接影响硬件如何访问数据。

Layout 会影响：

- 访存是否连续
- cache 或片上存储是否有效
- 向量化能否展开
- kernel 是否适配
- 是否需要额外 transpose
- 多设备通信是否需要重排
- 后续算子能否复用同一布局

很多性能问题看起来像计算慢，实际根源是 layout 不当导致额外搬运和重排。
一次 layout 转换本身可能不显眼，但如果出现在热路径里、出现在每层之间、出现在长序列推理中，它就会成为稳定开销。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/112_from_model_graphs_to_device_execution_plans_deep_revised__mermaid_05.png)

> 图112-5 执行计划中的三类关键决策
> 切分回答的是“谁和谁一起干活”，布局回答的是“数据怎么摆”，调度回答的是“大家什么时候动”。很多平台性能差异，都出在这三类决策没有协同好。

所以，layout 不是编译器里的小修小补。
它是在给数据设计道路。
道路设计得差，计算单元再强也会等数据。

---

## 7. 内存规划决定模型是否跑得稳、跑得满

AI 执行常见瓶颈不是“不会算”，而是“存不下、搬不动、复用不好”。
因此，内存规划是执行计划里的主问题。

它要回答：

- 哪些张量需要常驻
- 哪些中间结果可以尽快释放
- 哪些 buffer 可以复用
- 峰值内存会出现在哪一步
- 是否需要重计算来换内存
- 是否需要 offload 到主存、磁盘或其他设备
- KV cache 如何增长和回收
- 多 batch、多序列、多租户时如何隔离资源

好的内存规划能显著降低峰值占用，减少无谓搬运，让原本勉强可跑的模型变成稳定可跑。
差的内存规划则会带来爆显存、频繁 offload、延迟尖峰和不可预测性能。

这里还要注意一个工程事实：
最优算子计划和最优内存计划不一定一致。
某种融合可能减少 kernel 启动，却提高峰值内存。
某种并行方式可能提高吞吐，却让中间状态更难管理。
某种调度可能让计算重叠更好，却让 buffer 生命周期交叉得更复杂。

执行计划难，正是因为这些目标需要被同时平衡。

---

## 8. Schedule 组织的是计算、通信和搬运之间的时间关系

执行计划的价值，不只是把任务列出来。
更重要的是把计算、通信和数据搬运排成尽量顺畅的时间关系。

高效系统通常会努力让：

- 计算和通信重叠
- 数据搬运和前后计算重叠
- 不同 batch 或请求形成流水
- prefill 和 decode 的资源分配更均衡
- 不同设备尽量少空等
- 高优先级任务不被低优先级任务长期阻塞

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/112_from_model_graphs_to_device_execution_plans_deep_revised__mermaid_06.png)

> 图112-6 执行计划组织的是时间关系
> 很多执行问题不是“有没有做完”，而是“计算、通信、搬运之间有没有被排成顺畅时间关系”。这决定的是整体利用率，而不是某一步局部快不快。

这也是为什么 schedule 很难。
它不是简单排序，而是在处理依赖、资源、优先级、内存峰值、通信拓扑和运行时动态变化。

一个计划理论上通顺，实际可能通信爆炸。
一个计划局部 kernel 很快，整体可能没有形成有效重叠。
一个计划吞吐很高，但 tail latency 很差，用户体验仍然不可接受。

所以，schedule 的目标不是“每一步都最快”，而是“整条链最少等待、最少空转、最可预测”。

---

## 9. Runtime 把静态计划带进动态现实

很多执行计划不是一次性生成后就完全固定。
真实系统里，请求长度不同、batch 变化、设备负载变化、显存压力变化、网络状态变化，都会让静态计划面临动态现实。

Runtime 需要处理：

- 动态 batching
- 请求排队和优先级
- KV cache 分配与回收
- 多模型共存
- 资源抢占和隔离
- 失败重试与回退
- 性能指标采集
- 运行时 shape 变化
- 设备热状态和可用性变化

从这个角度看，执行计划不是编译器“吐出一个文件”就结束。
它还需要运行时不断执行、监控、微调和兜底。

这也是第112章与第113章之间的连接点。
一旦模型执行从单设备扩展到端侧、边缘、云侧，runtime 面临的就不只是单机资源调度，而是跨位置、跨网络、跨责任边界的执行组织。

---

## 10. 执行计划是模型世界与设备世界握手的地方

同样一个模型图、同一套硬件，实际表现可能差得很远。
原因往往不在模型定义本身，而在它被怎样安排执行。

执行计划决定了：

- 并行度能否被充分利用
- 内存是否会在错误时刻爆掉
- 通信是否会成为主瓶颈
- layout 转换是否吞掉收益
- 流水线是否真的重叠起来
- 延迟是否稳定
- 平台表现是否可预测

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/112_from_model_graphs_to_device_execution_plans_deep_revised__mermaid_07.png)

> 图112-7 执行计划的四类责任如何汇入结果
> 模型图不会自己变快。决定结果的，是先减掉浪费，再定清分工，随后把数据摆对、把时序排顺，并在运行时把资源管住。

执行计划的主要风险，不是某个局部没有调到极致，而是多个目标互相打架。
图优化减少了高层冗余，却可能让后续 layout 更难。
切分提升了并行度，却可能把通信成本拉爆。
调度试图重叠更多操作，却可能让峰值内存失控。
运行时追求更高吞吐，却可能牺牲尾延迟。

成熟平台的价值，就在于能把这些冲突变成可控工程，而不是让用户在每个模型、每个 batch、每个序列长度上反复试错。

## 11. 工程现实：很多平台性能问题都藏在转换链里

现实里，平台常见问题包括：

- 图优化导致结果不稳定或覆盖场景不足
- lowering 后 kernel 不适配真实硬件路径
- 划分后通信暴涨
- layout 转换过多
- 内存复用不充分
- KV cache 管理粗糙
- 异步流水没有形成有效重叠
- 动态 shape 下性能明显波动
- 同一模型换 batch 或序列长度后表现突然变差

这些问题如果不被系统分析，平台就会陷入一种尴尬状态：
理论性能很好，用户体验一般；纸面结构正确，真实执行处处卡顿。

更严重的是，可预测性会被破坏。
同一个模型今天能跑、明天掉速；换一个输入长度就爆显存；不同 batch 下延迟飘忽；某个版本升级后老模型性能退化。
这些都说明问题已经不只是局部调优，而是执行计划链条没有形成稳定制度。

从产业角度看，芯片平台真实竞争力最后不只看峰值参数。
它还要看这条转换链是否成熟，开发者能否相信平台表现稳定、迁移成本可控、问题可定位。

---

## 12. 容易误判的地方

### 误解一：执行计划主要是编译器自动搞定的小事

自动化很重要，但很多平台仍需要深度工程调优、profile、回归测试和运行时治理。
自动生成不等于自然最优。

### 误解二：模型图本身已经包含主要性能信息

模型图只表达计算结构。
真实性能经常藏在 lowering、layout、内存规划、划分、调度和 runtime 决策里。

### 误解三：只要做了算子融合，执行优化就差不多完成了

算子融合只是其中一环。
过度融合甚至可能带来寄存器压力、内存峰值和调度问题。

### 误解四：单个 kernel 快，整体系统就快

系统性能取决于整条链。
通信、搬运、排队、内存和尾延迟都可能吞掉单点 kernel 优势。

## 13. 本章小结

1. 模型图表达高层计算意图，设备执行计划决定这份意图如何落到真实硬件上。
2. 从模型图到执行计划，需要经过 IR、图优化、lowering、划分、placement、layout、内存规划、schedule 和 runtime。
3. 执行计划同时处理浪费、分工、摆放、内存、时序和动态资源变化，不是单点优化。
4. 很多平台性能问题并不在模型本身，而藏在转换链的细节和稳定性里。
