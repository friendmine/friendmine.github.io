# 第93章 人与 Agent 协作的真实流程

## 先说结论

人机协作的难点，不是能不能多聊几轮，而是能不能把人的模糊意图稳定翻译成目标、约束、上下文、工具、审批、执行、验收和复盘组成的工作链。

协作不是对话更长。
协作是责任链更完整。

![插图 第93章 协作泳道现场图]({{ site.baseurl }}/ai_ecosystem_book/images/chapter_093_real_workflow_of_human_agent_collaboration.png)

> 插图 第93章 协作泳道现场图

第92章讨论不同场景下 Harness 的形态，说明不同任务对象会逼出不同系统骨架。
本章进一步拆开真实协作流程：一个 Agent 要进入工作，不是接收一句指令就开始跑，而是要进入一条可观察、可审批、可接管、可交付、可复盘的协作链。

---

## 1. 只有把协作看成流程，AI 才会从工具变成生产力

很多人对 AI 协作的想象仍然很轻：
人提出需求，系统给出答案，事情结束。

这种想象适合问答，不适合工作。

真实工作通常包含：

- 目标设定
- 约束澄清
- 上下文准备
- 工具和权限配置
- 执行观察
- 关键审批
- 结果验收
- 返工与接管
- 复盘和沉淀

如果系统只能承担中间“生成一段答案”的角色，它当然有价值，但还不是完整协作。

成熟的人机协作，需要把整条工作链显式化。
只有这样，人和 Agent 才不是偶然配合，而是形成可重复、可检查、可改进的合作关系。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/93_real_workflow_of_human_agent_collaboration_deep_revised__mermaid_01.png)

> 图93-1 人与 Agent 协作的真实流程泳道图
> 协作不是一条笔直通道，而是一条带有审批、等待、返工、接管和回交节点的工作链。成熟系统需要把这些节点显式化。

协作被买单的，不是某一次回答，而是流程可重复性。
组织真正关心的是：
这套系统能不能稳定承接任务，而不是偶尔答得漂亮。

---

## 2. 任务入口要把“愿望”压成“可执行目标”

用户最初说出的需求，通常不是工程上可执行的任务。

它可能很模糊：

- 帮我整理一下
- 优化这个方案
- 看看哪里有问题
- 把这个做得更专业
- 帮我推进一下

这些表达对人类协作者来说可以继续追问。
对 Agent 来说，如果直接进入执行，很容易在错误方向上积极行动。

成熟协作流程的第一步，是把愿望压成可执行目标：

- 要完成什么
- 输出物是什么
- 成功标准是什么
- 时间和成本限制是什么
- 哪些信息还缺失
- 哪些动作不能做
- 哪些节点需要人确认

这一步看似慢，实际是在防止后面的高成本返工。

如果目标没有被压实，Agent 的执行越积极，偏差越容易被放大。

---

## 3. 目标和约束必须分开结构化表达

目标回答的是“要去哪里”。
约束回答的是“哪些路不能走”。

很多协作失败，不是因为目标没说，而是约束没有被结构化地带进系统。

同样是写一份报告，影响 Agent 行为的约束可能包括：

- 只能使用内部资料
- 不得修改既有结论
- 必须沿用公司模板
- 不能包含未确认数据
- 需要控制篇幅
- 需要避开某些敏感表述
- 成本或时间有上限
- 交付前必须由谁审批

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/93_real_workflow_of_human_agent_collaboration_deep_revised__mermaid_02.png)

> 图93-2 目标和约束为什么需要分开结构化表达
> 很多协作失败不是目标缺失，而是约束没有进入执行边界。把目标和约束拆开，才能避免系统在错误边界里越努力越偏。

执行型系统常见痛点，是“特别积极地做错事”。
问题往往不是它没有目标，而是约束没有被提到与目标同样高的位置。

目标和约束一旦混在自然语言里，系统会倾向于优先追求完成。
但在真实工作里，“能不能做”“该不该做”“做到什么程度才算合格”，往往比“快点做出来”更重要。

---

## 4. 准备工作现场，是协作里最容易被低估的一步

很多人以为启动 Agent 才是任务开始。
实际更关键的，是启动之前的工作现场准备。

成熟系统在执行前至少要完成：

- 明确目标
- 明确约束
- 装配上下文
- 选择工具
- 设定权限
- 设定审批节点
- 明确交付物格式
- 明确失败后的接管方式

这一步就像为一个人交接工作现场。
如果现场准备混乱，后面的执行质量不会稳定。

很多看似模型推理问题的失败，其实发生在这里：
目标还没有变成可执行对象，权限边界还没明示，材料还没有整理到足以支持本轮任务，系统就已经开始跑。

这会让后面的偏航看起来像模型问题，实际根子是任务现场交接失败。

---

## 5. 上下文装配决定 Agent 是在真实世界里工作，还是在残缺世界里推理

Agent 执行质量很大程度上取决于它看到了什么。

上下文装配不只是把资料塞进窗口。
它要决定：

- 哪些项目背景必须进入
- 哪些历史决策必须保留
- 哪些材料需要引用
- 哪些约束需要置顶
- 哪些状态需要实时更新
- 哪些旧信息已经失效
- 哪些内容必须因权限被隐藏

如果上下文漏了关键约束，Agent 会在残缺世界里推理。
如果上下文太多、太乱、太旧，Agent 会被噪声拖偏。
如果权限过滤不严，Agent 可能看见不该看见的信息。

所以成熟协作流程不会把上下文装配当成提示词技巧。
它会把它当成工作现场管理。

上下文装配越稳定，后面的工具调用、审批、验收和复盘才越有基础。

---

## 6. 工具和权限不是执行细节，而是协作边界

Agent 一旦可以调用工具，就不再只是生成内容。
它开始影响外部世界。

因此，工具和权限必须在协作流程中被显式设计。

系统要回答：

- 本任务需要哪些工具
- 哪些工具只读，哪些可以写入
- 哪些动作需要 dry-run
- 哪些动作需要审批
- 哪些资源对当前用户不可见
- 哪些副作用必须可回滚
- 哪些操作只能由人执行

如果工具暴露过宽，系统会更容易越界。
如果权限过粗，系统会难以进入真实组织。
如果审批节点缺失，自动化会一路滑向不可逆后果。

工具和权限不是协作的后端细节。
它们定义了 Agent 能做什么、不能做什么、什么时候必须停下来等人确认。

---

## 7. 成熟协作不是只看结果，而是持续观察过程

一旦任务进入多步执行，人类不能只在最后看一眼结果。

结果对了，可能只是侥幸。
结果错了，也需要知道错在何处。

因此，成熟协作流程必须把 trace 纳入日常使用。
人需要看见：

- 系统如何理解目标
- 它为什么选择当前步骤
- 它调用了哪些工具
- 工具返回了什么
- 它在哪个节点改了计划
- 它目前卡在哪里
- 它生成了哪些中间 artifact

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/93_real_workflow_of_human_agent_collaboration_deep_revised__mermaid_03.png)

> 图93-3 过程可见性为什么是协作前提
> 一旦任务进入多步执行，单看结果就不够了。过程可见不是为了看热闹，而是为了让人知道系统为什么这么做、哪里出了问题、接手后从哪里继续。

过程可见不是为了满足好奇心。
它是纠偏、接管、验收和复盘的前提。

没有过程可见性，协作会退化成等待黑箱给出结果。
这不适合高价值任务。

---

## 8. 审批节点是协作中最有价值的减速带

不是所有步骤都值得人工确认。
但高价值流程里，一定存在需要人明确点头的节点。

例如：

- 对外发布之前
- 修改关键文件之前
- 提交代码之前
- 调用付费资源之前
- 进入不可逆操作之前
- 正式提交结论之前
- 触碰敏感数据之前

审批的价值，在于它让自动化不会一路滑向不可逆后果。

一个没有关键审批节点的协作系统，表面上更流畅，实际是把最大成本留给了出事之后。

好的审批节点不是到处拦。
它应该围绕风险、成本、权限和不可逆性设置。
该快的地方快，该停的地方必须停。

---

## 9. 等待、返工和接管必须成为显式节点

真实工作很少是一条直线。

很多任务需要等待：

- 等人补材料
- 等审批
- 等外部系统返回
- 等测试完成
- 等客户确认

很多任务也会返工：

- 目标变了
- 约束新增
- artifact 被打回
- 结果不符合标准
- 外部条件发生变化

还有一些任务需要接管：

- 系统偏航
- 工具失灵
- 权限不足
- 价值判断冲突
- 高风险动作需要人执行

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/93_real_workflow_of_human_agent_collaboration_deep_revised__mermaid_04.png)

> 图93-4 等待、返工、接管为什么需要被显式化
> 真实工作不是一条直线。把等待、返工和接管做成系统结构，协作链才不是脆弱的理想流程，而是真能承接现实变化的工作链。

这些节点看起来让流程变长，实际上是在把隐形风险变成显性结构。
少了它们，流程也许更短，但真实复杂度会被藏到事故和返工里。

---

## 10. Artifact 验收，才算把事情做完

很多系统在任务结束时，只返回一句“已经完成”。
这对工作协作远远不够。

完整工作通常需要交付 artifact。

Artifact 可以是：

- 代码改动
- 文档
- 表格
- 报告
- 设计稿
- 工单状态
- 配置变更
- 数据分析结果
- 决策记录

这些对象之所以重要，是因为后续的人、系统或流程需要继续围绕它们工作。

因此，交付不能只看语言总结。
还要验收：

- artifact 是否存在
- 格式是否正确
- 内容是否完整
- 是否满足目标
- 是否遵守约束
- 是否能被后续流程使用
- 是否有版本记录
- 是否能回滚或继续编辑

如果系统只会给出语言说明，却不能稳定产生、修改和交付 artifact，它更像顾问，而不是协作者。

---

## 11. 复盘不是附加动作，而是协作能力开始积累的地方

协作流程如果到交付结果为止，就仍然是一次性行为。
能力积累发生在复盘阶段。

复盘要问：

- 目标是否一开始就清楚
- 哪些约束漏掉了
- 哪些上下文装配不足
- 哪些工具调用不稳定
- 哪些审批节点过多或过少
- 哪些等待节点没有暴露
- 哪些返工本可以避免
- 哪些失败需要进入回归样本
- 哪些经验应该变成模板

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/93_real_workflow_of_human_agent_collaboration_deep_revised__mermaid_05.png)

> 图93-5 复盘如何把一次任务变成长期协作资产
> 没有复盘，协作只能停留在一次次偶然成功。复盘的价值，是把任务里的经验和失败沉淀成下一次更稳的系统结构。

复盘把一次任务变成组织资产。
目标怎样被澄清、哪些约束常漏、哪个审批节点容易卡住、哪些返工常出现，这些都可以通过复盘变成下一次更稳的工作模板。

---

## 12. 成熟协作最终会要求 Harness

本章拆开的所有节点，最后都会指向一个结论：
真实人机协作需要 Harness。

因为仅靠模型无法稳定承接：

- 目标结构化
- 约束管理
- 上下文装配
- 工具协议
- 权限边界
- 审批节点
- trace 观察
- artifact 生命周期
- 等待、返工和接管
- 复盘和回归

这正是第94章要讨论的问题。
未来 AI 产品不会只围绕模型本身竞争，而会围绕谁更会把模型组织进真实工作竞争。

模型可以生成能力。
Harness 才能把能力变成可协作、可治理、可交付的工作链。

---

## 13. 工程现实：很多协作失败，其实是流程断裂

现实中常见的失败，并不是模型完全没有能力，而是协作链某一段断了。

典型表现包括：

- 目标进入了，约束没有结构化
- 材料给了，关键上下文没有装配
- Agent 开始跑了，过程不可见
- 工具接上了，权限边界没设计
- 结果给出来了，artifact 无法继续流转
- 出了错，没人知道在哪个节点接手
- 任务完成了，但经验没有沉淀

这些问题说明：
人机协作的关键，不只是增强中间那颗模型，而是把整条工作链补完整。

成熟系统卖的不是“中间几步看起来有多聪明”，而是整条链是否有断口。

---

## 14. 容易误判的地方

### 误解一：协作流程越短越先进

流程短不等于效率高。
缺少关键节点，常常只是把风险、返工和责任问题挪到后面。

### 误解二：只要结果正确，中间过程无所谓

没有过程可见性，就没有可接管性、可调试性、可审计性和可复用性。

### 误解三：审批会降低自动化价值

不一定。
好的审批节点围绕风险和不可逆动作设置，它保护的是长期信任。

### 误解四：Artifact 只是结果附件

很多场景里，artifact 才是主要结果。
语言说明只是附属说明。

### 误解五：复盘是项目结束后的管理动作

复盘是协作能力积累机制。
没有复盘，系统很难越用越稳。

## 15. 本章小结

1. 人与 Agent 的协作需要被理解成完整流程，而不是几轮对话。
2. 任务入口要把人的愿望压成可执行目标，目标和约束必须分开结构化。
3. 上下文、工具、权限、trace、审批、等待、返工、接管、artifact 和复盘，构成协作链的关键节点。
4. 过程可见决定可接管，artifact 决定可交付，复盘决定可积累。
5. 成熟协作最终会要求 Harness，因为模型本身无法独自承接完整工作链。

核心判断：

> 成熟的人机协作，不是一次回答成功，而是一条能被重复、被观察、被审批、被接手、被交付、也能越用越稳的工作链。
