# 第102章 推理优化

## 先说结论

推理优化不是把模型跑快一点，而是在质量、时延、吞吐、显存、成本和稳定性之间建立一套可度量、可回滚、可持续迭代的工程折中。

训练决定模型能力的上限。
推理决定这份能力能不能进入真实业务。

![插图 第102章 推理优化工具箱与瓶颈对照图]({{ site.baseurl }}/ai_ecosystem_book/images/chapter_102_inference_optimization.png)

> 插图 第102章 推理优化工具箱与瓶颈对照图

第101章讨论 Agent 与 Harness 的评测，重点是判断系统值不值得运行。
本章进入上线前后的性能与成本问题：当一个模型或 Agent 系统准备承接真实流量时，怎样让它跑得起、跑得快、跑得稳，并且不在质量上悄悄退化。

---

## 1. 推理一上线，就会成为成本、体验和容量的主战场

训练当然昂贵，但训练通常是阶段性的。
推理不同。
只要产品还在服务用户，推理成本就会随着每一次请求持续发生。

这会把团队从“模型能力是否足够”推到另一组更现实的问题：

- 首次响应是否足够快
- 生成过程是否足够顺
- 高峰期是否会排队
- 单次请求成本是否可接受
- 显存、带宽和算力是否被有效使用
- 优化后质量是否仍然稳定

因此，推理优化不是部署之后的补丁，而是线上 AI 产品能否成立的核心工程环节。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/102_inference_optimization_deep_revised__mermaid_01.png)

> 图102-1 推理优化不是单点技巧，而是一条工程链路
> 用户看到的是响应速度，平台承担的是请求进入、排队、预填充、解码、缓存、调度、返回和计费组成的完整链路。任一环节失衡，都会反映到体验或成本上。

一个常见误判是：只要模型参数更少，推理就一定更便宜；只要 GPU 更强，体验就一定更好。
真实系统很少这么简单。
如果请求都很长，瓶颈可能在 KV cache 和显存管理。
如果并发很高，瓶颈可能在 batching 和调度。
如果用户要求交互式响应，瓶颈可能在首 token 延迟。
如果任务质量要求高，瓶颈可能不是速度，而是优化后是否引入了质量回退。

推理优化的第一原则不是“先上技巧”，而是先确认系统到底卡在哪里。

## 2. 先建立指标框架，否则优化很容易变成局部自嗨

没有指标框架的推理优化，常常会出现一个问题：
某个数字变好看了，但产品并没有变好。

例如，平均延迟下降，尾部延迟却变差；
吞吐提高，交互请求的首 token 延迟被拖长；
单次推理成本降低，质量退化导致用户重试增多，最终总成本反而上升。

成熟团队通常会把指标至少拆成几组：

- `TTFT`，time to first token，表示从请求发出到第一个 token 返回的时间
- `ITL`，inter-token latency，表示连续 token 之间的间隔
- `TPS`，tokens per second，表示单位时间生成 token 的速度
- `E2E latency`，端到端延迟，表示完整请求从进入到结束的总时间
- `throughput`，吞吐，表示单位时间处理的请求数或 token 数
- `p95`、`p99`，尾部延迟，表示高分位用户是否被牺牲
- `cost per request` 或 `cost per 1K tokens`，表示成本是否具备商业可持续性
- `quality regression rate`，表示优化后质量是否发生可观测退化

这些指标不能孤立看。
推理优化要建立的是指标组合，而不是追逐一个单独冠军数字。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/102_inference_optimization_deep_revised__mermaid_02.png)

> 图102-2 先分清指标，再讨论优化手段
> 平均延迟、首 token 延迟、逐 token 速度、吞吐、尾部延迟和单位成本分别回答不同问题。团队如果不先分清这些指标，就很容易把一个局部优化误判成整体优化。

还要特别注意尾部延迟。
很多 AI 产品的体验不是被平均用户决定的，而是被最慢的那批请求决定的。
一个客服系统、代码助手或企业知识库，如果 `p99` 延迟经常失控，用户会把它理解成“不稳定”，即使平均值仍然好看。

这也是为什么推理优化必须与服务化、观测和回归测试连在一起。
只在离线压测里拿到漂亮数字，不等于线上系统已经可用。

## 3. 推理过程可以拆成预填充和解码，两者的瓶颈完全不同

大语言模型推理通常可以粗略拆成两个阶段：

- `prefill`，预填充阶段，模型处理输入上下文，建立后续生成所需的状态
- `decode`，解码阶段，模型逐 token 生成输出

这两个阶段消耗资源的方式不一样。

预填充阶段更容易受到输入长度、注意力计算和显存带宽影响。
输入上下文越长，预填充越重。
RAG、长文档问答、代码库分析、多轮对话，都可能让预填充成本显著上升。

解码阶段则更像连续的小步生成。
它常常受限于内存访问、KV cache 读写、调度粒度和并发策略。
用户觉得回复“滚动得顺不顺”，很大程度上就在看这一段。

因此，同样是“延迟高”，原因可能完全不同：

- 输入太长，主要卡在 prefill
- 输出太长，主要卡在 decode
- 首字慢，可能是排队、预填充或路由问题
- 生成慢，可能是解码吞吐或 KV cache 管理问题
- 高峰期慢，可能是调度和资源隔离问题

这就是推理优化最需要工程诊断的地方。
如果没有把请求生命周期拆开，团队很容易用错药。

## 4. 量化的核心不是“越低越好”，而是可控误差管理

量化看起来是把权重或激活从更高精度变成更低精度，例如从 FP16、BF16 到 FP8、INT8、INT4。
它能降低显存占用，提高吞吐，降低单位请求成本。

但量化从来不是免费午餐。

它真正交换的是：

- 数值精度
- 显存占用
- 吞吐能力
- 硬件适配性
- 质量稳定性
- 工程复杂度

不同量化方案的风险也不同。
有些方案主要量化权重，对部署更友好；
有些方案同时涉及激活，需要更严格的校准；
有些方案在通用问答上表现稳定，但在数学、代码、工具调用、长上下文或安全边界上退化明显。

所以量化验证不能只跑一组通用 benchmark。
它至少要覆盖目标业务里的高价值任务、长尾任务和高风险任务。
对 Agent 系统来说，还要看工具调用参数、路径选择和错误恢复有没有变化。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/102_inference_optimization_deep_revised__mermaid_03.png)

> 图102-3 量化是在质量、成本和可部署性之间做受控交换
> 成熟量化不是把位宽压到最低，而是在明确任务边界、硬件边界和质量边界之后，选择最划算的精度组合。

出版层面也需要避免把量化写成单向进步。
低位宽不是荣誉勋章。
如果它让核心任务质量退化、人工兜底增加、用户重试增多，最终可能只是把显存账单换成了产品账单。

## 5. KV cache、Prompt cache 和语义缓存解决的是不同层面的重复计算

缓存是推理优化里最容易被低估的一类能力。
但缓存也不是一个东西，而是一组分层机制。

`KV cache` 主要发生在模型解码过程中。
它保存历史 token 的 key/value 状态，避免每生成一个 token 都重复计算全部上下文。
对长上下文和多轮对话来说，KV cache 的管理效率会直接影响显存占用、吞吐和尾部延迟。

`Prompt cache` 更偏请求层。
当多个请求共享相同系统提示词、长文档前缀或固定上下文时，平台可以复用已经计算过的前缀状态。
企业知识库、代码助手和长文档问答都很依赖这类优化。

语义缓存则更靠近产品层。
它尝试识别语义相近的问题，并复用已有结果或中间结果。
这种缓存能显著降本，但风险也更高，因为“看起来相似”的问题并不总是可以复用同一个答案。

因此，缓存要分层设计：

- 模型内部缓存追求计算复用
- 请求前缀缓存追求长上下文复用
- 业务语义缓存追求结果复用
- 工具结果缓存追求外部调用复用

越靠近业务语义，缓存越需要谨慎。
对事实问答、法律、医疗、金融、权限敏感数据来说，缓存命中必须带有严格边界。
否则缓存会从性能优化变成错误传播机制。

## 6. Batching 和调度是在用等待换利用率，必须保护交互体验

批处理的目标很直接：
把多个请求组织到一起，让硬件更充分地工作。

在 GPU 上，单个请求常常无法吃满设备。
如果平台能把多个请求合批，吞吐通常会显著提高，单位成本也会下降。

但 batching 的代价是等待。
为了凑批，系统可能让某些请求多等一会儿。
如果这个等待发生在离线任务里，通常可以接受。
如果发生在交互式产品里，用户会立刻感知到卡顿。

现代推理系统常用 continuous batching、动态批处理、优先级队列等机制，试图在吞吐和延迟之间找平衡。
工程难点不是“能不能批”，而是：

- 哪些请求可以一起批
- 批多大才划算
- 等待多久还可接受
- 高优先级请求是否要插队
- 长请求是否会拖慢短请求
- 流式输出是否会被批处理影响

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/102_inference_optimization_deep_revised__mermaid_04.png)

> 图102-4 Batching、调度和体验之间的取舍
> 批处理提高吞吐，但也可能增加等待。推理平台真正需要的是按请求类型、优先级和业务场景做调度，而不是把所有请求粗暴放进同一个队列。

这里有一个非常实际的出版判断：
不要把 batching 写成单纯的性能技巧。
它本质上是一种资源分配制度。
当不同用户、不同租户、不同任务同时争抢模型能力时，batching 和调度决定谁等待、谁优先、谁被降级。
这已经进入下一章“模型服务化”的范围。

## 7. Speculative decoding 能降低等待感，但不是所有场景都适合

Speculative decoding 的基本思路是：
先用一个更小、更快的 draft model 预测一段候选 token，再用目标大模型验证这些候选 token。
如果候选被接受，就可以一次推进多个 token，从而减少解码等待。

它的价值在于改善用户感知延迟，尤其是逐 token 生成速度。
但它也不是无条件收益。

团队需要关注：

- draft model 的质量是否足够
- 验证过程是否真的省时
- 接受率是否稳定
- 硬件上是否有足够并行空间
- 系统复杂度是否值得
- 质量和一致性是否被影响

如果 draft model 经常猜错，验证成本可能抵消收益。
如果系统瓶颈根本不在 decode，而在排队、prefill 或外部工具调用，speculative decoding 也不会解决主问题。

所以它适合被看作一个针对性工具，而不是通用加速按钮。

## 8. 并行、卸载和内存管理决定大模型能否稳定落在硬件上

当模型变大、上下文变长、并发增加时，推理优化会迅速进入硬件和系统工程。

常见手段包括：

- tensor parallel，把单层计算切到多张卡上
- pipeline parallel，把模型不同层放到不同设备上
- expert parallel，用于 MoE 等结构的专家分布
- CPU 或 NVMe offload，把部分状态移出显存
- paged attention 或类似机制，更细粒度地管理 KV cache
- CUDA graph、kernel fusion、算子优化，减少调度和访存开销

这些技术不会自动让系统更好。
它们往往引入新的通信成本、调度复杂度和故障面。
例如，多卡并行可能降低单卡显存压力，却增加跨卡通信；
offload 可以缓解显存不足，却可能把瓶颈转移到 PCIe 或存储带宽；
更激进的算子优化可以提升吞吐，却可能带来兼容性和可维护性成本。

因此，大模型推理优化不是“硬件越多越好”，而是要让模型结构、请求形态和硬件拓扑匹配。
真正成熟的容量规划，会同时看模型大小、上下文长度、并发、输出长度、显存余量、峰值流量和故障冗余。

## 9. 多模型路由会把推理优化推向产品策略层

很多线上 AI 系统已经不再是单模型系统。
它们会同时使用大模型、小模型、专用模型、嵌入模型、重排模型、视觉模型、代码模型和工具调用模型。

这时，推理优化不只是底层性能问题，而会变成产品策略问题。

系统需要判断：

- 简单请求是否可以走小模型
- 高价值请求是否应该走强模型
- 长上下文是否需要先压缩或检索
- 是否可以先用分类器判断任务难度
- 哪些请求值得重试
- 哪些失败可以降级
- 哪些用户层级对应不同延迟和质量目标

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/102_inference_optimization_deep_revised__mermaid_05.png)

> 图102-5 多模型路由把推理优化推向策略层
> 当系统拥有多种模型和多种服务等级时，推理优化就不只是让某个模型更快，而是决定每类请求应该使用哪种能力、承担多少成本、接受怎样的质量边界。

模型路由最大的风险，是把优化隐藏成不可见的质量漂移。
用户同样发出一个问题，系统因为成本策略、负载状态或版本变更走了不同模型，结果可能不一致。
如果没有评测集、回放系统和线上观测，团队很难发现这种漂移。

所以多模型路由必须和第101章的评测体系连起来。
否则，推理优化会在表面上省钱，在深层上破坏系统一致性。

## 10. 推理优化必须带着质量回归和发布纪律

推理优化有一个危险之处：
它很容易让团队兴奋于性能数字，而忽略输出质量。

任何优化动作都应该被当成一次潜在的行为变更。
量化可能改变模型边界。
蒸馏可能压缩复杂推理能力。
缓存可能复用不该复用的结果。
路由可能让同一任务落到不同能力层。
batching 和并发策略可能改变超时、重试和流式体验。

因此，每次推理优化都应该配套三类验证：

- 性能验证：延迟、吞吐、显存、成本是否达标
- 质量验证：核心任务、长尾任务、高风险任务是否退化
- 生产验证：灰度、回滚、观测、告警是否准备好

这也是为什么推理优化不应只由底层工程团队完成。
模型团队、平台团队、产品团队、评测团队和业务方都需要参与边界定义。

没有质量回归意识的推理优化，很可能只是把问题从 GPU 账单转移到用户信任上。

## 11. 工程现实：推理优化是一套持续闭环，不是一次项目

现实系统里不存在一次完成的最优推理方案。

原因很简单：

- 请求分布会变
- 模型版本会变
- 上下文长度会变
- 硬件价格会变
- 用户规模会变
- 产品承诺会变
- 安全与合规边界会变

今天合理的 batch size，流量结构变化后可能不合理。
今天可接受的量化方案，任务类型变化后可能引入退化。
今天省钱的路由策略，付费用户增加后可能影响高价值体验。

成熟团队会把推理优化做成闭环：

1. 采集真实请求画像。
2. 定位主要瓶颈。
3. 选择最小优化动作。
4. 在代表性评测集和回放集上验证。
5. 灰度到线上小流量。
6. 观察性能、质量、成本和投诉信号。
7. 保留回滚路径。
8. 把结果沉淀为容量规划和服务等级策略。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/102_inference_optimization_deep_revised__mermaid_06.png)

> 图102-6 推理优化的持续闭环
> 推理优化不是一次性调参，而是从真实流量出发，经过诊断、实验、验证、灰度、观测和回滚，不断逼近当前业务条件下的最佳折中。

这个闭环的终点不是“再快一点”，而是“把有限资源稳定地给到最值得的地方”。

## 12. 容易误判的地方

### 误解一：推理优化就是把模型压小

模型变小只是手段之一。
真正的目标是让质量、时延、吞吐和成本组合起来成立。

### 误解二：平均延迟下降就代表体验变好

未必。
如果首 token 变慢、尾部延迟变差，用户体验仍然会下降。

### 误解三：量化一定会伤害质量

量化可能带来退化，但成熟量化会通过校准、评测和灰度控制退化边界。
关键不是拒绝量化，而是知道哪些任务不能退。

### 误解四：batching 总是越大越好

batching 提高吞吐，但也可能增加等待。
交互式请求、付费高优先级请求和离线批量任务不应该被同一种调度策略处理。

### 误解五：推理优化是底层工程团队自己的事

不是。
它会改变用户体验、产品成本、服务等级、质量边界和商业模型，因此必须进入跨团队决策。

## 延伸阅读建议

- 推荐起点：Chip Huyen《AI Engineering》、Martin Kleppmann《Designing Data-Intensive Applications》，适合把推理、服务、评测、数据和生产系统连起来理解。
- 工程延伸：关注 vLLM、TensorRT-LLM、Triton Inference Server、Ray Serve、KServe 等推理与服务框架的设计思路。
- 关键主题：`KV cache`、`continuous batching`、`paged attention`、`speculative decoding`、`quantization`、`latency percentile`、`capacity planning`。

## 13. 本章小结

1. 推理优化的目标不是单点加速，而是质量、时延、吞吐、显存和成本之间的系统折中。
2. 必须先建立指标框架，区分 TTFT、ITL、TPS、端到端延迟、吞吐、尾部延迟和单位成本。
3. prefill 与 decode 的瓶颈不同，长上下文、KV cache、batching 和调度都需要分开诊断。
4. 量化、缓存、批处理、speculative decoding、多模型路由都不是通用按钮，而是针对特定瓶颈的工具。
5. 推理优化必须带质量回归、灰度、观测和回滚纪律，否则性能收益可能换来产品退化。

核心判断：

> 推理优化要做的，不是让模型勉强跑起来，而是让模型能力以可测量、可承诺、可持续的方式进入真实世界。
