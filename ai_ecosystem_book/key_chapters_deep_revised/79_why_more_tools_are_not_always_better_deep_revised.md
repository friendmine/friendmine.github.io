# 第79章 工具不是越多越好

## 先说结论

工具给少了，Agent 很难完成真实任务。
工具给多了，系统会在选择、路由、权限、成本和失败恢复上迅速变复杂。

第78章讲到，Function Call 和 Tool Use 把模型接入受控执行层。
这一章要补上下一层判断：工具一旦进入执行系统，价值就不只取决于数量，而取决于工具集合是否清晰、可区分、可治理、可观测、可恢复。

一个成熟 Agent 的行动能力，通常不是由“挂了多少工具”直接决定，而是由工具世界的质量决定。
工具职责混乱，模型会选错；权限边界模糊，系统会放大风险；失败路径缺失，演示能跑通，生产却会失控。

## 1. 工具数量增加，能力边界和复杂度会一起上升

很多团队在 Tool Use 跑通以后，第一反应是继续加工具。
这个冲动可以理解：工具越多，模型似乎能做的事越多。

但工具系统不是插件陈列架。
每新增一个工具，系统同时新增了几类成本：

- 模型需要理解它做什么
- 路由层需要判断何时选择它
- 权限层需要定义谁能用、能做什么
- 观测层需要记录它是否被误用
- 失败恢复层需要知道它出错后如何处理

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/79_why_more_tools_are_not_always_better_deep_revised__mermaid_01.png)

> 图79-1 工具数量与可靠性、复杂度之间的张力图
> 工具增加会扩大能力边界，也会同步抬高选择难度、治理成本和失败恢复成本。

工具数量不是单向收益。
如果新增工具没有清楚职责、稳定 schema、明确权限和可观测结果，它带来的可能不是能力扩张，而是行动空间噪声。

---

## 2. 工具发现与选择，本身就是认知负担

对模型来说，工具列表不是简单配置。
它是一组需要阅读、比较和选择的行动选项。

如果工具命名相近、描述含糊、职责重叠，模型很容易在路由阶段出错。

例如：

- `search_web`
- `search_docs`
- `search_company_kb`
- `lookup_customer_record`

这些工具看起来都在“查东西”，但适用范围、可信来源、权限等级和返回结构可能差异很大。
如果描述里没有说清差别，模型就只能在相似名字之间猜。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/79_why_more_tools_are_not_always_better_deep_revised__mermaid_02.png)

> 图79-2 工具发现与选择，本身就是一个认知负担

所以，工具质量不只包括接口质量，也包括可理解性质量。
一个工具如果不能被模型可靠识别，轻则闲置，重则被错误调用。

---

## 3. 工具治理至少要处理注册、发现、路由和权限

工具系统一旦进入生产，就不能只靠“工具列表”管理。
它需要工具治理。

至少有四层需要明确：

- `注册`
  工具是否被正式纳入系统；名称、职责、schema、返回格式是否统一。

- `发现`
  模型能否看懂工具用途；工具描述是否说明适用场景、不适用场景和输入要求。

- `路由`
  多个工具都可能可用时，谁是默认选择，谁负责兜底，谁只在特定条件下调用。

- `权限`
  哪些工具只读，哪些工具可写，哪些动作需要审批，哪些环境不能被访问。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/79_why_more_tools_are_not_always_better_deep_revised__mermaid_03.png)

> 图79-3 工具治理至少要同时处理注册、发现、路由和权限四个维度

这四层缺一层，工具世界都会变得脆弱。
注册混乱，会让工具列表像临时脚本堆。
发现不清，会让模型看不懂工具。
路由缺失，会让相似工具互相竞争。
权限不明，会让一次选择错误变成业务风险。

---

## 4. 路由和优先级要显式设计

工具多起来以后，模型面对的不是“能不能调用”，而是“该调用哪个”。

系统需要提前说明：

- 哪个工具是默认入口
- 哪个工具只用于内部资料
- 哪个工具只用于外部公开资料
- 哪个工具适合只读查询
- 哪个工具会产生写入副作用
- 哪个工具失败后可以重试
- 哪个工具需要人工确认

没有这些规则，模型会在相似工具之间摇摆。
看起来是灵活，实际会变成不可解释、不稳定、难审计。

一个成熟工具系统，应当尽量减少模糊选择。
工具不是越多越好，而是每个工具的职责越清楚越好。
能合并的工具不要拆碎；需要拆开的工具，要把边界写明。

---

## 5. 权限分层决定风险半径

工具越接近真实执行层，权限分层越重要。

工具系统至少要区分：

- 只读查询
- 可写入但可回滚
- 可写入且影响外部用户
- 高风险业务动作
- 不可逆动作
- 生产环境动作

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/79_why_more_tools_are_not_always_better_deep_revised__mermaid_05.png)

> 图79-5 权限和边界决定了工具系统的风险上限

如果这些动作都暴露在同一层工具列表里，模型一次误判就可能触发高风险操作。
更稳妥的方式，是把工具按风险等级分层：低风险动作可以自动执行，高风险动作需要确认，不可逆动作需要更严格的审批、审计和回滚预案。

权限不是工具系统的附属项。
它决定了 Agent 的行动半径，也决定了错误能扩散多远。

---

## 6. 失败路径要和成功路径一起设计

成功调用一次，只能证明系统有一条顺畅路径。
生产系统还要面对失败路径。

工具可能会：

- 超时
- 返回空结果
- 返回结构变化
- 权限过期
- 外部状态已被别人修改
- 部分成功、部分失败
- 重试时产生重复副作用

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/79_why_more_tools_are_not_always_better_deep_revised__mermaid_04.png)

> 图79-4 工具系统是否成熟，往往要看失败路径是否被事先设计过

所以工具系统需要为失败设计规则：

- 哪些失败可以自动重试
- 哪些失败只能降级为只读查询
- 哪些失败需要交还给用户
- 哪些失败需要停止任务
- 部分成功后如何补偿
- 重试是否会造成重复执行

工具越多，失败组合越多。
没有失败治理，工具集合越大，系统越容易在异常场景里发散。

---

## 7. 工具质量就是 Agent 的行动空间质量

Agent 的推理能力再强，也要通过工具进入现实系统。
如果工具世界混乱，它的行动质量会被压低。

低质量工具集合常有这些表现：

- 名称相近，职责不清
- schema 不一致
- 返回格式不稳定
- 工具之间功能重叠
- 权限边界漂移
- 失败信息不可读
- 没有调用日志和误用统计

这种环境下，模型不是在一个清晰行动空间里做选择，而是在一组模糊接口里试探。
结果很可能是工具选错、参数填错、失败难以恢复、问题难以复盘。

从这个意义上说，工具质量就是行动空间质量。
工具集合越清晰，Agent 越容易把推理结果转成稳定行动。

---

## 8. 工具观测决定能否持续改进

很多团队只关注工具是否能调用，却忽略工具是否被正确使用。
生产系统需要观测工具行为。

至少要看：

- 哪些工具被频繁调用
- 哪些工具很少被用到
- 哪些工具误用率高
- 哪些工具经常失败
- 哪些工具返回结果被模型误解
- 哪些调用被权限层拦截
- 哪些任务在工具选择阶段失败

没有这些数据，团队只能凭感觉调工具描述和权限策略。
有了观测，才能判断是工具太多、描述太差、路由不清，还是上下文没有给够任务条件。

这也自然接到第80章。
工具选择不只是工具问题，也和 Context Engineering 有关。
模型能看到的工具描述、任务状态、权限信息和历史结果，都会影响它选择哪个工具。

---

## 9. 成熟做法通常是克制地扩展工具

成熟团队通常不会一开始开放大量工具。
更常见的做法是：

- 先接少量高质量工具
- 为每个工具写清职责和边界
- 统一 schema 和返回结构
- 明确只读、写入和高风险动作
- 给高风险动作加确认
- 观察误用率、失败率和恢复路径
- 再逐步扩展工具集合

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/79_why_more_tools_are_not_always_better_deep_revised__mermaid_06.png)

> 图79-6 工程里常见的正确做法，看起来往往都很克制

这种克制不是保守。
它是把行动空间做清楚以后再扩张。

如果工具集合还没有治理好，继续增加工具，只会把问题藏得更深。
系统表面能力变多，实际可靠性可能下降。

---

## 10. 容易误判的地方

### 误解一：工具越多，Agent 越强

工具数量会扩大动作空间，也会扩大选择难度、治理成本和风险半径。

### 误解二：工具描述写清楚，模型就会稳定选对

描述很重要，但还需要职责分层、路由优先级、权限规则和失败恢复。

### 误解三：工具失败只是接口团队的问题

失败恢复、降级、回滚和人工交接，本来就是 Agent 系统设计的一部分。

### 误解四：工具治理会拖慢产品迭代

缺少治理时，早期看起来更快，后期会在误用、回滚、权限和排障上付出更高成本。

## 11. 本章小结

1. 工具数量增加，会同时扩大能力边界和系统复杂度。
2. 工具发现、选择和路由本身就是 Agent 的认知负担。
3. 工具治理至少要处理注册、发现、路由和权限。
4. 权限分层决定工具错误能扩散多远。
5. 失败路径、观测数据和误用统计决定工具系统能否持续改进。
6. 工具质量就是 Agent 的行动空间质量。

核心判断：

> 对 Agent 来说，工具丰富度不等于能力成熟度；清晰、可治理、可恢复的工具集合，才是可执行智能的基础。
