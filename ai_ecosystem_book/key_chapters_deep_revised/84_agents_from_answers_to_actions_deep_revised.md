# 第84章 Agent：从回答走向行动

Agent 让模型从“回答问题”跨进“推进任务”。

一旦系统拥有目标理解、任务规划、工具调用、状态更新和持续执行能力，风险就会从“一次回答出错”升级成“一整条行动链偏航”。

这正是 Agent 值得重视、也必须被谨慎对待的地方。它可能创造连续推进任务的价值，也可能在表面看起来始终合理的情况下，把错误一步步推进成真实后果。

## 1. Agent 成为中心词，是因为系统对象变了

当模型主要用来回答问题时，把它理解成强大的语言接口通常够用。

用户发出请求，系统生成回复，交互在一轮或几轮对话中结束。

这个阶段的核心问题，是理解能力、表达能力、事实准确性和对话体验。

但只要我们对系统提出新的期待，旧框架就会立刻不够。

比如我们希望它连续执行步骤、调用工具、查看中间结果、根据反馈调整路径，并最终把一项任务做完。

这时，“聊天模型”这个概念就显得过轻。

因为系统已经从输出器走向执行体。

Agent 之所以成为新关键词，原因不只是行业喜欢换名字，而是我们讨论的对象已经变了。

我们谈论的，已经不再只是一次回答，而是一种持续行动结构。

这也是很多从业者开始感到难度跃迁的地方。

回答系统只需要对这一轮输出负责。

行动系统则要对目标解释、后续状态、工具副作用、失败回退和最终交付负责。

一旦系统开始“把事情继续做下去”，评价标准就会从“答得像不像”迅速变成“能不能稳定推进，并且不把局面弄乱”。

---

## 2. 从上下文供给到行动闭环

第83章讨论的是检索、压缩与上下文装配。

那一章的核心问题是：系统怎样把正确材料送进模型当前窗口，让模型能在这一轮工作里使用知识。

Agent 在这个基础上又往前走了一步。

它不只是消费上下文，还会根据上下文做计划、采取动作、观察结果，并把新结果写回后续上下文。

这意味着上下文工程不再只是“回答前的材料准备”，而成为行动闭环的一部分。

一个 Agent 在执行任务时，通常会反复经历：

- 读取当前上下文；
- 判断任务状态；
- 制定下一步；
- 调用工具或改变外部状态；
- 观察返回结果；
- 更新任务状态；
- 再次决定是否继续、修正、停止或升级。

这条链一旦成立，系统就不再是单轮输出增强，而是在围绕目标不断推进。

Agent 把上下文工程向前推进了一步：信息不再只服务于回答，还会驱动计划、工具调用和状态更新。

这也是风险升级的起点。

信息装配错了，回答可能错；行动链建立在错误信息上，后续每一步都可能继续放大这个错误。

## 3. 一个 Agent 至少要具备四种能力

如果把定义压到最简，一个 Agent 至少要同时具备四种能力。

第一，感知。

系统要知道当前输入、环境状态、任务阶段、可用工具和约束条件。

第二，规划。

系统要能决定下一步应该采取什么动作，而不是只生成一段完整回复。

第三，执行。

系统要能把动作施加到工具、文件、系统、流程或现实环境上。

第四，反馈。

系统要能回看动作结果，并在失败、偏航、信息不足或环境变化时调整后续动作。

少了其中任何一项，系统都更像增强型调用接口，而不是严格意义上的 Agent。

Agent 的边界，落在能否围绕目标形成连续闭环上，而不是落在输出内容有多长上。

一个只会写出计划但不会执行的系统，是规划助手。

一个只会调一次工具但不会根据返回结果继续调整的系统，是工具增强问答。

一个能执行但不记录状态、不评估后果的系统，则是危险的自动化脚本。

Agent 的边界，应该落在持续闭环上，而不是落在名词包装上。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/84_agents_from_answers_to_actions_deep_revised__mermaid_01.png)

> 图84-1 Agent 的最小闭环
> Agent 的本质不在“会不会想”，而在“能不能持续形成感知、规划、执行、反馈的循环”。

## 4. Agent 是一条 loop，不是一次更长的推理

很多人初次理解 Agent，会把它简单想成“让模型自己多想几步”。

这个直觉并不算错，但仍然停留在回答式思维里。

Agent 的重点不在一次推理变长，而在整个系统进入循环：

看当前状态，做决定，执行动作，观察新状态，然后继续下一轮。

只要环境会变化、任务会展开、错误会出现，这个循环就不能省略。

所以 Agent 与普通问答模型的分界，不是字数更多、链路更长，而是有没有形成“感知、规划、执行、反馈”的闭环。

很多团队会把更长的 chain-of-thought 误当成 Agent 雏形。

但只要那条推理链没有进入外部动作，也没有被外部结果反向改写，它仍然停留在一次输出里。

Agent 的难点不在多想几步，而在于每一步都会改变后面的地面。

一个普通回答错了，用户可以忽略它。

一个 Agent 的动作错了，它可能已经改了文件、发了消息、写了数据库、触发了流程，甚至把错误状态带进下一轮判断。

这也是为什么一旦系统走向 Agent，问题会从推理质量快速升级成状态管理、权限管理和责任管理问题。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/84_agents_from_answers_to_actions_deep_revised__mermaid_02.png)

> 图84-2 更长推理与 Agent loop 的差别
> 这张图的关键，是把“想得更久”和“真的进入行动循环”分开。前者仍然停留在一次输出里，后者则会把动作结果反过来改写下一轮判断。

## 5. Agent 和普通问答的差别，在于是否改变状态

这是最需要讲清的一层。

普通问答系统通常更接近：

- 用户给任务；
- 模型生成答案；
- 用户自行判断是否使用；
- 交互结束。

而 Agent 面对同一件事时，更可能是：

- 先理解目标和边界；
- 再拆成可执行步骤；
- 中途调用工具或环境能力；
- 根据中间结果调整路径；
- 更新任务状态或外部状态；
- 最后回交一个可继续使用的产物。

这两者的关键差别，不只在步骤多少，还在系统是否改变状态。

回答系统主要改变的是屏幕上的文本。

Agent 系统可能改变的是文件、数据库、代码仓库、工单状态、日程、客户沟通、资金流、配置项或审批流程。

一旦系统能改变状态，它就必须面对新的工程问题：

- 动作是否可撤销；
- 修改是否幂等；
- 权限是否足够且不过宽；
- 外部状态是否真的按预期变化；
- 错误动作是否会污染后续判断；
- 谁能观察、暂停、回滚或接管。

Agent 的意义不在于把对话变复杂，而在于让模型跨进“做事”这件事里。

而“做事”意味着系统交付的不再只是一个看起来不错的回答，而是某种可继续使用的结果物或状态变化。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/84_agents_from_answers_to_actions_deep_revised__mermaid_03.png)

> 图84-3 问答模式与 Agent 模式的差别
> 分界不在“思考更久”，而在是否形成持续行动闭环。

## 6. 真实工作本来就不是一锤子买卖

Agent 之所以吸引人，是因为它更接近真实工作。

真实世界里的任务很少是一锤子买卖。

大多数工作都要先理解目标，再拆步骤，在过程中查资料、调用工具、发现问题、修正路径，最后形成可交付结果。

例如，用户不是要“写一段说明”，而是要“把一个需求变成可提交的代码改动”。

这时系统就需要：

- 理解需求；
- 查找相关代码；
- 判断影响范围；
- 设计修改方案；
- 编辑文件；
- 运行测试；
- 发现失败；
- 回退或继续修改；
- 最后形成可交付说明。

这已经是一条任务推进链。

再比如，企业流程里用户不是要“告诉我这个客户该怎么处理”，而是要“完成客户投诉分级、准备回复、判断是否需要升级、生成可审批的处理方案”。

这同样属于多步状态推进。

Agent 的价值来自持续推进，风险也来自持续推进。

它越能把事情做下去，就越需要回答谁来观察、谁来中止、谁来接管、谁对结果负责。

## 7. Agent 会把模型问题放大成系统问题

在单纯问答模型中，一次幻觉最多只是屏幕上的一句错误输出。

用户看到后可以质疑、忽略、重问或修正。

但在具备真实执行能力的 Agent 系统里，一次小判断失误可能被逐步放大成系统级问题。

一次理解偏差，可能把系统带到错误的数据源、错误的工具或错误的目标对象上。

一次起点错位的计划，可能在后续每一步都被“继续执行”不断放大。

一次错误工具调用，可能改变外部状态，并让后续规划建立在已经被污染的状态上。

一次不恰当重试，可能把成本、限流、权限和副作用全部放大。

很多团队吃到的亏，并不是模型答错了一句，而是系统在错误前提下仍然持续执行、不断重试、不断消耗资源，最后把成本、权限和外部影响一起扩大。

因此，Agent 标记的是系统责任边界的变化。

麻烦的地方在于，这类错误通常不像普通问答那样立刻暴露。

一次问答出错，用户大多当场就能感觉到。

一次行动偏航，却可能先表现为“系统还挺忙、看起来也有进展”，直到几步之后才暴露出整条链已经建在错误前提上。

Agent 时代最怕的不是明显不会做，而是认真且连续地做错。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/84_agents_from_answers_to_actions_deep_revised__mermaid_04.png)

> 图84-4 Agent 如何把局部错误放大成系统性后果
> 一次回答错了，通常只是回答错了；一次行动错了，却可能改写外部状态、污染后续步骤，进而把局部问题放大成整条任务链的问题。

## 8. 工程难点不是初次跑起来，而是长期可托付

演示中的 Agent 往往看起来很顺。

一串思考，一串动作，任务完成。

真实工程的难点通常不在于“初次能不能跑起来”，而在于：

- 能不能稳定跑；
- 能不能在复杂环境里不发飘；
- 能不能让人持续看见它在做什么；
- 能不能在关键时刻被接管、被中止、被回放；
- 能不能在失败后解释自己为什么失败；
- 能不能把错误限制在可恢复范围内。

成熟 Agent 产品通常会把观测、调试、权限、审批、回放和回滚能力一起带进来。

没有这些，Agent 只是一个危险地扩张了执行范围的模型外壳。

判断一个 Agent 是否从 demo 进入可托付系统，至少要看六件事。

第一，状态是否可见。

人能不能看清系统现在处在哪一步、已经做了什么、接下来准备做什么。

第二，轨迹是否可回放。

每一次计划、工具调用、返回结果、重试和失败是否能被追踪。

第三，权限是否被约束。

系统能否区分读、写、删除、发布、支付、提交等不同风险等级的动作。

第四，失败是否有路径。

失败后是重试、换路、暂停、回滚，还是升级给人，不能临时靠模型自由发挥。

第五，人工是否能接管。

人接手时是否知道当前状态、风险、剩余任务和可选操作。

第六，评测是否覆盖过程。

不能只看最终输出，还要看过程是否可靠、成本是否可控、错误是否可解释。

> 图84-5 Agent 从 demo 到可交付系统的门槛链

## 9. 一个真实任务在 Agent 里如何流动

如果把一个真实 Agent 任务写成工作流，它通常更接近下面这条链：

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/84_agents_from_answers_to_actions_deep_revised__mermaid_06.png)

> 图84-6 Agent 的任务推进流程
> 这张图帮助读者把“Agent 是闭环系统”这句话落到真实任务推进过程里，而不是停留在抽象定义上。

这也是为什么 Agent 靠近工作流、状态机和协作系统。

它把模型拉进了任务推进链，而不只是增强单轮输出。

在这条链里，最关键的不是某一轮推理多漂亮，而是每一步是否能被稳定承接。

目标解释要能进入计划。

计划要能变成工具动作。

工具动作要能改变或读取外部状态。

外部结果要能回到任务状态。

任务状态要能影响下一步计划。

异常要能触发停止、重试、回滚或升级。

这才是 Agent 和普通问答之间真正的结构差别。

## 10. Agent 不是自动化脚本，也不是万能员工

Agent 很容易被两种错误想象拉偏。

一种想象是把它看成自动化脚本。

脚本按照预设步骤执行，适合边界稳定、路径固定、异常有限的任务。

Agent 则更适合目标明确但路径可能变化的任务。它需要根据中间反馈调整动作。

如果任务路径完全固定，用传统 workflow 或脚本可能更稳定、更便宜、更容易验证。

另一种想象是把它看成万能员工。

这同样危险。

Agent 没有天然的组织责任，也不会自动理解所有业务边界。它需要清晰目标、工具契约、权限限制、状态管理和人工接管。

把 Agent 当万能员工，容易让团队过早交出过宽权限；把 Agent 当普通脚本，又会低估它在不确定环境里的适应价值。

更准确的理解是：

Agent 是一种围绕目标持续行动的系统结构。

它介于静态 workflow 和完全人类执行之间，适合那些需要持续判断、但又能被清晰约束和观察的任务。

## 11. Agent 是下册中段的支点

一旦承认 Agent 可以理解为“围绕目标持续行动”，后续很多系统设计问题都会顺势浮现：

- 行动边界如何定义；
- 工具如何安全暴露；
- 状态如何保存和更新；
- 过程如何被记录与评估；
- 失败如何被截断和回滚；
- 人如何在关键节点参与；
- 多个 Agent 或 Agent 与人的协作如何被组织。

也就是说，Agent 不是一章新的功能说明，而是下册后半段的支点概念。

它把全书从“模型会不会”推进到“系统怎么做”。

第85章会继续拆开 Agent 的内部结构：目标层、规划层、执行层、状态层、反馈层、控制层和可观测性分别承担什么职责。

如果第84章解决的是“为什么 Agent 已经不是普通回答系统”，第85章解决的就是“这个行动系统内部到底长什么样”。

## 12. 容易误判的地方

### 误解一：Agent 就是更长一点的链式思考

不是。

思考只是其中一部分，关键是有没有感知、执行、反馈和状态更新闭环。

### 误解二：会调工具的模型自然就是 Agent

不一定。

没有目标持续性、状态管理和反馈修正，它更像工具增强型问答器。

### 误解三：Agent 主要难在模型够不够聪明

很多实际难点在观测、权限、回退、评测和治理机制上。

### 误解四：能跑一次就说明 Agent 可用

不能。

能跑一次只说明 demo 成立。可长期托付还需要状态可见、轨迹可回放、权限可控、失败可恢复。

### 误解五：Agent 越自主越好

不一定。

自主性必须和风险等级匹配。越能改变真实世界，越需要控制边界、审批节点和人工接管。

## 13. 本章收束

1. Agent 是目标驱动的连续行动系统，而不是单次回答器。
2. 它至少要具备感知、规划、执行和反馈四类能力。
3. Agent 可以理解为 loop，而不是一次更长的推理。
4. Agent 和普通问答的关键差别，在于是否形成持续行动结构并改变状态。
5. 一旦进入行动闭环，模型问题会放大为系统问题。
6. 成熟 Agent 需要伴随状态可见、轨迹可回放、权限可控、失败可恢复和人工可接管。
7. Agent 是下册后半段的支点，因为后续的内部结构、工作模式、多智能体、人在环和 Harness 都从这里展开。

Agent 开始能推进现实任务，也因此放大了错误、权限、副作用和责任。最危险的情况，不是系统明显不会做，而是它一边看起来很合理，一边把错误持续往前推。后面讨论 Agent 内部结构、工作模式、多智能体、人在环和 Harness，都要从这条行动链出发。
