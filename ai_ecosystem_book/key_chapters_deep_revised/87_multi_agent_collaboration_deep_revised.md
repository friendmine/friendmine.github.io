# 第87章 多智能体协作：什么时候真的比单 Agent 更好

## 先说结论

多智能体协作值得做的时刻，不是因为它听起来更像未来，而是因为任务已经需要多角色分工，并且这份分工收益足以覆盖额外系统税。

如果任务不能稳定拆分、角色边界不清、中间状态无法移交，多 Agent 通常不会让系统更聪明，只会让错误链更长、通信成本更高、责任归因更难。

多智能体不是单 Agent 的自然升级版，而是一种更重的系统架构选择。

---

## 1. 多 Agent 不是更炫的单 Agent

第86章把 Agent 的工作模式分成了单 Agent、Workflow + Agent、Supervisor、多 Agent 和人在环。

在这些模式里，多 Agent 最容易让人产生“未来感”。多个角色同时工作、互相讨论、彼此复核，演示时确实更有戏剧性。

但工程上首先要问的不是“能不能多 Agent”，而是：

- 任务是否真的可拆；
- 子任务之间是否可以并行；
- 不同角色是否拥有不同能力或不同信息源；
- 中间结果是否能稳定表达和移交；
- 最后是否有统一收敛、冲突裁决和停止条件；
- 多出来的通信、调度和调试成本是否值得。

如果这些问题没有成立，多 Agent 常常只是把一个不稳定系统复制成几份，再叠上一层沟通噪声。

这类失败在真实系统里很常见。

原本单 Agent 做不稳，是因为目标不清、工具链不稳、状态不可见、权限边界混乱。团队没有先解决这些底层问题，而是加了规划 Agent、执行 Agent、检查 Agent、反思 Agent。结果不是系统变强，而是每个角色都在继承同一份混乱。

所以，多 Agent 的第一条原则是：不要用更多 Agent 掩盖任务结构的不清楚。

多 Agent 只有在任务本身已经呈现出清晰的多角色结构时，才是合理选择。

---

## 2. 先做可拆分性测试

判断一件事能不能上多 Agent，最重要的不是看它难不难，而是看它能不能被稳定切开。

可以用五个问题做可拆分性测试。

第一，任务是否存在天然阶段。

比如研究任务可以拆成资料收集、证据交叉验证、观点归纳和最终写作。复杂代码改造可以拆成影响面分析、接口设计、实现、测试和回归验证。不同阶段有不同输入、不同输出和不同判断标准。

第二，子任务之间是否能并行。

如果每一步都强依赖前一步的完整结论，多 Agent 的并行收益会很小，反而会增加等待和同步成本。

第三，角色之间是否有真实差异。

如果几个 Agent 使用同样上下文、同样工具、同样目标，只是换了名字，它们往往不会带来实质协作，只会重复推理。

第四，中间结果是否能结构化移交。

下游角色必须知道哪些结论已经确定、哪些只是候选、哪些约束不能被改写、哪些证据还需要补充。不能稳定移交的任务，不适合拆成多 Agent。

第五，系统是否有统一收敛机制。

多角色可以并行，但最后必须有人或某个机制负责汇总、裁决、终止和对外交付。否则协作会停留在讨论层面。

只要这五个问题里有几个答不上来，就应该谨慎。

此时更好的选择，往往不是上多 Agent，而是先把任务重构成 workflow，或者先用单 Agent 把状态、工具和失败路径做稳。

## 3. 角色边界是多智能体的第一道工程约束

多 Agent 真正有价值时，往往不是因为每个角色都更聪明，而是因为每个角色只负责一段清楚的工作。

常见角色包括：

- 规划者：拆解目标，制定路线，定义阶段成果；
- 检索者：寻找资料、代码、文档、日志和外部证据；
- 执行者：调用工具、修改文件、生成交付物；
- 检查者：验证结果、发现漏洞、挑战假设；
- 调度者：分配任务、合并结果、处理冲突和决定停止；
- 观察者：记录过程、监控异常、生成审计轨迹。

这些角色不是随便命名就成立。

每个角色都应该有明确的输入、输出、权限、停止条件和责任边界。

比如“检查者”不能只是再读一遍执行者的输出，然后说“看起来没问题”。它必须知道自己检查什么：事实一致性、代码行为、测试覆盖、引用来源、合规风险，还是最终交付质量。

又比如“规划者”不能在执行过程中不断重写目标，否则执行者会失去稳定约束。规划者可以调整路线，但必须说明为什么调整、影响哪些子任务、是否需要重新审批。

多 Agent 的角色边界越模糊，协作越容易退化成三种问题：

- 责任重叠：多个角色都以为自己在负责同一件事；
- 责任空洞：关键节点没有任何角色明确负责；
- 责任漂移：角色在过程中不断扩大或缩小自己的职责。

稳定的多 Agent 系统，通常不是“大家都参与所有事”，而是每个角色都清楚知道自己不该做什么。

## 4. 通信协议和共享状态，是这类系统的地基

单 Agent 的主要复杂度集中在一个执行体内部。

多 Agent 一旦出现，系统就必须回答一组新的问题：

- 谁可以和谁通信；
- 通信内容用什么格式；
- 哪些状态是共享的；
- 谁有权修改共享状态；
- 冲突如何记录和裁决；
- 下游角色如何判断上游输出是否可信；
- 哪些信息必须进入审计记录。

因此，多 Agent 首先是协议设计问题，其次才是模型能力问题。

如果协议松散、共享状态混乱、责任边界不明，系统会在沟通成本里迅速失速。

最常见的错误，是让 Agent 之间只靠自然语言长段对话移交。

自然语言适合解释，但不适合作为唯一状态载体。因为它容易含糊、容易遗漏约束，也容易让下游角色把“建议”“假设”“事实”“决定”混在一起。

更稳的做法，是让关键中间结果结构化。

例如一个研究 Agent 的输出不应只是“我认为 A 更好”，而应该包含：

- 结论；
- 支持证据；
- 反对证据；
- 置信程度；
- 未验证假设；
- 待补充信息；
- 对下游角色的明确请求。

一个代码执行 Agent 的输出也不应只是“修改完成”，而应该包含：

- 修改范围；
- 行为变化；
- 已运行的验证命令；
- 未验证风险；
- 需要检查者重点看的区域。

共享状态可以采用黑板模式、任务队列、结构化 artifact、事件日志或显式 memory，但无论采用哪种形式，都要避免一个危险：所有角色都能随意改写公共事实。

一旦公共状态被随意覆盖，多 Agent 就会变成难以复盘的协作聊天，而不是可治理的执行系统。

## 5. Supervisor 和 peer-to-peer 不是同一种复杂度

多智能体常见有两种组织形态。

第一种是`supervisor 模式`。

系统里有一个中心调度者，负责任务分配、进度汇总、冲突裁决、失败重试和最终收敛。

它的优势是责任路径更清楚，系统更容易观察和接管。缺点是 supervisor 可能成为瓶颈，也可能因为承担过多细节而失去调度视角。

第二种是`peer-to-peer 模式`。

多个 Agent 相对平等地互相协商、互相请求、互相影响。

它的优势是表面上更灵活，适合探索性更强、角色之间相互依赖较深的任务。缺点是责任更容易扩散，共识更慢，错误定位更困难。

现实里能稳定落地的系统，多半会更偏向带中心调度的结构。

这不是因为 peer-to-peer 在概念上不美，而是因为自由协商会把很多问题放大：

- 谁最后拍板；
- 谁决定停止；
- 谁判定冲突；
- 谁负责对外输出；
- 谁保证共享状态一致；
- 谁在系统失控时暂停执行。

如果这些问题没有明确答案，peer-to-peer 很容易从协作变成消耗。

更实用的结构，往往是“中心调度 + 局部对话”：supervisor 负责收敛和裁决，局部角色在明确边界内交换信息。

这既保留了协作能力，也避免了全局责任失焦。

## 6. 多 Agent 的主要代价，是系统税全面上升

很多人想到多 Agent，先想到 token 成本。

成本当然重要，但它只是表层代价。

多 Agent 真正昂贵的地方，是系统税全面上升。

第一，通信成本上升。

角色越多，中间消息越多，等待、同步、解释和重复确认就越多。并行收益如果不够大，很快会被通信吞掉。

第二，延迟链变长。

一个角色的输出可能成为下一个角色的输入。前面的小误解，会在后面被放大成错误决策。

第三，调试难度上升。

单 Agent 出错，至少还能沿一条轨迹排查。多 Agent 出错时，你要同时看角色输入、角色输出、共享状态、调度决策、通信顺序和外部工具结果。

第四，评测难度上升。

你不能只看最终答案是否正确，还要看协作是否真的贡献了增量收益。否则多 Agent 可能只是用更多资源得到同样结果。

第五，权限治理更复杂。

不同角色是否拥有不同工具权限？检查者能否修改生产环境？检索者能否访问敏感数据？执行者能否绕过 supervisor 直接对外发送？这些问题必须提前设计。

第六，安全边界更长。

角色越多，提示注入、数据泄漏、越权调用和错误传播的路径也越多。一个角色受到污染，可能通过共享状态影响整个系统。

所以，多 Agent 的代价不只是“多几个 API 调用”。

它是通信、同步、权限、观测、调试、评测和安全的一整套复杂度上升。

## 7. 什么时候多 Agent 真的值得

多 Agent 最值得的场景，通常满足以下条件中的大部分。

第一，任务自然可拆。

任务本身就存在不同阶段、不同责任、不同输入输出，而不是为了使用多 Agent 才强行拆开。

第二，子任务可以并行。

多个角色可以同时推进，并且并行结果最后能被稳定合并。

第三，角色能力有真实差异。

比如一个角色擅长检索和证据整理，一个角色擅长代码修改，一个角色擅长测试，一个角色擅长风险审查。差异越真实，协作越有价值。

第四，中间结果可被清楚移交。

每个角色产出的 artifact 能被下游直接使用，而不是需要重新猜测意图。

第五，结果需要独立验证。

在高价值任务里，让检查者使用不同视角、不同证据或不同工具验证执行者的输出，可能显著降低盲点。

第六，仍有统一收敛。

多角色不是为了无限讨论，而是为了更快或更稳地到达一个可交付结果。必须有人或机制决定何时结束、如何合并、怎样对外交付。

典型场景包括：

- 大型研究：检索、筛选、交叉验证、归纳、写作可以分工；
- 复杂代码改造：影响面分析、接口调整、实现、测试、审查可以分工；
- 多来源情报分析：收集、去重、矛盾识别、可信度评估、汇总可以分工；
- 企业流程自动化：资料提取、规则校验、异常升级、审批准备、归档可以分工；
- 安全分析：日志聚合、异常检测、攻击路径推断、修复建议、复核可以分工。

这些场景里，多 Agent 不是为了“更多角色看起来更强”，而是因为工作本身已经要求多种能力和多条证据链。

## 8. 什么时候不该上多 Agent

同样重要的是知道什么时候不要上。

以下场景通常不适合多 Agent。

第一，任务很短。

如果一个单 Agent 在几步内就能完成，拆成多个角色只会增加协调成本。

第二，目标还不清楚。

目标不清时，多 Agent 不会自动澄清目标，只会让不同角色围绕模糊目标各自发挥。

第三，角色没有真实差异。

几个角色只是换了名字，却使用同样上下文、同样工具和同样判断标准，这不是协作，而是重复。

第四，中间状态不可表达。

如果一个角色无法把自己的结论、证据、假设和风险清楚交给下游，多 Agent 会迅速累积解释偏差。

第五，调试和观测能力不足。

没有 trace、事件日志、角色输入输出记录和共享状态版本，系统一旦出错很难复盘。

第六，权限边界还没设计。

如果每个角色都拥有过宽权限，系统会更危险；如果权限过窄，又会频繁卡住。权限设计没准备好时，不要急着加角色。

一个实用判断是：

如果你无法写清楚每个 Agent 的“输入、输出、权限、成功标准、失败处理、交接对象”，那就还不应该做多 Agent。

## 9. 多智能体评测必须单独设计

评测单 Agent 时，通常可以先看任务是否完成、答案是否正确、工具调用是否成功。

评测多 Agent 时，这还不够。

你还要评测协作本身是否值得。

至少需要看八类指标。

第一，结果质量是否提升。

多 Agent 是否带来了更高准确率、更少遗漏、更强鲁棒性，还是只是让输出更长。

第二，总成本是否可接受。

包括 token、工具调用、外部 API、延迟和人工复核成本。

第三，协作链是否过长。

如果角色之间大量重复解释、互相等待、循环争论，系统税可能已经超过收益。

第四，角色边界是否稳定。

执行者是否越权做决策，检查者是否变成第二个执行者，supervisor 是否陷入细节。

第五，中间状态是否可复盘。

每个关键结论是否能追溯到输入、证据、工具调用和角色判断。

第六，冲突处理是否稳定。

当两个角色给出相反结论时，系统是否有明确裁决机制。

第七，接管成本是否可控。

人类需要介入时，能否快速看懂当前状态、角色分歧和下一步选择。

第八，相比单 Agent 是否有增量收益。

这是最容易被忽略的基线。多 Agent 不是和想象中的弱系统比较，而要和一个认真设计过的单 Agent 或 workflow 比较。

如果多 Agent 没有显著超过这个基线，就不值得引入。

## 10. 多 Agent 最需要的不是热闹，而是收敛

多智能体协作最容易出现一种错觉：讨论越多，系统越认真；角色越多，结果越可靠。

真实情况往往相反。

如果没有明确收敛机制，多角色会制造大量看似合理的中间产物，却没有人对最终答案负责。

好的多 Agent 系统，必须持续回答三个问题：

- 当前讨论是否正在减少不确定性；
- 新增角色是否带来了新证据或新能力；
- 系统是否正在接近一个可交付结果。

如果答案是否定的，就应该停止协作，回到单 Agent、workflow 或人工决策。

收敛机制通常包括：

- 明确的完成定义；
- 最大轮次或最大成本限制；
- supervisor 裁决；
- 冲突升级；
- 人工复核；
- 最终验收标准。

多 Agent 的目标不是让角色一直协作，而是让正确的角色在正确阶段产生足够好的中间结果，然后尽快合并成可交付输出。

## 11. 多 Agent 自然引出人在环

多 Agent 能扩大系统能力，也会扩大系统需要治理的范围。

角色越多，越需要回答：

- 谁可以批准高风险动作；
- 谁可以改写任务目标；
- 谁能访问敏感信息；
- 谁负责冲突裁决；
- 谁能终止任务；
- 谁对最终交付负责。

这些问题不能全部交给 Agent 自己决定。

一旦多 Agent 系统开始影响真实业务，人在环就不再是附加功能，而是控制结构的一部分。

多 Agent 负责把复杂任务拆开、并行推进和局部专业化；人在环负责在高风险、不可逆、价值判断密集和边界不清的节点上保留责任。

这正是下一章要展开的问题。

当系统从单个执行体变成多角色协作网络时，人类不一定要介入更多日常细节，但必须站在更关键的控制位置上：审批、接管、复核、升级和验收。

---

## 12. 容易误判的地方

### 误解一：多智能体自动比单 Agent 更高级

不是。

没有清晰分工时，它通常只是更贵、更慢、更难调。

### 误解二：多角色讨论就等于更可靠

不一定。

没有协议、证据、责任边界和收敛机制，讨论只会制造更多噪声。

### 误解三：只要能并行，就应该上多 Agent

并行只是条件之一。

如果中间结果不可稳定移交，并行收益会被协调成本吃掉。

### 误解四：检查 Agent 可以自动解决质量问题

检查者只有在拥有清晰标准、独立证据和真实否决权时才有意义。

否则它只是把同一份错误再包装一遍。

### 误解五：多 Agent 能替代治理

不能。

角色越多，越需要权限、审计、冲突裁决、人工接管和最终责任归属。

## 13. 本章小结

1. 多 Agent 不是单 Agent 的自然升级，而是一种更重的系统架构选择。
2. 多 Agent 成立的前提，是任务可拆、角色边界清楚、中间结果可移交，并且存在统一收敛机制。
3. 通信协议和共享状态治理，是多智能体系统的地基。
4. Supervisor 模式和 peer-to-peer 模式复杂度不同，真实落地更常依赖中心调度与局部协作的组合。
5. 多 Agent 的主要系统税，不只是调用成本，而是通信、延迟、权限、调试、评测和安全边界全面上升。
6. 多 Agent 的评测必须和单 Agent 或 workflow 基线比较，否则无法证明它真的值得。
7. 多 Agent 能扩大系统能力，也会扩大责任边界，因此自然引出下一章的人在环。

核心判断：

> 多智能体值得做的时刻，不是你想把系统讲得更未来，而是任务本身已经要求你认真设计多角色分工，并且你有能力治理这份分工带来的复杂度。
