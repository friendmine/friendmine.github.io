# 第123章 Agent 时代的新安全问题

## 先说结论

Agent 时代最危险的新变化，是系统不再只输出内容，而开始理解环境、调用工具并采取行动，这让安全问题从内容治理升级成执行治理。

## 1. Agent 带来的不是“更多风险名词”，而是风险层级的整体上移

单纯 LLM 的主要风险，大多仍集中在内容层：

- 内容错误
- 信息泄露
- 不良输出

Agent 则进一步具备了新的能力：

- 读取环境
- 调用工具
- 写入状态
- 触发动作

这意味着安全问题不再只是“它说了什么”，还会迅速变成“它做了什么”。  
系统一旦开始触碰真实对象，风险层级就整体上移了。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/123_new_security_issues_in_the_agent_era_deep_revised__mermaid_01.png)

> 图123-1 风险层级为什么会整体上移  
> Agent 时代的关键变化，不是风险名称变多了，而是错误开始沿着计划、执行、状态和现实世界一路下沉，后果等级随之整体抬高。

---

## 2. 提示注入在 Agent 场景下更危险，因为误导会被转化成动作

如果模型只是回答问题，被恶意提示诱导已经足够麻烦。  
但如果模型还能执行，注入就不再只是认知层攻击，而会进一步诱导系统：

- 调错工具
- 暴露敏感信息
- 绕开正常工作流
- 自动执行不该执行的步骤

所以 Agent 对提示注入的脆弱性更值得担心。  
不是因为注入变成了新概念，而是因为被误导后的后果已经不再停留在文本里。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/123_new_security_issues_in_the_agent_era_deep_revised__mermaid_02.png)

> 图123-2 Agent 时代的核心攻击面  
> Agent 的危险不在某一类攻击单独有多新，而在于输入、工具、权限、记忆这几类攻击面一旦汇入同一条决策链，最后就可能被放大成现实动作。

---

## 3. 工具投毒之所以危险，是因为 Agent 往往会把工具返回当成世界事实

Agent 信任的不只是用户输入，它还高度依赖工具返回。  
而一旦工具输出被污染、伪造、截断或恶意嵌入指令，模型就可能把这些内容当成可信事实继续推进。

这意味着工具链本身也成为核心攻击面。  
攻击者不一定需要直接欺骗模型，他只要能污染模型眼中的环境，就足以让后续整个决策链偏航。

从这个角度看，工具投毒攻击的本质并不是破坏某个接口，而是在重新书写 Agent 看见的现实。

---

## 4. 越权执行会在 Agent 时代被显著放大，因为系统终于开始“带权限思考”

一旦 Agent 接入真实权限，边界设计错误就会直接转化成现实损害。  
常见后果包括：

- 超范围访问
- 非授权操作
- 高风险动作被自动化触发
- 原本必须审批的流程被绕开

这类问题比普通内容错误严重得多。  
因为它们不只是让用户看到错误信息，而是会真正改写文件、流程、记录或外部状态。

Agent 安全真正让人紧张的地方，就在于系统第一次开始“带着权限思考”。

这也解释了“会行动”为什么会放大风险半径。  
一个只会回答的系统，即使犯错，影响范围常常仍停留在认知层；  
一个会调用工具、会修改状态、会驱动流程的系统，一旦犯错，影响就会沿着权限和工作流继续扩散。  
风险半径因此不再取决于一句话说错有多严重，而取决于系统能把这句话继续变成多少步现实动作。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/123_new_security_issues_in_the_agent_era_deep_revised__mermaid_03.png)

> 图123-3 风险半径的层层放大  
> Agent 风险真正可怕的地方，不是第一步判断失误本身，而是错误会沿着计划、工具、状态和流程逐层放大，最后进入真实世界。

把这张图换成人话，就是一条很朴素的工程事实：  
系统越能持续地把中间判断转成下一步动作，局部误导就越可能积累成系统性事故。  
所以 Agent 安全从来不是“先把模型回答管好”就结束了，而是要不断追问：

- 这个判断接下来会驱动什么动作
- 这个动作会改写什么状态
- 这个状态又会影响后续哪些动作

只有把这三层问清楚，风险半径才会真正变得可控。

从真实事故的视角看，很多高代价事件也并不是第一步就很惊人。  
更常见的情况是：

- 第一层只多信了一点上下文
- 第二层只多放宽了一点工具权限
- 第三层只少做了一次确认

每一层看起来都像“小问题”，但一旦它们沿着同一条执行链串起来，系统就会从“有点不严谨”迅速滑向“已经造成现实后果”。  
这也是为什么 Agent 安全最忌讳只盯住最显眼的单点漏洞，因为真正把系统击穿的，往往是几处看似普通的松动同时发生。

---

## 5. 隐式数据泄露在复杂 Agent 系统里更常见，也更难察觉

数据泄露不一定表现为整库导出。  
在复杂上下文和多工具系统里，更常见的恰恰是隐蔽泄露：

- 回答里带出敏感片段
- 工具参数中夹带内部信息
- 日志和 trace 中残留敏感字段
- artifact 里保留了原始敏感上下文

这类泄露之所以更危险，是因为它们不像“下载数据库”那样显眼，却常常更容易在正常流程中持续发生。  
Agent 系统连接的组件越多，信息路径越复杂，这类隐式泄露就越值得警惕。

---

## 6. Agent 安全必须系统化，因为单一防线在这里几乎注定不够

Agent 安全不能只靠：

- 内容过滤
- 一条更严的系统提示
- 对模型“足够聪明”的期待

它必须同时覆盖：

- 上下文来源分级
- 工具权限设计
- 行动审批机制
- 结果输出控制
- 审计与回放

这说明 Agent 安全不是“在模型前面挡一层”就能解决的问题。  
它更像是在整条行动链上布防，防止误导一步步被放大成现实后果。

也因此，Agent 安全的真正难点不是找到一个“万能防火墙”，而是给每一层动作都安上边界：

- 输入先分级
- 工具先限权
- 高风险动作先审批
- 输出和痕迹可回放

只有这样，系统即使被误导，也不至于立刻把局部错误放大成整条链的系统事故。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/123_new_security_issues_in_the_agent_era_deep_revised__mermaid_04.png)

> 图123-4 Agent 安全必须沿行动链逐层布防  
> Agent 安全不是在模型前面加一道墙，而是在输入、工具、审批和回放这些不同节点逐层缩小风险半径。

---

## 7. 工程现实：最常见的失败不是单一大洞，而是多处小洞一起漏

现实里，Agent 安全最常见的问题并不是某种特别戏剧化的零日攻击，而是多处看起来都“不至于”的弱点叠加：

- 工具描述过宽
- 提示边界不清
- 上下文来源未标记
- 审批规则过粗
- 日志保留过多明文

这种问题特别难治理，因为它们往往潜伏在正常流程里，很少以明显异常的方式提前示警。  
只有当系统出事时，人们才会发现这些小洞其实已经沿整条链串成了一条真正的攻击路径。

把它压成一个最短事故例子，会更容易看清：  
某个外部网页先把恶意指令塞进可检索内容里；  
系统没有区分来源，直接把这段内容装进上下文；  
模型因此改写了原计划，又调用了本来权限就偏宽的内部工具；  
日志系统还把中间结果完整记了下来。  
单看每一步，都像只是“差一点不严谨”；连在一起，就会变成一次完整的越权与泄漏事故。

这也是为什么 Agent 安全必须按链条治理，而不能只盯着某一个看起来最显眼的模块。  
很多事故并不是“第一层防线完全不存在”，而是每一层都只漏了一点，最后这些“差一点”沿着同一条执行链叠成了真正的系统事故。

进一步说，Agent 安全成熟与否，最后看的也不是你列出了多少攻击名词，而是整条链有没有真正的 owner。  
如果输入污染归内容团队、工具越权归平台团队、日志泄露归基础设施团队、审批失效归业务团队，但没人对“整条动作链的现实后果”负责，那么再多分工也只是在把事故切碎，而不是把风险接住。  
所以 Agent 时代的安全，最终会逼组织回答一个比技术更难的问题：谁对这条会行动的链条承担最终责任。

---

## 8. 容易误判的地方

### 误解一：Agent 安全只是更复杂的内容审核

它已经涉及工具、权限、状态和行动链。

### 误解二：只要模型更聪明，就能自己识别恶意提示和风险

安全边界不能建立在模型自觉上，必须由系统硬约束承接。

### 误解三：工具投毒属于外部系统问题，与 Agent 架构无关

Agent 是否盲目信任工具结果，本身就是架构问题。

## 延伸阅读建议

- 推荐起点：Jonathan Katz / Yehuda Lindell《Introduction to Modern Cryptography》是密码学入口；顺着差分隐私、TEE、MPC、FHE 和 AI assurance 往后看，会更容易形成完整地图。

- 检索关键词：`modern cryptography`、`differential privacy`、`FHE`、`prompt injection`、`agent security`、`AI assurance`。

## 10. 本章小结

1. Agent 把安全问题从内容层提升到了执行层。  
2. 提示注入、工具投毒、越权执行和隐式泄露构成了新的核心风险面。  
3. Agent 安全必须覆盖上下文、工具、权限、输出和审计全链条。  
4. 安全边界不能建立在模型“自己会分辨”这种期待上。  

核心判断：

> Agent 时代的新安全问题，本质上来自一句非常简单的话：系统开始会动了。
