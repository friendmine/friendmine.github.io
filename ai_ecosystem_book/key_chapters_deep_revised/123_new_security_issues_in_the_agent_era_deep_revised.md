# 第123章 Agent 时代的新安全问题

## 先说结论

Agent 时代的新变化，是系统不再只输出内容，而开始读取环境、调用工具、写入状态并触发动作。

这让安全问题从内容治理升级为执行治理。
风险不再只取决于模型说错了什么，还取决于这句话会不会继续驱动工具、权限、状态和现实流程。

一句错误回答本来可能只是信息风险；一旦它被接到行动链上，就可能变成权限风险、数据风险、业务风险甚至物理世界风险。

## 1. Agent 带来的不是“更多风险名词”，而是风险层级的整体上移

单纯 LLM 的主要风险，大多集中在内容层：

- 内容错误
- 信息泄露
- 不良输出
- 误导性建议

Agent 则进一步具备新的能力：

- 读取环境
- 规划步骤
- 调用工具
- 写入状态
- 触发动作
- 在多轮任务中保持记忆

这意味着安全问题不再只是“它说了什么”，而会迅速变成“它做了什么”“它改写了什么”“它把什么带出了边界”。

系统一旦开始触碰真实对象，风险层级就整体上移了。
过去内容错误主要影响用户判断；Agent 错误则可能沿着工具、权限、状态和工作流继续扩散。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/123_new_security_issues_in_the_agent_era_deep_revised__mermaid_01.png)

> 图123-1 风险层级为什么会整体上移
> Agent 时代的关键变化，不是风险名称变多了，而是错误开始沿着计划、执行、状态和现实世界一路下沉，后果等级随之整体抬高。

所以，Agent 安全的第一条判断是：
不要只看模型是否回答正确，而要看回答之后是否会推动行动。
只要系统能行动，安全边界就必须从内容层扩展到执行层。

---

## 2. Agent 的核心攻击面，是输入、工具、权限、记忆和状态汇入同一条决策链

Agent 的危险不在某一类攻击单独有多新，而在于多个攻击面会被接到同一条决策链里。

典型攻击面包括：

- 输入：用户请求、网页、邮件、文档、检索结果
- 工具：搜索、数据库、代码执行器、文件系统、业务 API
- 权限：读取、写入、发送、删除、审批、外部调用
- 记忆：长期偏好、历史上下文、任务状态
- 输出：对用户的回答、生成的 artifact、外部消息和日志

这些东西过去可能分散在不同系统里，各自有边界。
Agent 则会把它们临时装配成一个工作循环。

这带来一个新的安全问题：
一处被污染的输入，可能影响模型计划；
一个过宽的工具，可能把错误计划转成现实动作；
一段不该保留的记忆，可能在未来任务里继续污染判断；
一次过量日志，可能把本来临时出现的敏感信息长期保存下来。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/123_new_security_issues_in_the_agent_era_deep_revised__mermaid_02.png)

> 图123-2 Agent 时代的核心攻击面
> Agent 的危险不在某一类攻击单独有多新，而在于输入、工具、权限、记忆这几类攻击面一旦汇入同一条决策链，最后就可能被放大成现实动作。

因此，Agent 安全不能只问“有没有过滤输入”。
它还要问：

- 这个输入会不会进入计划
- 计划会不会触发工具
- 工具会不会改写状态
- 状态会不会影响后续任务
- 输出和日志会不会把敏感信息带出边界

这才是 Agent 时代更真实的风险结构。

---

## 3. 提示注入在 Agent 场景下更危险，因为误导会被转化成动作

如果模型只是回答问题，被恶意提示诱导已经足够麻烦。
但如果模型还能执行，提示注入就不再只是认知层攻击，而会进一步诱导系统：

- 调错工具
- 暴露敏感信息
- 绕开正常工作流
- 执行不该执行的步骤
- 把错误状态写入系统

所以 Agent 对提示注入的脆弱性更值得担心。
不是因为注入变成了全新概念，而是因为被误导后的后果已经不再停留在文本里。

在普通聊天系统中，恶意提示可能让模型说错话。
在 Agent 系统中，同样的恶意提示可能让系统发送邮件、修改文件、访问数据库、创建工单或调用外部 API。

这就是风险半径的关键变化：
注入不再只污染回答，而可能污染行动链。

---

## 4. 工具和环境会成为攻击面，因为 Agent 往往会把工具返回当成世界事实

Agent 信任的不只是用户输入，它还高度依赖工具返回和环境状态。
工具告诉它搜索结果是什么，数据库里有什么，文件是否存在，任务是否完成，外部系统返回了什么错误。

一旦这些返回被污染、伪造、截断、过量暴露，或者夹带恶意指令，模型就可能把它们当成可信事实继续推进。

这意味着工具链本身也成为核心攻击面。
攻击者不一定要直接欺骗模型，他只要能污染模型眼中的环境，就可能让后续决策链偏航。

从这个角度看，工具投毒的本质不是破坏某个接口，而是在重新书写 Agent 看见的现实。

这里有一个重要工程原则：
工具返回不应被默认视为可信事实，更不应被默认视为可执行指令。
工具层需要来源标记、结构化返回、字段裁剪、参数校验和独立审计。
否则，工具就会从“帮助模型理解世界”变成“把外部污染引入决策链”的入口。

---

## 5. 越权执行会被显著放大，因为系统开始“带权限思考”

一旦 Agent 接入真实权限，边界设计错误就会直接转化成现实损害。

常见后果包括：

- 超范围访问
- 非授权操作
- 高风险动作被错误触发
- 原本需要审批的流程被绕开
- 不该写入的状态被写入
- 不该外发的信息被发送出去

这类问题比普通内容错误严重得多。
因为它们不只是让用户看到错误信息，而是会改写文件、流程、记录、账户、审批或外部系统状态。

Agent 安全让人紧张的地方，就在于系统开始“带着权限思考”。
模型不再只是判断“我该怎么回答”，而是在某些系统里变成“我该调用哪个工具、用什么参数、执行哪一步”。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/123_new_security_issues_in_the_agent_era_deep_revised__mermaid_03.png)

> 图123-3 风险半径的层层放大
> Agent 风险的关键，不是第一步判断失误本身，而是错误会沿着计划、工具、状态和流程逐层放大，最后进入真实世界。

把这张图换成工程语言，就是一条很朴素的事实：
系统越能持续地把中间判断转成下一步动作，局部误导就越可能积累成系统事故。

所以 Agent 安全不是“先把模型回答管好”就结束了，而是要不断追问：

- 这个判断接下来会驱动什么动作
- 这个动作会使用什么权限
- 这个动作会改写什么状态
- 这个状态又会影响后续哪些动作

只有把这几层问清楚，风险半径才会变得可控。

## 6. 隐式数据泄漏在复杂 Agent 系统里更常见，也更难察觉

数据泄漏未必表现为整库导出。
在复杂上下文和多工具系统里，更常见的是隐蔽、持续、分散的泄漏：

- 回答里带出敏感片段
- 工具参数中夹带内部信息
- 工具返回被原样复述
- 日志和 trace 中残留敏感字段
- artifact 里保留原始敏感上下文
- 记忆系统长期保存不该保存的信息
- 外部 API 收到超出任务所需的上下文

这类泄漏之所以更难处理，是因为它们不像“下载数据库”那样显眼，却常常更容易在正常流程中持续发生。
Agent 系统连接的组件越多，信息路径越复杂，这类隐式泄漏就越值得警惕。

从隐私 AI 的系统设计视角看，这也承接了上一章的判断：
隐私不是一个单点算法问题，而是数据在整个系统路径中如何流动、暴露、缓存、记录和删除的问题。
Agent 让这条路径变得更动态，因此也更需要数据最小化、上下文裁剪、日志脱敏和输出检查。

---

## 7. Agent 安全需要沿行动链逐层布防，而不是寻找单一防火墙

Agent 安全不能只靠：

- 内容过滤
- 一条更严的系统提示
- 对模型“足够聪明”的期待
- 事后人工复盘

它需要同时覆盖：

- 上下文来源分级
- 工具权限设计
- 行动审批机制
- 结果输出控制
- 状态写入边界
- 审计与回放

这说明 Agent 安全不是“在模型前面挡一层”就能解决的问题。
它更像是在整条行动链上布防，防止误导一步步被放大成现实后果。

也因此，Agent 安全的难点不是找到一个万能防火墙，而是给每一层动作安上边界：

- 输入先分级
- 工具先限权
- 高风险动作先审批
- 状态写入先校验
- 输出和痕迹可回放

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/123_new_security_issues_in_the_agent_era_deep_revised__mermaid_04.png)

> 图123-4 Agent 安全需要沿行动链逐层布防
> Agent 安全不是在模型前面加一道墙，而是在输入、工具、审批和回放这些不同节点逐层缩小风险半径。

只有这样，系统即使被误导，也不至于立刻把局部错误放大成整条链的系统事故。

---

## 8. 工程现实：常见失败不是单一大洞，而是多处小洞沿同一条链叠加

现实里，Agent 安全常见的问题并不是某种特别戏剧化的零日攻击，而是多处看起来都“不至于”的弱点叠加：

- 工具描述过宽
- 提示边界不清
- 上下文来源未标记
- 检索结果未分级
- 审批规则过粗
- 记忆写入过于随意
- 日志保留过多明文

这种问题特别难治理，因为它们往往潜伏在正常流程里，很少以明显异常的方式提前示警。
只有当系统出事时，人们才会发现这些小洞其实已经沿整条链串成了一条攻击路径。

一个简短事故例子可以看清这个结构：
某个外部网页先把恶意指令塞进可检索内容里；
系统没有区分来源，直接把这段内容装进上下文；
模型因此改写了原计划，又调用了本来权限就偏宽的内部工具；
工具返回了过量内部数据；
日志系统还把中间结果完整保存了下来。

单看每一步，都像只是“差一点不严谨”；连在一起，就会变成一次完整的越权与泄漏事故。

进一步说，Agent 安全成熟与否，最后看的也不是列出了多少攻击名词，而是整条链有没有明确 owner。
如果输入污染归内容团队，工具越权归平台团队，日志泄漏归基础设施团队，审批失效归业务团队，但没人对“整条行动链的现实后果”负责，那么再多分工也只是在把事故切碎，而不是把风险接住。

所以 Agent 时代的安全，会逼组织回答一个比技术更难的问题：
谁对这条会行动的链条承担责任。

## 9. 容易误判的地方

### 误解一：Agent 安全只是更复杂的内容审核

它已经涉及工具、权限、状态、记忆和行动链。
内容审核只能管“说了什么”，不能完整约束“接下来做了什么”。

### 误解二：只要模型更聪明，就能自己识别恶意提示和风险

安全边界不能建立在模型自觉上，需要由系统硬约束承接。
模型可以参与风险判断，但不能成为唯一防线。

### 误解三：工具投毒属于外部系统问题，与 Agent 架构无关

Agent 是否盲目信任工具结果，本身就是架构问题。
工具返回需要被标记、裁剪、校验和审计，而不能直接当成世界事实。

### 误解四：Agent 会行动，所以只要审批高风险动作就够了

审批很重要，但不够。
输入分级、工具限权、状态隔离、输出裁剪、日志脱敏和责任 owner 同样决定安全边界。

## 10. 本章小结

1. Agent 把安全问题从内容层提升到了执行层。
2. 新风险不只来自模型回答错误，而来自输入、工具、权限、记忆、状态和输出汇入同一条行动链。
3. 提示注入、工具投毒、越权执行和隐式泄漏构成了核心风险面。
4. Agent 安全需要覆盖上下文、工具、权限、状态、输出和审计全链条。
5. 安全边界不能建立在模型“自己会分辨”这种期待上，而要依赖系统级硬约束和明确责任 owner。

核心判断：

> Agent 时代的新安全问题，来自一句很简单的话：系统开始会动了；一旦系统会动，安全就必须从“管回答”升级为“管整条行动链”。
