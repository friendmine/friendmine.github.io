# 第69章 代码模型与工具使用

## 先说结论

第68章讲的是物理世界里的行动闭环。
第69章转向另一类行动环境：数字系统。
在这里，模型的输出不再只是回答，也可以变成代码、命令、API 调用、数据库查询、文件改动和测试执行。

代码模型的价值，不只是它能写出程序片段。
更深一层的变化是：模型开始进入可执行、可验证、可回滚的工作流。
代码有语法、类型、依赖、测试、权限、运行时和错误反馈，这让模型第一次大规模接触一种结构严密的行动环境。

工具使用也是同一条线。
当模型调用搜索、计算器、编译器、浏览器、数据库或企业 API 时，它开始把参数外的能力接进任务流程。
这会显著扩展能力，也会把风险从语言层推向执行层。
所以，代码模型和工具调用的成熟，不在于“模型更会说”，而在于它能否在真实系统约束下安全地做事。

## 1. 代码是可执行的语言

自然语言可以表达意图，但反馈常常模糊。
代码也是符号序列，却有更严格的结构：语法、类型、依赖、接口、运行环境和测试。

这让代码成为一种很特殊的训练与应用对象。
模型生成一段自然语言说明，读者可能觉得有道理，也可能觉得不够好；
模型生成一段代码，系统可以编译、运行、测试、报错。

这种可执行性改变了模型和任务之间的关系。
模型不只是生成看起来合理的文本，而是在一个能给出清晰反馈的环境里工作。

因此，代码模型不只是程序员效率工具。
它让大模型进入了“可执行符号系统”的前场。

---

## 2. 代码任务提供了比自然语言更清晰的反馈闭环

代码补全、bug 修复、重构、测试生成、日志分析和依赖升级，都有一个共同特点：结果可以被检查。

检查方式包括：

- 语法解析是否通过
- 类型检查是否通过
- 单元测试是否通过
- 集成测试是否通过
- 运行日志是否正常
- 性能和安全扫描是否触发警告

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/69_code_models_and_tool_use_deep_revised__mermaid_01.png)

> 图69-1 代码任务跑得快，是因为它比开放自然语言更早拥有结构化反馈闭环

这种反馈闭环让模型有机会从“生成候选答案”进一步走向“观察结果后修正”。
它也解释了为什么代码能力会成为大模型能力进化中的高价值方向：代码世界给模型提供了可验证的行动场。

---

## 3. 代码模型的难点在仓库上下文，而不只是语法

早期演示里，模型写出一个函数就很惊艳。
但真实工程里，代码很少孤立存在。

一个代码改动通常依赖：

- 当前仓库结构
- 已有接口和抽象
- 依赖版本
- 测试约定
- 构建系统
- 代码风格
- 权限和部署流程

所以，代码模型的现实难点，不是能不能写一段语法正确的函数，而是能不能在真实仓库里理解局部改动和全局约束。
一个补丁能跑过本地测试，也还要考虑兼容性、可维护性、回滚和发布风险。

这也是代码模型从补全工具走向工程 Agent 时会变难的原因。
它不再只面对语言和语法，而是在面对一个长期演化的系统。

---

## 4. 工具调用让模型输出“动作”

工具调用的变化在于，模型不只是输出答案，而是输出下一步动作。

动作可以是：

- 调用一个函数
- 查询数据库
- 执行命令
- 打开浏览器
- 运行测试
- 写入文件
- 读取外部知识库

函数、API、命令行、SQL、浏览器自动化，都在描述如何调用外部能力。
代码模型和工具使用因此有很强关联：它们都要求模型理解任务、选择接口、组织参数、读取结果，再决定下一步。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/69_code_models_and_tool_use_deep_revised__mermaid_02.png)

> 图69-2 代码模型与工具使用的基本闭环
> 代码模型靠近 Agent 的地方，不在于它能写出一段语法正确的程序，而在于它开始进入理解任务、组织调用、接回反馈、继续修正的执行闭环。

工具使用不是把工具名写进提示词。
可用系统需要清楚的接口、参数规则、结果解析和错误恢复。

---

## 5. Schema、协议和权限决定工具是否可控

工具调用看起来像“模型会调函数”，实际需要一层严格接口。
如果接口不清楚，模型很容易选错工具、填错参数、误解返回值或越过权限边界。

一个可靠工具接口通常需要说明：

- 工具能做什么
- 输入参数是什么
- 参数有哪些约束
- 返回值如何解释
- 出错时如何处理
- 调用是否需要权限
- 调用后是否会改变外部状态

这就是 schema 和协议的价值。
它们把自然语言意图转换成系统可执行的边界。

权限也很关键。
读取文件、写入文件、发送邮件、删除资源、调用付费 API，风险级别不同。
工具系统需要区分只读、可写、高风险和需要人工确认的动作。

---

## 6. 关键跃迁是“执行、观察、修正”

一次性生成代码并不等于进入工作流。
更有价值的闭环是：

1. 理解任务
2. 读取上下文
3. 生成改动或工具调用
4. 执行检查
5. 观察错误
6. 修正方案
7. 必要时回滚或请求确认

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/69_code_models_and_tool_use_deep_revised__mermaid_03.png)

> 图69-3 代码模型推动的跃迁，是从输出结果走向输出动作

这个闭环让模型从“建议者”更接近“执行者”。
但执行者需要承担更多工程约束：测试是否充分，错误是否被正确理解，修复是否引入新问题，改动是否可回滚。

这也是代码模型为什么会推动 Agent 形态。
Agent 不是更长的回答，而是持续地在任务、工具、反馈和修正之间循环。

---

## 7. 沙箱、测试和回滚是工具系统的安全底座

工具把模型从语言层推向执行层，也把风险推进了系统层。

常见风险包括：

- 执行了错误命令
- 修改了错误文件
- 使用了错误权限
- 误读了工具返回结果
- 造成资源消耗或数据泄露
- 把一次小错误放大成连锁故障

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/69_code_models_and_tool_use_deep_revised__mermaid_05.png)

> 图69-5 工具使用不会自动带来可靠行动，它只是把风险从语言层推进到了执行层

因此，工具系统需要沙箱、测试、权限、审计和回滚。
沙箱限制影响范围；测试验证结果；权限控制动作边界；审计记录发生了什么；回滚提供恢复路径。

缺少这些保护，工具越多，系统越像在扩大风险表面。

---

## 8. 外部工具让模型借用参数外能力

单靠参数知识，模型会遇到边界。
知识会过时，计算可能不准，企业系统状态会变化，外部世界也不会被完整压进参数。

工具调用让模型可以借用参数外能力：

- 搜索提供实时信息
- 计算器提供精确计算
- 编译器和测试提供执行反馈
- 数据库提供结构化状态
- 浏览器提供网页交互
- 企业 API 提供业务系统入口
- 文件系统提供项目现场材料

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/69_code_models_and_tool_use_deep_revised__mermaid_04.png)

> 图69-4 工具调用改变的，是模型和参数外真实能力之间的关系

这不是说模型变成无边界系统。
相反，它说明能力开始被重新分工：模型负责理解任务、规划流程和解释结果；工具负责提供外部状态、精确执行和可验证反馈。

这也自然引向第70章。
当模型开始依赖参数外能力时，长上下文、记忆、检索和外部知识就不再是附属功能，而是系统能力的一部分。

---

## 9. 从代码工具到数字 Agent

代码模型和工具使用是数字 Agent 的前置层。
一个数字 Agent 要在软件环境中行动，通常需要：

- 读懂任务和上下文
- 选择合适工具
- 维护中间状态
- 调用外部系统
- 观察结果
- 修正计划
- 控制权限和风险

这和第68章的机器人闭环有相似结构，只是环境不同。
机器人在物理世界里移动和抓取；数字 Agent 在文件、API、浏览器、数据库和代码仓库里行动。

两者都不是一次性回答系统。
它们都需要把感知、状态、计划、动作和反馈串起来。
区别在于，数字系统的反馈更容易记录、测试和回滚；物理系统的反馈更昂贵，也更难恢复。

---

## 10. 本章小结

1. 代码是可执行语言，给模型提供了比开放自然语言更清晰的反馈。
2. 代码模型的难点不只是语法，而是真实仓库、依赖、测试和工作流约束。
3. 工具调用让模型从输出文本走向输出动作。
4. Schema、协议和权限决定工具调用是否可控。
5. 执行、观察、修正构成代码模型走向 Agent 的基本闭环。
6. 沙箱、测试、审计和回滚是工具系统的安全底座。
7. 外部工具让模型借用参数外能力，也让长上下文、记忆和外部知识变得更重要。
8. 代码模型和工具使用，是数字系统中可执行 Agent 的前置能力。

核心判断：

> 代码模型的意义，不是模型更会写程序，而是它开始进入可执行、可验证、可回滚的数字系统闭环；工具调用进一步把模型从回答者推向行动者。
