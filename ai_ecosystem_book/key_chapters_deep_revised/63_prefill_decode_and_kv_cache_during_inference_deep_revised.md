# 第63章 推理阶段：prefill、decode 与 KV cache

## 先说结论

如果模型在生成每个 token 时都要把全部历史重新完整计算一遍，推理几乎不可能在现实成本和时延里交付。  
KV cache 的作用，就是把前面已经算过的关键信息暂时留下来，避免每一步都从零开始；而它带来的代价，则会直接落到显存、带宽和并发能力上。
大模型真正难交付的，往往不是“它会不会”，而是“它能不能在现实延迟和成本里稳定地会”。推理系统最后常常不再是单纯技术优化题，而会变成一组产品优先级管理题：先保首字延迟、先保吞吐、先保成本，还是先保长上下文体验。

---

## 1. 训练决定能力上限，推理决定用户到底能不能碰到这份能力

很多讨论喜欢停在训练和参数规模上，好像只要模型够强，产品就会自然跟上。  
现实完全不是这样。

一旦模型进入真实服务系统，问题立刻变成：

- 首次响应要多快
- 一次请求要花多少钱
- 并发上来时能否撑住
- 长上下文会不会把系统拖垮

也就是说，训练主要决定模型会什么，推理才真正决定用户能否稳定、实时、可负担地用到这些能力。  
从这个角度看，推理不是训练后的附属过程，而是模型商业化的第一现场。

---

## 2. 先把两个阶段分开：prefill 和 decode 根本不是同一种系统问题

推理阶段常被一句“模型在生成”带过去，但真正的系统现实并不是一段连续均匀的计算，而是两个性质明显不同的阶段。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/63_prefill_decode_and_kv_cache_during_inference_deep_revised__mermaid_01.png)

> 图63-1 prefill 与 decode 的基本时序  
> 这张图想强调的不是名词，而是系统性质：prefill 更像整段读入，decode 更像逐步在线生成。

这一区分非常关键。  
很多用户感受到的“慢”，其实并不是一个统一问题，而是这两个阶段在不同地方出了问题。

---

## 3. prefill 的本质，是先把用户给的一整段上下文读进模型

当用户输入一大段内容，模型不会立刻开始一个字一个字往外吐。  
它首先必须整体处理已有上下文，把这些 token 编码成当前计算状态。  
这个阶段就是 prefill。

prefill 的特点很鲜明：

- 要一次性处理整段输入
- 计算量通常较大
- 更像吞吐型问题，而不是交互节奏问题

你可以把它理解成，模型先把舞台搭好、把已有材料都读完，之后才有资格真正开始回答。  
因此，长提示词、长文档、长历史记录对首 token 延迟的影响，很多都首先发生在这里。

从复杂度直觉上说，如果上下文长度是 `n`，attention 的全局关系计算通常会让 prefill 呈现出接近：

$$
O(n^2)
$$

的压力。  
这也是为什么上下文一长，首 token 等待时间会明显上涨。

## 4. decode 是另一种完全不同的挑战，因为用户在盯着它一字一字发生
当模型开始逐 token 生成时，进入的是 decode 阶段。  
它和 prefill 虽然连着，但系统性质明显不同。

在 decode 里：

- 每一步只多生成一个 token
- 每一步都必须足够快
- 用户会直接感知卡顿和节奏

这意味着 decode 更像低延迟服务问题，而不只是计算问题。  
模型也许很强，但如果 decode 每一步都慢，用户体验就会立刻变差。  
所以交互式 LLM 的“像不像会对话”，很大程度上也取决于 decode 是否顺滑。

如果没有缓存，decode 每一步几乎都要重新回看前面全部历史，成本会很快失控；  
而有了缓存之后，单步计算更接近在“旧状态基础上只处理新增部分”。  
所以 decode 的关键，不只是算力，而是如何把每一步的重复劳动压到最低。

这也是为什么很多人第一次做在线服务时会被“模型明明不小，为什么用户却觉得它卡”这件事打醒。  
系统并不是在一次性算出整段答案，而是在不断做细小、重复、必须马上返回的在线步骤。  
一旦这些步骤里的任何一环节奏不稳，用户体感会立刻崩掉。

---

## 5. KV cache 之所以重要，是因为它把“每次都重算一遍”变成了“尽量别再白算”

如果模型每生成一个新 token，都把前面所有内容重新完整过一遍 attention，成本会高得惊人。  
KV cache 的出现，就是为了避免这种重复计算。  
它用额外的显存换取更低的重算成本，但代价也很明确：上下文越长、并发越高，缓存本身就越可能反过来成为推理系统的主要压力源。

也正因为重算代价太高，系统才必须把之前算过的 Key 和 Value 缓存起来，后续生成时尽量直接复用。  
这能显著减少重复计算，但也意味着显存会被长上下文和高并发迅速吞掉。所谓“实时聊天体验”，背后其实依赖的是一套非常昂贵的缓存策略。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/63_prefill_decode_and_kv_cache_during_inference_deep_revised__mermaid_02.png)

> 图63-2 KV cache 的最小工作方式  
> 它并没有让模型“少理解一点”，而是在尽量避免每生成一个 token 就把已经算过的历史再完整重算一遍。

从系统视角看，KV cache 几乎是现代自回归模型能否高效交互的前提之一。  
没有它，聊天系统的现实可用性会被迅速摧毁。

---

## 6. 为什么用户体感延迟主要在推理系统层形成

很多用户把“模型聪不聪明”和“模型快不快”混在一起。  
这其实是两个不同层的问题。

用户体感里最明显的通常有两类延迟：

- `TTFT`：time to first token，首个 token 出来的时间
- `TPS`：tokens per second，后续输出 token 的生成节奏

前者更容易被 prefill、排队、服务调度、长上下文拖累；  
后者则更容易被 decode、KV cache、内存带宽、并发策略拖累。

这也是为什么很多“模型很强但不好用”的问题，最后都不是模型本体问题，而是推理系统问题。  
用户感知到的体验，往往首先是推理层交付出来的，而不是训练曲线本身。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/63_prefill_decode_and_kv_cache_during_inference_deep_revised__mermaid_03.png)

> 图63-3 用户体感延迟往往主要在推理系统层形成，而不是训练层  

如果要把这两个指标和用户感受压成最短映射，可以这样记：  
`TTFT` 决定用户“多久看到第一下响应”，所以它最像聊天产品里的起步迟钝感；  
`TPS` 决定后续输出是流畅往外冒，还是一字一顿地挤出来，所以它更像持续交流时的节奏感。  
很多系统之所以“看起来不聪明”，并不是答案错了，而是 `TTFT` 太长让用户以为系统没反应，或者 `TPS` 太低让用户觉得它像卡住了一样。

---

## 7. 一旦上下文变长，问题很快就从“算力够不够”变成“带宽和缓存能不能撑住”

长上下文听上去像能力升级，现实里它首先是一种系统压力放大器。  
上下文越长，需要缓存的内容越多，访存和带宽压力越大，decode 时的成本也越难压住。

这就是为什么很多推理瓶颈并不完全来自纯计算量，而来自：

- cache 是否能高效访问
- 内存带宽是否足够
- 批处理如何组织
- 长短请求是否会相互拖累

所以“更长上下文”从来不只是模型宣传词，它会直接改写整套推理系统的代价结构。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/63_prefill_decode_and_kv_cache_during_inference_deep_revised__mermaid_04.png)

> 图63-4 长上下文真正放大的，往往不是模型能力本身，而是缓存与带宽压力

## 8. 推理之所以是系统工程，是因为同一个模型会被不同服务架构放大成完全不同体验

这也是为什么近两年很多推理优化开始从单点技巧，走向更系统的套路。  
例如 `paged attention` 这类做法，重点在于把 KV cache 的管理方式做得更像成熟内存系统；  
`speculative decoding` 这类做法，则是在试图用便宜模型或并行路径先猜，再让主模型做确认。  
它们的共同点不是名字新，而是都在承认：推理问题已经从“怎么算”进入了“怎样调度、怎样复用、怎样减少等待”。

现实中，同一个模型在不同推理系统里可能表现差异极大。  
真正影响用户体验的，不只是权重文件本身，还有：

- 批处理和调度策略
- cache 管理方式
- 张量并行和流水并行
- 请求合并与分流策略
- 服务层限流和回退机制

于是，模型能力只是起点，推理系统才是能力兑现器。  
你可以把它理解为：训练决定模型有多聪明，推理系统决定这份聪明能否被稳定送到用户眼前。

很多工程优化其实都在同一条张力线上找平衡：  
批处理做大了，吞吐会上去，但 `TTFT` 可能变差；  
单请求追求极致低延迟，并发利用率又可能掉下来。  
连续批处理、分页缓存、推测解码这些方法之所以重要，正因为它们不是单点技巧，而是在试图重新组织这条张力曲线。

> 图63-5 同一个模型会被不同推理系统放大成完全不同的用户体验

## 9. 工程里最痛的，往往不是不会答，而是“会答但答不起”

商业化产品最常遇到的麻烦，往往不是模型完全不会，而是：

- 首 token 太慢
- 长上下文请求拖垮整体吞吐
- KV cache 太重导致并发急剧下降
- 不同请求长度混在一起造成系统抖动

这些问题不会出现在能力宣传里，却会直接决定业务能否成立。  
从这个意义上说，推理优化不是锦上添花，而是大模型能否走进真实世界的核心条件之一。

## 11. 本章小结

1. 训练决定能力上限，推理决定能力能否被用户碰到。  
2. prefill 负责整体读入上下文，decode 负责逐 token 生成。  
3. prefill 和 decode 是两种不同的系统问题，不能混成一个“推理速度”指标。  
4. KV cache 通过复用历史计算，使自回归交互变得现实可行。  
5. 长上下文会把问题迅速推向缓存、带宽和调度压力。  
6. 推理系统是模型能力真正的在线兑现器。  

核心判断：

> 大模型最难交付的，并不是“它会不会”，而是“它能不能在真实延迟、并发和成本约束下持续地会”；一旦进入真实服务系统，这件事就会迅速变成产品优先级如何分配的问题。
