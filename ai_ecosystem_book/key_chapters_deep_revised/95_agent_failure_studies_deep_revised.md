# 第95章 Agent 失败学

## 先说结论

真正成熟的 Agent 系统，不是沿着理想成功路径长出来的，而是沿着各种现实失败路径被一点点倒逼出来的。  
如果只看演示成功，不看失败链条，系统一进入真实流程就会暴露出最昂贵的问题。

Agent 失败学之所以重要，不是因为悲观，而是因为行动系统的错误不会停留在答案层。  
一次小的理解偏差、一次工具选择失误、一次权限判断错误，都可能沿执行链被不断放大，最后演化成系统性事故。  
所以失败分析不是附属工作，而是 Agent 进入产业前必须补上的主课。

## 1. 为什么 Agent 必须先研究失败，而不是先沉迷成功演示

问答模型失败时，常见后果是答错、答偏或说得不够好。  
Agent 失败时，后果会更具体，也更昂贵。  
它可能选错工具，误改文件，重复执行，越权操作，或者在错误状态上持续追加动作。

这意味着，Agent 的核心工程问题已经变了。  
不再只是“怎么让它显得更聪明”，而是“当它理解错、判断错、执行错时，系统如何不跟着一起坠落”。

所以，Agent 真正成熟的起点，往往不是第一次演示成功，而是第一次有人认真追问：它会以哪些方式失败。

这也是为什么失败学不是悲观主义，而是产品成熟的开始。  
演示只会告诉你理想路径长什么样；  
失败分析才会告诉你系统真正的边界长什么样。  
一旦系统开始进入真实流程，组织最关心的往往也不是“最好时有多强”，而是“出问题时会坏成什么样、能不能及时刹住”。

> 真正高代价的失败，很少只是“回答错了”。它往往是规划、上下文、工具和控制中的某个局部问题，被放大成了一整条行动链的偏航。

## 2. 幻觉在 Agent 场景里，不再只是信息错误，而会升级成行动错误

在聊天场景里，幻觉往往表现为事实不准、引用虚构或判断失真。  
在 Agent 场景里，这类偏差会直接进入执行链。

系统可能：

- 错误理解外部状态
- 误判某个工具已经成功
- 误以为任务已经完成
- 把不存在的约束当成真实规则

一旦幻觉进入行动环路，它就不再只是知识问题，而会变成执行问题。  
更危险的是，Agent 还常常会以一种高度自信、结构清晰的方式把这种错误说出来，使人误以为系统一切正常。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/95_agent_failure_studies_deep_revised__mermaid_02.png)

> 图95-2 幻觉如何从信息错误升级成行动错误  
> 在问答里，幻觉往往停留在说错；在 Agent 里，它会沿着计划、工具和动作一路下沉，最终变成真实后果。这就是为什么 Agent 失败学必须把幻觉重新理解成执行风险。

---

## 3. 很多失败并不是“一击致命”，而是“链式扩散”

Agent 系统里最典型的失败形态，并不是某一步立刻完全崩溃，而是第一步偏一点，第二步再偏一点，最后整个任务在看似连贯的状态下彻底跑歪。

例如：

- 目标理解错一点
- 导致计划偏一点
- 导致工具选择再偏一点
- 中间结果继续误导后续判断

这类链式失败尤其难发现，因为每一步单独看，似乎都还有点道理。  
等到最终结果明显错误时，系统已经走了很远。  
因此，Agent 最需要的并不是“最后发现错了”，而是在链条中途就能看见它开始偏航。

这也是执行型系统和问答系统最重要的感知差别之一。  
问答出错，用户通常在终点感知错误；  
Agent 出错，系统更需要在过程里识别偏航。  
如果只能在终点发现问题，很多外部动作、成本消耗和状态污染都已经发生了。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/95_agent_failure_studies_deep_revised__mermaid_03.png)

> 图95-3 Agent 失败更常见的形态是链式偏航  
> 这类失败之所以难抓，是因为每一步看上去都“还行”。真正危险的不是某一步突然崩，而是偏差被静悄悄地累加成整条任务链的错误方向。

## 4. 漏上下文，是最常见也最隐蔽的失败来源

很多团队最初会把失败主要归因于模型不够强。  
但真实工程里，一个极常见的根源其实是上下文缺失。

系统漏掉的可能是：

- 一条关键业务约束
- 一个最新状态更新
- 某个权限边界
- 历史中已经被否定过的方案

这种失败之所以危险，是因为从表面上看，Agent 并没有“明显异常”。  
它照样会规划、会执行、会解释，甚至显得很自洽。  
问题只在于，它是在一个残缺世界里进行逻辑推理。

如果把失败再压成更贴近工程的案例，会更容易看清它们为什么难防。

`规划失败` 的典型样子是：  
目标本来要求“先验证再执行”，系统却把“尽快完成”误当成最高优先级，于是直接进入动作阶段。  
它不是不会行动，而是在错误的任务分解上行动。

`工具失败` 的典型样子是：  
系统拿到了一个看似成功的返回值，却没有意识到结果只完成了一半，或者参数范围根本不对。  
后续步骤继续建立在这个“半真半假”的结果上，最后整条链都偏了。

`上下文失败` 的典型样子是：  
新约束已经出现，但上下文窗口里还留着旧规则；或者关键状态更新没有被写回，系统于是继续沿着上一轮世界模型做判断。  
这类失败最容易伪装成“系统很聪明，只是偶尔想偏了”，其实问题出在它看到的是旧世界。

`权限失败` 的典型样子是：  
系统本来只该生成建议，却被放进了可直接执行的动作链；  
或者高风险动作没有审批门槛，于是错误决定立刻转成了现实后果。  
这类失败的本质不是推理差，而是控制层没有把风险半径收住。

---

## 5. 越权、死循环和误触发，是必须被单独对待的高代价失败

并不是所有失败都属于同一等级。  
有些失败只会浪费一点时间，有些失败则可能直接触发现实风险。

其中最典型的三类是：

- 越权：访问或操作了不该接触的资源
- 死循环：不断重复无效动作，持续消耗成本
- 误触发：在错误条件下执行了本不该执行的关键动作

这些问题很难靠更好的 prompt 解决。  
它们本质上属于控制层问题，必须由权限、审批、停止条件和异常检测来硬性处理。  
一旦试图把这些风险完全交给模型自觉约束，系统迟早会出大问题。

从工程角度看，这三类失败最该被当成“默认会发生”的情况，而不是小概率例外。  
也就是说，系统不该问“模型会不会刚好犯这个错”，而该问“如果它犯了，我的结构能不能把损失困在局部”。  
谁还在用“模型应该知道不该这么做”来面对高代价动作，谁其实还没有真正开始做 Agent 工程。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/95_agent_failure_studies_deep_revised__mermaid_04.png)

> 图95-4 三类必须单独治理的高代价失败  
> 这三类失败的共同点，不是“模型答得不好”，而是它们一旦发生就会迅速转成现实代价。因此它们不能只靠提示词缓解，必须由系统结构直接约束。

---

## 6. 失败学真正教我们的，是“约束比炫技更值钱”

许多演示令人兴奋，是因为系统看起来行动自由、反应迅速、甚至带有一点即兴发挥的聪明感。  
但在生产环境里，真正换来稳定性的往往不是更多自由，而是更多合理约束。

例如：

- 缩小工具暴露面
- 提高高风险动作的审批门槛
- 在关键节点强制校验中间状态
- 为异常情况提供明确中止条件

这些设计不够炫，却最有效。  
它们体现的是一个非常朴素的工程原则：  
在真实世界里，可靠往往来自约束，而不是来自无限扩张的能力。

成熟团队后期最明显的变化，往往不是系统更会冒险，而是系统更懂得在什么时候收手。  
他们开始接受一个事实：  
少做一点、慢一点、先卡一下，并不一定意味着体验更差；  
在高代价任务里，它反而常常意味着系统终于开始尊重失败成本。

---

## 7. 一个成熟的 Agent 系统，必须内建“失败后怎么办”

真正需要被设计进去的，不只是成功路径，还有失败之后的制度安排。  
系统必须回答：

- 如果工具返回异常怎么办
- 如果状态不一致怎么办
- 如果计划明显偏航怎么办
- 如果人类想中止怎么办
- 如果任务不能继续推进，如何安全退出

这些问题如果没有提前回答，系统就只是把风险推迟到上线之后。  
而一旦进入真实使用，失败往往不会按演示稿的节奏发生。

所以失败后怎么办，必须在架构阶段就决定，而不是等事故发生后再补制度。  
真正的成熟不是“希望失败少一点”，而是“承认失败一定会来，并提前决定如何把它困住、看见、回放、接管和修复”。  
这也是失败学最后会自然连到后面的观测、评测和调试体系。

> 图95-5 成熟 Agent 必须设计失败后的制度安排

## 8. 工程现实：很多关键进步都来自更懂得止损

现实里，一些 Agent 产品真正变稳，不是因为模型突然发生了质变，而是因为团队开始在失败面前变得更克制。  
他们做的事通常是：

- 限制自由度
- 增加检查点
- 加强权限边界
- 把高风险动作从默认自动中拿出来

这些变化看似保守，实际上才是系统走向成熟的标志。  
因为这意味着团队开始明白，长期价值来自稳定的成功，而不是偶尔亮眼的成功。

---

## 9. 容易误判的地方

### 误解一：失败主要来自模型还不够强

模型能力当然重要，但许多高代价失败其实来自上下文、控制和工作流设计不当。

### 误解二：只要总体成功率高，少量失败可以忽略

在高风险场景里，少量失败就足以摧毁信任。

### 误解三：失败分析是运维阶段才要做的事

真正成熟的系统，会在架构阶段就按失败路径设计。

## 11. 本章小结

1. Agent 的失败会从信息层放大到执行层，因此代价远高于普通问答系统。  
2. 幻觉、链式偏航、漏上下文、越权、死循环和误触发，是最值得警惕的失败形式。  
3. 可靠性通常来自约束、审批、校验和中止机制，而不只是更强模型。  
4. 一个成熟系统必须提前设计“失败后怎么办”，而不是等出事后再补。  

核心判断：

> Agent 真正成熟，不是因为它第一次跑通，而是因为系统终于学会了怎样在它跑偏时及时刹住。
