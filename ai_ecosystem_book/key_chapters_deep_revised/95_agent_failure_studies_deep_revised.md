# 第95章 Agent 失败学

## 先说结论

成熟的 Agent 系统不是沿着理想成功路径长出来的，而是沿着理解偏差、上下文缺失、工具误用、权限越界、状态污染和失败恢复这些现实问题被倒逼出来的。

如果只看演示成功，不研究失败链条，系统一进入真实流程就会暴露出高代价问题。

![插图 第95章 失败类型事故墙]({{ site.baseurl }}/ai_ecosystem_book/images/chapter_095_agent_failure_studies.png)

> 插图 第95章 失败类型事故墙

第94章讨论 Harness，重点是模型如何被组织进真实工作。
本章继续往下走：只要系统开始行动，失败就不再只是“答错”，而会沿着规划、工具、权限、状态和外部副作用被放大。

---

## 1. 为什么 Agent 要先研究失败，而不是只展示成功

问答模型失败时，常见后果是答错、答偏、漏答或说得不够好。
Agent 失败时，后果会更具体，也更容易转成现实代价。

它可能：

- 选错工具
- 误读外部状态
- 传错参数
- 重复执行动作
- 修改不该修改的对象
- 越权访问资源
- 在错误状态上继续推进
- 把半完成任务误判为完成

这意味着 Agent 的核心工程问题已经变化。
问题不再只是“怎样让它更聪明”，而是：
当它理解错、判断错、执行错时，系统怎样避免跟着一起失控。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/95_agent_failure_studies_deep_revised__mermaid_01.png)

> 图95-1 Agent 失败类型树
> 高代价失败很少只是“回答错了”。它往往是理解、规划、上下文、工具、状态和控制层中的局部问题，被放大成一整条行动链的偏航。

失败学不是悲观主义。
它是行动系统进入现实世界前必须补上的基础课。
演示告诉团队理想路径是什么样，失败分析告诉团队系统边界在哪里。

---

## 2. Agent 失败可以按“进入执行链的位置”来分类

研究 Agent 失败，不能只用一句“模型不稳定”概括。
更有用的方式，是看失败从执行链哪个位置进入系统。

常见类型至少包括：

- 意图理解失败
- 上下文装配失败
- 规划失败
- 工具选择失败
- 参数构造失败
- 工具结果解释失败
- 状态更新失败
- 权限控制失败
- 停止条件失败
- 人机交接失败
- artifact 交付失败

这套分类的价值在于定位责任层。

如果是意图理解失败，可能需要改任务澄清和输入界面。
如果是上下文装配失败，可能需要改检索、记忆和状态注入。
如果是工具结果解释失败，可能需要改工具 schema、返回格式和错误语义。
如果是权限控制失败，就不能只靠 prompt，而要改 Harness 控制层。

失败分类越粗，修复越容易误伤。
团队越早把失败拆细，越容易把修复落到正确层。

## 3. 幻觉在 Agent 场景里会从信息错误升级成行动错误

在聊天场景里，幻觉通常表现为事实不准、引用虚构或判断失真。
在 Agent 场景里，幻觉会进入执行链。

系统可能：

- 误以为某个文件存在
- 误以为某个工具已经成功
- 误以为用户已经授权
- 误以为任务已经完成
- 把不存在的约束当成真实规则
- 把不完整结果当成可靠输入

一旦幻觉进入行动环路，它就不再只是知识问题，而会变成执行风险。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/95_agent_failure_studies_deep_revised__mermaid_02.png)

> 图95-2 幻觉如何从信息错误升级成行动错误
> 在问答里，幻觉往往停留在说错；在 Agent 里，它会沿计划、工具和动作一路下沉，进而变成现实后果。

更麻烦的是，Agent 常常会以结构清晰、语气自信的方式解释错误动作。
用户看到的是一条看似合理的执行链，实际里面已经混入了错误假设。

所以 Agent 系统不能只要求模型“少幻觉”。
它还需要状态校验、工具确认、权限检查和关键步骤可回放。

---

## 4. 很多失败不是一步崩溃，而是链式偏航

Agent 失败最典型的形态，往往不是某一步突然崩溃。
而是第一步偏一点，第二步再偏一点，最后整条任务链在看似连贯的状态下走错方向。

例如：

- 用户目标被轻微误解
- 计划因此少了验证步骤
- 工具选择又基于错误计划
- 工具返回被过度解释
- 后续状态继续建立在错误结果上
- 最终交付看似完整，实则偏离目标

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/95_agent_failure_studies_deep_revised__mermaid_03.png)

> 图95-3 Agent 失败更常见的形态是链式偏航
> 这类失败难抓，是因为每一步单独看都似乎有道理。危险的不是突然崩溃，而是偏差被静悄悄地累加成错误方向。

链式偏航有两个工程含义。

第一，系统不能只在终点评估。
如果只到最后才发现错，很多外部动作、成本消耗和状态污染已经发生。

第二，系统需要中间校验点。
计划确认、工具结果校验、状态一致性检查、关键动作审批，都是为了在偏航早期把系统拉回来。

---

## 5. 上下文缺失，是最隐蔽的失败来源之一

很多团队会把失败归因于模型不够强。
但真实工程里，一个常见根因是上下文缺失。

系统漏掉的可能是：

- 最新状态更新
- 用户已经否定过的方案
- 项目里的稳定约束
- 某个权限边界
- 工具返回的警告信息
- 业务流程中的前置条件
- 历史决策背后的原因

这类失败危险在于表面很正常。
Agent 仍然会规划、执行、解释，甚至显得自洽。
问题只在于，它是在一个残缺世界里推理。

从工程上看，上下文失败至少有三种来源：

- 没有取到该取的信息
- 取到了但排序或摘要丢掉了关键信息
- 写回状态失败，导致系统继续沿旧世界行动

所以上下文工程不是简单“塞更多内容”。
它需要回答：什么信息必须稳定进入任务状态，什么信息可以按需检索，什么信息必须被显式确认，什么信息必须写回系统记录。

---

## 6. 工具失败不只是调用失败，也包括“调用成功但语义失败”

很多工具失败并不会以报错形式出现。
工具可能返回 200，动作也执行了，但语义上仍然失败。

典型情况包括：

- 参数格式正确，但范围错了
- 工具返回成功，但只完成部分动作
- 查询结果为空，系统却误以为没有风险
- 工具返回警告，模型没有理解
- 重试造成重复副作用
- 外部系统状态变化，导致后续假设失效

这类失败最容易被端到端完成率掩盖。
系统最后也许交付了结果，但过程里已经发生了不必要的动作、重复写入或错误状态更新。

工具失败治理不能只靠模型“更小心”。
它需要：

- 清晰 schema
- 可读错误码
- 幂等设计
- dry-run
- 参数校验
- 副作用分级
- 高风险动作确认
- 工具结果结构化返回

Agent 一旦连接工具，就进入了软件工程世界。
工具层的契约质量，会直接决定 Agent 的可靠性上限。

---

## 7. 越权、死循环和误触发必须被单独治理

并不是所有失败都属于同一风险等级。
有些失败只是浪费时间，有些失败会直接造成现实损失。

三类失败尤其要单独治理。

第一类是越权。
系统访问或操作了不该接触的资源。
这类问题不能靠提示词兜底，必须由权限层硬约束。

第二类是死循环。
系统不断重复无效动作，持续消耗时间、token、API 调用和外部资源。
这类问题需要预算、步数、重复检测和中止条件。

第三类是误触发。
系统在错误条件下执行了本不该执行的关键动作。
例如发出邮件、提交代码、修改配置、关闭工单、删除数据。
这类问题需要审批、dry-run、双确认和回滚路径。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/95_agent_failure_studies_deep_revised__mermaid_04.png)

> 图95-4 三类需要单独治理的高代价失败
> 越权、死循环和误触发一旦发生，就会迅速转成现实代价。它们不能只靠提示词缓解，必须由系统结构直接约束。

严肃的 Agent 工程，不会问“模型应该知道不该这么做吗”。
它会问：
如果模型真的这么做，系统能不能把损失困在局部。

---

## 8. 状态污染比单次错误更危险

Agent 失败还有一种更隐蔽的后果：
状态污染。

一次错误如果只停留在回答里，用户可能看完就结束。
但如果错误写进项目状态、记忆、artifact、工单、配置或数据库，后续系统会继续基于这个错误行动。

状态污染可能表现为：

- 错误结论被写入项目记忆
- 半完成 artifact 被当成正式版本
- 错误工具结果被保存为事实
- 用户临时偏好被误记为长期规则
- 失败任务的中间状态被后续任务继承

这类问题会把一次错误变成持续错误。

因此，Agent 系统需要区分临时上下文、任务状态、长期记忆和系统记录。
不同层级的写入必须有不同门槛。

越长期、越共享、越会影响后续任务的状态，越需要校验、审批和可回滚。

---

## 9. 失败学教我们的，是约束比展示更重要

很多演示令人兴奋，是因为系统看起来行动自由、反应迅速、几乎什么都能试。
但在生产环境里，稳定性往往来自合理约束。

有效约束包括：

- 缩小工具暴露面
- 按任务分配最小权限
- 对高风险动作强制审批
- 在关键节点校验中间状态
- 设置预算、步数和时间上限
- 为异常情况提供中止条件
- 将副作用动作分成预览和执行两阶段

这些设计不够显眼，却能显著降低风险半径。

成熟团队后期最明显的变化，往往不是系统更会冒险，而是系统更懂得什么时候收手。
少做一点、慢一点、先确认一下，在高代价任务里并不是体验倒退，而是系统开始尊重失败成本。

---

## 10. 成熟系统必须内建“失败后怎么办”

成功路径需要设计。
失败路径也需要设计。

系统要提前回答：

- 工具返回异常怎么办
- 状态不一致怎么办
- 计划明显偏航怎么办
- 人类要求中止怎么办
- 外部系统部分成功怎么办
- artifact 生成失败怎么办
- 任务不能继续推进时如何安全退出
- 失败后哪些信息要保存
- 哪些情况需要人工接管
- 哪些情况必须自动回滚

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/95_agent_failure_studies_deep_revised__mermaid_05.png)

> 图95-5 成熟 Agent 需要设计失败后的制度安排
> 成熟系统不是假设失败不会发生，而是提前决定失败之后怎样收口、隔离、记录、回放、接管和修复。

失败后怎么办，不能等事故发生后再临时补。
它必须进入架构设计。

因为真实失败往往不会按演示稿发生。
它会发生在权限边界、工具状态、用户中断、外部系统异常和长尾任务混在一起的时候。

---

## 11. 事故复盘要从“谁错了”转向“结构哪里没有兜住”

Agent 事故复盘最不该停在“模型为什么这么傻”。

更有价值的问题是：

- 哪一层最早出现偏差
- 哪一层本该发现但没发现
- 哪一层本该阻断但没阻断
- 哪个观测信号缺失
- 哪个权限边界过宽
- 哪个工具契约不清
- 哪个状态写入门槛过低
- 哪个回滚路径不存在

这类复盘会把事故变成系统改进。

如果复盘只归因于模型能力，团队很容易反复等待“下一个更强模型”。
但许多高代价失败并不会因为模型变强自动消失。
它们需要 Harness 层的结构性约束。

这正好回到第94章：
未来 AI 产品的长期价值，不只是模型本身，而是怎样把模型放进可控、可接管、可审计、可交付的工作结构里。

---

## 12. 失败学会自然通向第96章的观测、回放和调试

本章讨论失败类型和失败治理。
但如果系统没有观测、回放和评测能力，失败学也只能停留在经验总结。

团队需要把失败变成材料：

- 保存原始请求
- 保存上下文
- 保存计划
- 保存工具调用
- 保存状态变化
- 保存 artifact
- 保存人工判断
- 保存修复前后对比

只有这样，失败才可以被回放、被比较、被修复、被加入回归样本库。

第96章会继续讨论这一层：
怎样把 Agent 行为记录成可理解、可回放、可评估、可调试的工程对象。

换句话说，失败学告诉我们要怕什么；
观测与回放告诉我们怎样看见它、复现它、修掉它。

## 13. 工程现实：很多关键进步来自更懂得止损

现实里，一些 Agent 产品变稳，不是因为模型突然质变，而是因为团队开始更懂得止损。

他们做的事通常是：

- 限制自由度
- 缩小权限面
- 增加检查点
- 强制 dry-run
- 把高风险动作从默认自动中拿出来
- 给任务设置预算和中止条件
- 把失败样本沉淀进回归集

这些变化看似保守，实际是系统走向成熟的标志。

长期价值来自稳定成功，而不是偶尔亮眼的成功。
Agent 系统越接近真实工作，越要把失败成本当成一等公民来设计。

## 14. 容易误判的地方

### 误解一：失败主要来自模型还不够强

模型能力当然重要，但许多高代价失败来自上下文、工具、权限、状态和工作流设计。

### 误解二：总体成功率高，少量失败可以忽略

在高风险场景里，少量失败就足以摧毁信任。
要看失败类型和风险半径，而不只是平均成功率。

### 误解三：失败分析是上线后的运维工作

不是。
成熟系统会在架构阶段就按失败路径设计权限、审批、停止条件、回放和恢复机制。

### 误解四：加更多 guardrail 就一定更可靠

Guardrail 也需要设计。
过度粗糙的约束会误伤正常任务，过窄的约束又挡不住真实风险。

### 误解五：失败只要修掉当次问题就结束

不够。
失败应该进入回放材料、回归样本和结构性复盘，避免同类问题换一种形式回来。

## 15. 本章小结

1. Agent 失败会从信息层进入执行层，沿规划、工具、权限、状态和副作用被放大。
2. 幻觉、链式偏航、上下文缺失、工具语义失败、越权、死循环、误触发和状态污染，是最值得警惕的失败形式。
3. 可靠性来自约束、校验、权限、审批、停止条件和失败恢复，而不只是更强模型。
4. 成熟系统需要提前设计“失败后怎么办”，并把失败材料交给观测、回放和评测体系。
5. Agent 走向成熟的关键，不是证明它能成功一次，而是证明它跑偏时系统能看见、收住、接管和修复。

核心判断：

> Agent 走向成熟，不是因为它跑通过一次，而是因为系统学会了怎样在它跑偏时及时收住，并把失败变成下一轮可靠性的材料。
