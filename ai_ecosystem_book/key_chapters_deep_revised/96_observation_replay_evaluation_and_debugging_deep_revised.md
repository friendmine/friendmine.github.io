# 第96章 观察、回放、评估与调试

## 先说结论

没有可观察性，Agent 就只是一个会动却难以理解的系统；没有回放、评估和调试体系，团队就无法把偶然成功变成稳定能力。

---

## 1. 观察能力不是运维附件，而是 Agent 工程的起点

普通软件已经需要日志、监控和报警，Agent 系统对这些能力的依赖只会更强。  
原因很简单：它的行为路径更长，状态变化更多，失败也更不稳定。

如果你只能看见最终结果，就几乎无法回答几个最重要的问题：

- 它为什么成功
- 它为什么失败
- 它究竟在哪一步开始偏航

这意味着，可观察性不是把系统上线之后再补的一层工具，而是从第一天开始就应该内建的基础设施。  
没有这层基础设施，后面的一切优化都会变得像猜谜。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/96_observation_replay_evaluation_and_debugging_deep_revised__mermaid_01.png)

> 图96-1 可观测性与回放链路图  
> 一个成熟 Agent 系统，不只记录“发生过什么”，还要能把事件、路径、产物和判断重新接起来，用于回放、评估和调试。只有这样，问题才会从事故变成材料。

---

## 2. Log 记录事件，Trace 还原路径，二者缺一不可

很多团队觉得“有日志”就算有可观察性。  
这远远不够。  
日志和 Trace 解决的是两类不同问题。

Log 更像离散事件记录。它告诉你：

- 调了什么工具
- 传了什么参数
- 返回了什么状态码
- 哪个节点抛了错

Trace 更像任务路径记录。它告诉你：

- 当前任务经过了哪些阶段
- 中间状态如何变化
- 系统何时改了计划
- 哪一段路径开始出现异常

只看 Log，容易只见树木不见森林。  
只看 Trace，又容易缺少关键细节。  
两者结合，系统才真正从“有痕迹”变成“可理解”。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/96_observation_replay_evaluation_and_debugging_deep_revised__mermaid_02.png)

> 图96-2 Log 与 Trace 分别解决什么问题  
> 很多团队以为有日志就够了，但真正的 Agent 调试需要同时看见局部细节和整条路径。少了任何一层，问题都会变得似是而非。

## 3. Checkpoint 和 Replay，把调试从玄学拉回工程
Agent 最让人头疼的一点，在于它的很多错误并不稳定复现。  
同样一个任务，换一次上下文、换一个版本、换一个外部状态，路径就可能完全不同。  
如果没有 checkpoint 和 replay，团队面对问题时经常只能说：“我们再跑一次试试看。”

Checkpoint 的意义，在于把关键时刻的状态保存下来。  
Replay 的意义，在于在接近原始条件的情况下重放这条路径。  
这两者一起工作，才让调试真正具备工程属性。

否则，系统每次失败都像一次独立事故，你很难确认自己修掉的到底是根因，还是某次偶然条件。

这里尤其要把四类观测对象拆开。

- `event`
  指离散事件，例如某次工具调用、某条错误返回、某个状态更新。  
  它回答“具体发生了什么事”。

- `trace`
  指把一连串事件串成可理解路径。  
  它回答“这件事是怎样一步步走到这里的”。

- `artifact`
  指过程中真实生成的对象，例如文档、代码 diff、报告、工单状态。  
  它回答“系统到底留下了什么可继续工作的结果”。

- `judgment`
  指对过程或结果的评价，例如人工判定、规则校验、模型打分、回归标注。  
  它回答“这条路径和这个结果到底好不好、问题出在哪”。

很多系统之所以调不动，就是因为只记录了前两类，没有沉淀后两类。  
结果团队能看到系统“做过什么”，却看不清这些动作最后产出了什么，也很难稳定比较“这次修复到底有没有更好”。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/96_observation_replay_evaluation_and_debugging_deep_revised__mermaid_03.png)

> 图96-3 Checkpoint 与 Replay 怎样把调试变成工程过程  
> 没有这两层，团队往往只能“一遍遍再跑一次看看”。有了 checkpoint 和 replay，失败才会从偶然事故变成可以稳定分析和比较的材料。

## 4. 评估不能只看最后答案，因为 Agent 的质量存在于整条路径里

如果系统只是回答问题，看最终答案常常已经能说明很多事。  
但对 Agent 来说，最终答案只是质量的一部分。

你通常还需要评估：

- 任务是否完成
- 工具调用是否正确
- 中间步骤是否稳定
- 是否遵守权限和流程要求
- 用户对过程是否满意

因为一个系统完全可能“最后看起来做对了”，但过程里充满高风险操作和不必要的绕路。  
如果只看终点，不看路径，团队很容易把侥幸当成能力。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/96_observation_replay_evaluation_and_debugging_deep_revised__mermaid_04.png)

> 图96-4 Agent 评估为什么必须看整条路径  
> Agent 的质量不只在终点，还存在于行动过程、工具正确性、控制合规和用户是否放心这几层里。只看完成率，很容易把侥幸路径误判成稳定能力。

---

## 5. 人类评估在 Agent 时代不会消失，反而会更重要

有些指标当然可以自动计算，例如任务完成率、工具调用成功率、平均延迟和重试次数。  
但真正高价值的协作系统里，总有一些关键维度很难被简单自动化：

- 过程是否让人放心
- 交付物是否真的可用
- 接管是否顺滑
- 审批节点是否合理

这些维度并不只是情绪体验，它们直接影响系统能否进入真实组织流程。  
因此，人类评估并不会因为系统更智能而退场，反而会因为任务更复杂而更加重要。

---

## 6. 调试 Agent，比调试普通程序更像调试一个动态系统

调试普通程序，很多时候是在定位一段固定逻辑。  
调试 Agent，却常常是在定位一个动态系统里多种因素的耦合问题。

问题可能来自：

- 上下文装配
- 模型版本
- 工具协议
- 工作流配置
- 权限控制
- 评测标准

这意味着，你调的已经不只是代码，而是整套行为系统。  
正因如此，Agent 调试若没有统一 trace、版本比较和回放环境，复杂度会迅速高到团队无法承受。

这里还有一个非常现实的问题：不同团队常常记录了大量材料，却没有形成统一的观测语义。  
A 组记录的是“步骤”，B 组记录的是“工具调用”，C 组记录的是“模型轮次”，最后虽然看起来数据很多，但彼此根本接不起来。  
没有统一事件模型，回放就只是一堆碎片，评估也很难跨版本、跨产品、跨团队比较。

所以真正成熟的可观察性，不只是把东西记下来，而是先定义清楚“系统里什么算一个事件、什么算一次状态跃迁、什么算一次可比较的交付动作”。  
这层标准一旦缺失，团队越往后做，历史材料越多，定位问题反而越慢。

---

## 7. 评测体系真正的价值，在于让系统演化变得可比较

没有持续评测，团队每次更新之后都只能凭感觉判断“似乎变好了”。  
而感觉在复杂系统里通常不可靠。

一个成熟的评测体系，至少要让团队能够比较：

- 新版本是否比旧版本更稳定
- 哪一类任务改善了，哪一类退化了
- 控制更严格之后，成功率和成本如何变化
- 哪些失败模式仍在高频出现

只有当迭代结果可以被稳定比较，产品进化才不再依赖个人经验，而开始进入工程闭环。

而且，评测体系真正服务的不是“写报告”，而是“支持干预”。  
当一次更新导致表现退化时，团队真正要回答的是：

- 应该回滚模型，还是回滚工作流
- 是工具层出了问题，还是上下文层出了问题
- 是评测集过期了，还是线上分布已经换了

如果评测体系不能把这些干预决策支撑起来，它就仍然更像仪表盘，而不是工程抓手。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/96_observation_replay_evaluation_and_debugging_deep_revised__mermaid_05.png)

> 图96-5 评测体系把演化变成可比较过程  
> 这张图的重点，是把“感觉好像变强了”转成“哪些任务变好了、哪些退化了”。只有演化可比较，团队才真正拥有持续优化的工程抓手。

---

## 8. 工程现实：没有观测体系，团队越努力，系统越不透明

现实里，许多团队迟迟做不稳 Agent，并不是因为他们不够努力，而是因为他们看不清系统到底在发生什么。  
当没有统一 trace、没有回放、没有持续评测、没有版本对比时，每一次修复都像是在系统外侧做局部试探，希望结果刚好变好一点。

这种状态不会因为再加几个 prompt 技巧自然改善。  
它只能通过系统化的观察、评估和调试基础设施来扭转。

---

## 9. 容易误判的地方

### 误解一：有日志就等于有可观察性

日志只是材料，可观察性还要求路径理解、状态关联和问题定位能力。

### 误解二：评估只要看任务完成率

工具正确性、路径稳定性、权限合规和人类满意度同样关键。

### 误解三：调试 Agent 主要是在改 prompt

很多更核心的问题其实出在上下文、动作层、控制层和工作流设计上。

## 11. 本章小结

1. 可观察性是 Agent 工程的起点，而不是附属能力。  
2. Log、Trace、Checkpoint 和 Replay 共同构成了理解与调试系统的基础。  
3. 评估不能只看结果，还要看路径、控制和用户体验。  
4. 只有建立持续评测与回放能力，团队才可能把偶然成功变成稳定进步。  

核心判断：

> Agent 工程真正开始成熟的标志，不是它第一次跑通，而是你第一次能稳定看清它为什么这样跑、哪里出了问题、以及修完后到底有没有变好。
