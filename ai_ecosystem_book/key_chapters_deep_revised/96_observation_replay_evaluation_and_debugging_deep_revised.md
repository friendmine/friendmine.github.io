# 第96章 观察、回放、评估与调试

## 先说结论

Agent 系统要从偶然成功走向稳定能力，必须把行为过程记录成可理解、可回放、可评估、可比较、可调试的工程对象。

没有观察能力，Agent 只是一个会行动却难以解释的系统。
没有回放和评测，团队就无法判断一次修复到底是在解决问题，还是只是在制造新的偶然。

第95章讨论 Agent 失败学，重点是理解失败怎样沿着规划、上下文、工具和权限链条扩散。
本章讨论怎样把这些失败变成可诊断材料：看得见、放得回、比得出、调得动。

---

## 1. 可观察性不是运维附件，而是 Agent 工程的起点

普通软件已经需要日志、指标、追踪和报警。
Agent 系统对这些能力的依赖更强。

原因很直接：
Agent 的行为路径更长，状态变化更多，外部依赖更多，失败也更不稳定。

如果团队只能看见最终结果，就很难回答：

- 它为什么成功
- 它为什么失败
- 它在哪一步开始偏航
- 它看到了哪些上下文
- 它为什么选择这个工具
- 它是否理解了工具返回
- 它是否越过了权限边界
- 它产出的 artifact 是否真的可用

这意味着，可观察性不应该是上线之后再补的一层工具。
它应该从系统设计第一天就内建。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/96_observation_replay_evaluation_and_debugging_deep_revised__mermaid_01.png)

> 图96-1 可观测性与回放链路图
> 成熟 Agent 系统不只记录“发生过什么”，还要把事件、路径、状态、产物和判断重新接起来，用于回放、评估和调试。

没有这层基础设施，团队会不断陷入同一种状态：
系统偶尔跑通，大家很兴奋；
系统偶尔失败，大家只能猜。
这不是工程迭代，而是围绕黑箱做试探。

## 2. Log、Metric 和 Trace 解决的是不同问题

很多团队说“我们有日志”，就以为自己有了可观察性。
这远远不够。

Log、Metric 和 Trace 各自回答不同问题。

Log 记录离散事件。
它告诉你某一刻发生了什么：

- 调用了哪个工具
- 传入了什么参数
- 返回了什么状态码
- 哪个节点报错
- 哪条规则被触发

Metric 记录可聚合的数字。
它告诉你系统整体是否正在变坏：

- 成功率
- 失败率
- 延迟
- 重试次数
- 工具错误率
- 人工接管率
- 单任务成本

Trace 记录路径。
它告诉你任务怎样一步步走到当前状态：

- 任务经过哪些阶段
- 中间状态如何变化
- 系统何时改了计划
- 哪一步开始偏航
- 哪个工具结果影响了后续决策

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/96_observation_replay_evaluation_and_debugging_deep_revised__mermaid_02.png)

> 图96-2 Log 与 Trace 分别解决什么问题
> Log 记录局部事件，Metric 观察整体趋势，Trace 还原任务路径。Agent 调试需要三者结合，少任何一层都会看不完整。

只看 Log，容易只见树木不见森林。
只看 Metric，知道系统变差了，却不知道为什么。
只看 Trace，又可能缺少足够细节定位具体错误。

成熟可观察性不是“多记点东西”，而是建立一套能回答问题的观测结构。

## 3. 统一事件模型，是回放和跨版本比较的前提

很多系统记录了大量数据，却仍然调不动。
原因不是材料不够，而是材料没有统一语义。

A 组记录“步骤”，B 组记录“工具调用”，C 组记录“模型轮次”，D 组记录“用户操作”。
最后数据很多，却很难拼成一条可理解路径。

成熟 Agent 可观察性需要先定义事件模型。
至少要明确：

- 什么算一次任务
- 什么算一个阶段
- 什么算一次状态跃迁
- 什么算一次模型调用
- 什么算一次工具调用
- 什么算一次用户确认
- 什么算一次 artifact 变更
- 什么算一次判断或评分
- 什么算一次失败和恢复

没有统一事件模型，回放就会变成拼碎片。
评测也很难跨版本、跨团队、跨产品比较。

事件模型不是文档洁癖。
它决定系统能否把历史行为变成可分析资产。

---

## 4. 观测对象要同时覆盖 event、trace、state、artifact 和 judgment

Agent 工程里，最常见的观测错误是只记录动作，不记录状态和产物。

更完整的观测对象至少包括五类。

`event` 是离散事件。
例如一次工具调用、一次错误返回、一次权限判断、一次用户确认。
它回答“具体发生了什么”。

`trace` 是事件组成的路径。
它回答“任务怎样一步步走到这里”。

`state` 是系统在某个时刻相信的世界。
它包括当前目标、上下文、计划、变量、权限、工具结果和未完成事项。
它回答“系统当时以为自己在哪里”。

`artifact` 是实际生成或修改的对象。
例如文档、代码 diff、报告、工单状态、表格、配置。
它回答“系统留下了什么可继续工作的结果”。

`judgment` 是对过程或结果的评价。
例如人工判定、规则校验、模型打分、回归标签、质量 rubrics。
它回答“这条路径和这个结果到底好不好”。

很多系统调不动，是因为只记录了 event 和 trace，没有沉淀 state、artifact 和 judgment。
团队能看到系统“做过什么”，却看不清它“当时相信什么”“最后留下什么”“到底好不好”。

---

## 5. Checkpoint 和 Replay 把调试拉回工程

Agent 最让人头疼的一点，是很多错误并不稳定复现。

同样一个任务，换一次上下文、换一个模型版本、换一个工具状态、换一个外部系统返回，路径就可能变化。
如果没有 checkpoint 和 replay，团队面对问题时只能说：
再跑一次试试看。

Checkpoint 的意义，是保存关键时刻的状态。
它应该包括：

- 输入请求
- 当前上下文
- 模型版本
- 提示词版本
- 工具 schema
- 外部返回
- 中间状态
- artifact 快照
- 权限判断
- 随机性设置

Replay 的意义，是在接近原始条件的情况下重放路径。
它让团队能够比较：

- 原版本为什么失败
- 新版本是否修复
- 修复是否引入新问题
- 失败是否来自模型、工具、上下文还是控制层

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/96_observation_replay_evaluation_and_debugging_deep_revised__mermaid_03.png)

> 图96-3 Checkpoint 与 Replay 怎样把调试变成工程过程
> 没有 checkpoint 和 replay，团队只能一遍遍再跑一次看看。有了它们，失败才会从偶然事故变成可以稳定分析和比较的材料。

Replay 不一定要求完全复现所有外部世界。
但它至少要能保存足够多的关键条件，让团队定位主要差异。

没有这一步，所谓“修复”往往只是换了一次随机条件。

---

## 6. 评估不能只看终点，因为 Agent 的质量存在于路径里

如果系统只是回答问题，看最终答案常常已经能说明一大部分质量。
但 Agent 不一样。
Agent 的质量存在于整条路径里。

一个系统可能最后看起来完成了任务，但中间过程充满风险：

- 工具选错后靠重试碰巧成功
- 多次访问不必要数据
- 绕过用户本该确认的节点
- 生成了难以维护的 artifact
- 在错误假设上继续推进
- 用过高成本完成低价值任务

因此，Agent 评估至少要看：

- 任务是否完成
- 路径是否合理
- 工具调用是否正确
- 状态更新是否一致
- 权限和审批是否合规
- 中间 artifact 是否可用
- 人工接管是否顺滑
- 成本和时延是否可接受

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/96_observation_replay_evaluation_and_debugging_deep_revised__mermaid_04.png)

> 图96-4 Agent 评估为什么需要看整条路径
> Agent 的质量不只在终点，还存在于行动过程、工具正确性、状态一致性、权限控制和用户能否放心接管。

只看完成率，团队很容易把侥幸路径误判成稳定能力。
只看最终输出，团队也会忽略那些已经发生但尚未暴露的状态污染和责任风险。

---

## 7. 人类评估不会消失，它会变成关键质量层

有些指标可以自动计算。
例如：

- 任务完成率
- 工具调用成功率
- 平均延迟
- 重试次数
- 错误码分布
- 成本

但高价值协作系统里，总有一些关键维度很难完全自动化：

- 过程是否让人放心
- 交付物是否真的可用
- 接管是否顺滑
- 审批节点是否合理
- 决策解释是否足够
- 结果是否适合进入组织流程

这些不是软性体验，而是生产质量。

人类评估需要结构化。
否则评估会退回“我觉得不错”。

成熟做法通常包括：

- 明确 rubric
- 定义评分维度
- 维护正负样例
- 记录分歧样本
- 多评审交叉
- 把人工 judgment 写回评测集

人类评估不是自动评估的失败替代品。
它是开放任务和组织场景里必要的质量层。

---

## 8. 调试 Agent 不是只改 prompt，而是在调一个动态系统

很多团队把 Agent 调试理解成改 prompt。
Prompt 当然重要，但它只是其中一层。

真实问题可能来自：

- 任务定义
- 上下文装配
- 模型版本
- 提示词模板
- 工具 schema
- 工具返回格式
- 工作流编排
- 权限控制
- 记忆策略
- 评测标准
- 外部系统状态

所以调试 Agent 更像调试动态系统。
你调的不是一段固定逻辑，而是模型、上下文、工具、状态、流程和人类判断之间的耦合关系。

一个有效调试流程通常是：

1. 先定位失败发生在哪一层。
2. 再用 replay 固定问题条件。
3. 修改最小变量。
4. 跑回归样本。
5. 比较 trace、artifact 和 judgment。
6. 决定是继续推进、回滚，还是扩大评测集。

如果没有这套流程，团队会在 prompt、模型、工具和产品设计之间来回试，最后很难知道到底什么起了作用。

---

## 9. 评测体系的价值，是让系统演化变得可比较

没有持续评测，团队每次更新之后都只能凭感觉判断“似乎变好了”。
复杂系统里，感觉通常不可靠。

成熟评测体系至少要比较：

- 新版本是否比旧版本更稳定
- 哪类任务改善了
- 哪类任务退化了
- 成本是否变化
- 时延是否变化
- 人工接管是否变化
- 工具失败是否减少
- 旧失败模式是否复发

评测体系服务的不是写报告，而是支持干预。

当一次更新导致表现退化时，团队需要回答：

- 应该回滚模型，还是回滚工作流
- 是工具层出了问题，还是上下文层出了问题
- 是评测集过期，还是线上分布变了
- 是控制变严格导致完成率下降，还是模型能力退化

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/96_observation_replay_evaluation_and_debugging_deep_revised__mermaid_05.png)

> 图96-5 评测体系把演化变成可比较过程
> 评测体系的重点，是把“感觉好像变强了”转成“哪些任务变好了、哪些任务退化了、哪些失败仍然复发”。

只有当演化可比较，产品进步才不再依赖个人经验。
系统才开始进入工程闭环。

---

## 10. 回归样本库，是 Agent 系统的失败记忆

每一次严重失败，都应该变成系统资产。

成熟团队不会只修一个线上问题，然后把它忘掉。
它们会把失败沉淀进回归样本库。

回归样本至少要记录：

- 原始任务
- 失败 trace
- 关键上下文
- 工具返回
- artifact 快照
- 人工 judgment
- 失败分类
- 修复版本
- 后续验证结果

这样做的意义，是让系统不重复犯同样的错。

没有回归样本库，团队会陷入循环：
今天修了一个问题，下个月改模型、改工具、改流程以后，同一个失败又换一种形式回来。

Agent 系统越复杂，失败记忆越重要。
因为很多失败并不会在平均指标里明显体现，只会在特定组合条件下爆发。

---

## 11. 可观察性必须服务于第97章的平台化

本章讨论的观察、回放、评估和调试，不只是为了修 bug。
它还为第97章的长期协作平台做准备。

一个长期协作平台必须能回答：

- 谁发起了任务
- 系统看到了什么
- 做过哪些计划
- 调用了哪些工具
- 修改了哪些 artifact
- 谁批准了关键动作
- 哪一步失败或回滚
- 最终交付是否可用

如果没有这些记录，系统就不能承接长期责任。
它只能完成一次任务，不能成为团队可以信任的协作平台。

因此，可观察性不是底层运维问题。
它是平台化的责任基础。

当系统开始进入多人协作、项目记忆、artifact 生命周期和权限治理时，trace、replay、judgment 和 audit trail 会变成平台核心能力。

---

## 12. 工程现实：没有观测体系，团队越努力，系统越不透明

现实里，许多团队迟迟做不稳 Agent，并不是因为他们不够努力。
他们的问题是看不清系统到底在发生什么。

典型表现包括：

- 有日志，但没有任务路径
- 有 trace，但没有状态快照
- 有输出，但没有 artifact 版本
- 有人工反馈，但没有写回评测集
- 有线上事故，但不能 replay
- 有修复，但不能证明旧问题不复发
- 有指标，但无法定位到失败层

在这种条件下，每一次修复都像从系统外侧做试探。
团队不断改 prompt、换模型、调工具、加规则，却很难建立稳定进步。

Agent 工程成熟的起点，是把系统行为从黑箱里拿出来。
先看清，再回放，再评估，再调试，最后才谈稳定放量。

## 13. 容易误判的地方

### 误解一：有日志就等于有可观察性

日志只是材料。
可观察性还要求路径理解、状态关联、artifact 追踪、judgment 记录和问题定位能力。

### 误解二：评估只要看任务完成率

任务完成率重要，但路径质量、工具正确性、权限合规、人工接管和产物质量同样关键。

### 误解三：调试 Agent 主要是在改 prompt

Prompt 只是一个变量。
很多核心问题出在上下文、工具、状态、权限、工作流和评测标准上。

### 误解四：Replay 必须完全复现外部世界才有价值

不必。
Replay 的价值在于固定关键条件、定位主要差异、支持跨版本比较。

### 误解五：观测体系是上线后再补的运维能力

不是。
Agent 的行为链太长，观测体系必须从设计阶段内建，否则后续调试和评测都会失真。

## 14. 本章小结

1. 可观察性是 Agent 工程的起点，不是上线后的运维附件。
2. Log、Metric、Trace、Checkpoint 和 Replay 共同构成理解与调试 Agent 的基础。
3. Agent 评估不能只看终点，还要看路径、状态、工具、权限、artifact 和人工 judgment。
4. 回归样本库是系统失败记忆，决定旧问题会不会反复回来。
5. 可观察性会自然支撑第97章的平台化，因为长期协作必须能解释责任链、状态变化和交付过程。

核心判断：

> Agent 工程开始成熟的标志，不是它跑通过一次，而是团队能稳定看清它为什么这样跑、哪里出了问题、修完之后到底有没有变好。
