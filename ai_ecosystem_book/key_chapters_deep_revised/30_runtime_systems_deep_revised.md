# 第30章 运行时系统

上一章讲驱动与固件：它们把具体设备接入操作系统。设备可用之后，程序仍然要面对一个更动态的问题：对象怎样创建和释放，任务怎样排队和唤醒，异步操作怎样推进，设备怎样被提交工作，错误怎样被捕获和恢复。

运行时系统可以理解为程序开始执行之后的现场组织层。编译器更偏运行前的安排，操作系统更偏整机资源秩序，运行时则站在程序执行现场，持续管理内存、任务、异步、资源、错误和设备调用。

## 1. 程序开始运行后，问题继续展开

一段程序被编译出来，并不代表执行问题已经结束。真实程序不是静止的指令文件，而是在时间中不断展开的过程。

运行过程中，内存会被申请、释放、复用和争抢；任务会创建、阻塞、唤醒和取消；I/O 会等待；设备会忙闲变化；异常会沿调用链传播；多个请求会同时进入系统。编译器可以提前做许多规划，但它无法预知每一次输入、每一次排队、每一次资源波动。

运行时系统的作用，就是在程序实际执行时持续组织这些动态过程。它让高层程序不必把每个对象生命周期、每次异步等待、每个设备队列细节都写在业务逻辑里。

---

## 2. 运行时是一层执行环境

运行时有很多形态。Java 虚拟机、.NET CLR、Python 解释器、JavaScript 引擎、WebAssembly runtime、C/C++ runtime library、CUDA runtime、深度学习框架 runtime，都可以放在这条线里理解。

它们形态不尽相同，但都在承担一类工作：把程序从“可执行形式”带入“正在执行的现场”，并在现场管理对象、栈、堆、线程、任务、异常、资源和与系统的交互。

有些运行时很厚，像 JVM，会负责字节码解释或 JIT、垃圾回收、类加载、线程协调和安全边界。有些运行时较薄，像 C runtime，提供启动入口、标准库支持、内存分配、异常或线程相关基础。AI runtime 则会负责图执行、kernel 调度、设备内存和异步队列。

所以 runtime 不只是单一技术名词，而是一类执行现场组织机制。

## 3. 内存管理是运行时的核心现场

很多运行时首先要解决内存问题：对象放在哪里，什么时候创建，什么时候释放，能否复用，谁拥有它，谁还在引用它。

在 C/C++ 这类体系里，开发者通常承担更多显式管理责任，malloc/free、new/delete、RAII、智能指针和内存池都是围绕生命周期展开的机制。在 Java、Go、JavaScript 等体系里，垃圾回收器会自动识别不再可达的对象并回收内存，但它也会引入停顿、吞吐和内存占用之间的取舍。

AI runtime 的内存问题更具体。大张量、显存、pinned memory、中间激活、KV cache、临时缓冲区和通信 buffer 都需要被规划。频繁分配释放会带来开销和碎片，过度保留又会抬高峰值占用。

因此，运行时常常通过内存池、对象池、缓冲区复用和生命周期分析，把动态内存压力控制在可预测范围内。

---

## 4. 任务、线程、协程和事件循环组织执行流

程序不只需要内存，还需要组织执行流。传统线程直接对应操作系统调度的执行单元，适合并行处理，但线程过多会带来切换成本、锁竞争和调度复杂度。

协程、future、promise、async/await、事件循环，则提供了更轻量的并发组织方式。它们让程序可以在等待 I/O、网络或设备结果时让出执行权，而不是让线程空等。

运行时在这里承担的是调度和协调工作：哪些任务可以继续，哪些任务在等待，错误怎样传播，取消怎样处理，资源在异步边界上怎样保持一致。

这解释了为什么同样的业务逻辑，在不同 runtime 上会有不同表现。一个事件循环被阻塞，整个服务可能延迟飙升；一个线程池配置不当，吞吐和尾延迟都会受影响；一个异步任务没有正确取消，资源泄漏可能长期累积。

---

## 5. 运行时负责把异常和资源边界收住

真实系统会失败。文件打不开，网络超时，设备 reset，内存不足，任务取消，用户断开连接，模型输出不合格式。运行时需要让这些失败沿着可控路径传播，而不是把局部错误扩大成整机混乱。

异常处理、错误码、panic/recover、defer/finally、上下文取消、超时控制、资源清理，都是运行时或语言运行机制的一部分。它们的共同目标，是让程序在异常发生时仍能释放资源、保留诊断信息、维持边界。

AI 服务里，这一点尤其重要。一次推理请求可能占用显存 buffer、网络连接、线程池任务、日志上下文和缓存项。如果请求超时或失败，运行时要确保这些资源能被清理，否则系统会出现隐性泄漏，进而表现为吞吐下降或节点不稳定。

---

## 6. AI runtime 组织模型执行现场

AI runtime 面对的不只是普通函数调用，而是一组张量算子、设备队列、显存布局、异步 kernel、通信操作和 batch 策略。

一个模型从框架层进入执行，runtime 可能要做图执行、算子调度、kernel launch、stream 管理、事件同步、显存分配、tensor 生命周期管理、跨设备拷贝和错误恢复。编译器可以提前优化计算图，但运行时要在真实输入、真实 batch、真实设备状态下把这些计划执行出来。

例如，runtime 可能把多个请求合成 batch，提高设备利用率；也可能为了降低首 token 延迟，限制 batch 等待时间。它可能复用 KV cache 和中间 buffer，也可能在显存压力下触发淘汰或降级。它可能把计算和数据搬运重叠起来，也可能因为同步点设计不好而让设备空等。

这就是 AI runtime 的核心位置：它直接影响吞吐、延迟、显存占用、尾部抖动和设备利用率。

---

## 7. 编译器、驱动和 runtime 是一条执行链

编译器、驱动和 runtime 经常被分开讨论，但在真实系统里它们是一条链。

编译器负责把高层程序或模型图变成更适合执行的形式。驱动负责把具体设备接入操作系统，并提供提交命令、管理内存、处理中断和恢复错误的通道。runtime 则负责在执行过程中决定什么时候提交任务、怎样安排队列、怎样复用内存、怎样处理同步和异常。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/30_runtime_systems_deep_revised__mermaid_02.png)

> 图30-2 编译器与 runtime 的分工
> 编译器偏运行前规划，runtime 偏执行现场调度，驱动则把现场调度落到具体设备能力上。

如果这条链其中一段不成熟，系统就会失真。编译器生成了好计划，但 runtime 调度不好，设备仍可能吃不饱；runtime 想做异步推进，但驱动同步语义不清，错误会难以定位；驱动可以高效提交任务，但上层没有合理 batch 和内存规划，峰值能力也难以兑现。

---

## 8. 性能提升常常发生在运行时层

用户看到一次版本升级后性能提升，往往以为是模型、算法或芯片变了。实际工程中，很多收益来自 runtime。

内存池减少频繁分配释放，异步队列减少空转等待，batch 调度提高设备饱和度，缓存策略降低重复计算，预取和流水线让 I/O 与计算重叠，错误恢复减少整机重启。这些改动不像新模型那样显眼，却会显著影响生产环境的稳定性和成本。

运行时优化的特点，是它经常不改变用户可见功能，却改变系统兑现功能的方式。它让同一份模型、同一块硬件、同一套接口，在真实负载下表现得更稳、更快或更省。

因此，runtime 成熟度常常决定系统能否从 demo 进入生产。

---

## 9. 从运行时走向数据结构与算法工程

运行时负责组织执行现场，但它组织现场时依赖大量具体结构和策略：队列怎样设计，缓存怎样淘汰，内存池怎样分桶，任务怎样排序，图怎样遍历，请求怎样路由，状态怎样索引。

这些问题开始进入下一章的数据结构与算法工程。

也就是说，runtime 让程序在复杂现实中持续运行；数据结构与算法工程则进一步回答：系统内部的信息、状态、任务和路径，怎样组织得足够高效、稳定、可维护。

---

## 10. 几个需要建立的判断

第一，程序能启动不等于运行时成熟。能跑通 demo，只说明最小路径存在；能处理真实流量、异常、资源波动和长期运行，才说明 runtime 站得住。

第二，运行时不是语言小细节。它影响内存、并发、异常、调度、I/O、设备提交和资源效率。

第三，AI 性能不是只由模型和芯片决定。runtime 的 batch、队列、内存、cache 和异步策略，会直接改变吞吐、延迟和成本。

第四，运行时的价值在于吸收动态变化。现实系统里输入会变、资源会变、设备会忙、错误会来，runtime 要把这些变化组织进可控执行过程。

---

## 本章收束

1. 运行时系统位于编译结果、操作系统、驱动和真实执行现场之间。
2. 它负责组织内存、任务、异步、资源生命周期、异常和设备调用。
3. 不同语言和平台的 runtime 形态不同，但都在管理程序运行后的动态过程。
4. 内存池、GC、线程、协程、事件循环和错误处理，都是运行时组织能力的体现。
5. AI runtime 直接影响模型执行的吞吐、延迟、显存占用、同步和设备利用率。
6. 编译器、驱动和 runtime 是连续执行链，任何一段薄弱都会影响平台体验。
7. 很多生产性能提升发生在运行时层，而不是模型或芯片本身。

运行时系统把“程序能够启动”推进成“程序能够在真实负载、真实资源和真实错误中持续可控地执行”。
