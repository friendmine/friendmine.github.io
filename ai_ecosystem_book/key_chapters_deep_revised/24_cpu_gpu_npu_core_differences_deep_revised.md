# 第24章 CPU、GPU、NPU 的关键差别

上一章把 ISA 和微架构分开：软件看到的是接口，硬件内部做的是实现。到了这一章，要把这个视角放到三类常见处理器上：CPU、GPU、NPU 为什么会长成不同样子。

它们的差别不能简单排成快慢顺序，而是面向不同工作负载形成的资源组织差别。CPU 偏向复杂控制和低时延响应，GPU 偏向大规模数据并行吞吐，NPU 偏向围绕神经网络模式做更专门的能效优化。

## 1. 关键问题不是谁更快

公众讨论 CPU、GPU、NPU 时，常见说法是：CPU 通用，GPU 更快，NPU 更适合 AI。这些说法有直觉价值，但很容易把系统问题说成广告口号。

更准确的问题应该是：一类硬件把硅面积、功耗、存储带宽、控制逻辑和软件复杂度，优先分配给了什么。

同一笔芯片资源，可以用来做更强的分支预测、更大的 cache、更深的乱序执行窗口，也可以用来做更多并行计算单元、更宽的显存通路，或者做更贴近神经网络数据流的矩阵阵列。选择不同，适合的任务就不同。

## 2. CPU 擅长复杂控制和低时延

CPU 面向的是通用计算。操作系统调度、浏览器、数据库、业务服务、编译器、脚本语言运行时，都包含大量分支、指针跳转、系统调用、短任务和不可预测的内存访问。

为了应对这些不规则任务，现代 CPU 把大量资源用于控制能力和低时延响应：复杂分支预测、乱序执行、较强的单线程性能、多级 cache、虚拟内存支持、异常和中断处理、系统级特权机制。

这意味着 CPU 的优势不在于把同一个简单操作复制成海量并行，而在于面对变化多、依赖强、延迟敏感的任务时，仍然能保持灵活执行。它像系统里的调度者和通用工人，未必在每一种数值任务上单位能效最高，但能接住复杂而多变的工作。

---

## 3. GPU 擅长吞吐和数据并行

GPU 的历史起点是图形渲染。图形任务有一个鲜明特征：大量像素、顶点、纹理和矩阵运算可以并行处理，控制逻辑相对规整，重复计算很多。

这种工作负载让 GPU 形成了不同的资源组织方式。它把更多面积和功耗投入到大量并行计算单元和高带宽存储系统上，用许多线程组同时推进任务，通过并发度隐藏单次访存延迟。

现代 GPU 不只是图形芯片，也成为 AI 训练和推理的重要平台，原因就在于矩阵乘、卷积、attention 等任务同样具有大规模数据并行结构。只要数据能被切成足够多块，并且通信与访存能跟上，GPU 就能把吞吐优势发挥出来。

但 GPU 不是更大的 CPU。分支严重发散、批量太小、数据访问不连续、任务粒度太碎，都会降低它的效率。

---

## 4. NPU 更强调专用数据流和能效

NPU 通常指围绕神经网络负载设计的专用或半专用加速器。它不追求覆盖尽可能广的通用任务，而是抓住 AI 计算里反复出现的模式：矩阵乘加、卷积、低精度数值、张量搬运、激活函数和固定结构的数据流。

因此，NPU 往往会使用 MAC 阵列、systolic array、片上 SRAM、低精度计算单元、专门的数据复用路径和算子调度机制。目标是在特定模型和特定精度下，以更少能耗完成更多有效计算。

代价也很清楚：适配范围更窄，对编译器、算子库、模型格式和运行时要求更高。模型结构一旦偏离硬件擅长的模式，或者算子支持不完整，理论能效就未必能转化成实际性能。

## 5. 三者的核心取舍是通用性、吞吐和能效

CPU、GPU、NPU 可以放在同一条取舍轴上理解，但这条轴不是优劣排序。

CPU 把大量资源用于通用控制、兼容性和低时延，适合复杂分支、系统调度和不规则任务。GPU 把更多资源用于大规模并行吞吐，适合规则、可批量化、数据量大的数值计算。NPU 把更多资源用于特定 AI 模式下的数据流和能效，适合结构稳定、算子匹配、精度明确的神经网络负载。

这三者不是互相淘汰的关系，而是在不同约束下做了不同选择。通用性越强，单位任务的专用效率可能越难做高；专用性越强，对工作负载和软件栈的要求也越苛刻。

---

## 6. 内存和数据路径常常决定实际表现

很多硬件比较只看峰值算力，却忽略了数据从哪里来、往哪里去。对 CPU、GPU、NPU 来说，内存层级和数据路径经常比宣传中的峰值 TOPS 或 FLOPS 更能解释真实表现。

CPU 依赖多级 cache 降低不规则访问的平均代价。GPU 依赖高带宽显存、共享内存、寄存器文件和访存合并，让大量线程持续获得数据。NPU 则常常依赖片上 SRAM 和数据复用，把权重、激活和中间结果尽量留在离计算阵列更近的位置。

如果数据搬运成为瓶颈，计算单元就会等待。一个 NPU 的峰值算力很高，但如果算子切分、内存布局或数据复用做不好，阵列可能吃不饱；一个 GPU 很强，但如果 batch 太小或访存模式零散，也可能表现平平；一个 CPU 峰值吞吐不高，却可能在短小、不规则、延迟敏感任务上更合适。

---

## 7. 软件栈决定硬件能力能否兑现

硬件只提供可能性，软件栈决定这些可能性能否被稳定调用。

CPU 有成熟的编译器、操作系统、调试工具和通用库，因此开发门槛较低。GPU 依赖 CUDA、ROCm、图形 API、计算库、编译器和 runtime，让开发者把任务组织成适合并行执行的 kernel。NPU 通常更依赖模型编译器、算子库、图优化、量化工具和设备 runtime。

这也是为什么同一颗芯片在不同软件环境下表现可能差异很大。算子没有适配、图编译效果不好、内存规划不合理、fallback 到 CPU、数据格式频繁转换，都会让硬件优势被软件路径抵消。

评价 CPU、GPU、NPU，不能只看芯片参数，还要看编译器、库、框架、驱动、运行时和开发工具是否成熟。

---

## 8. AI 系统通常需要异构协作

AI 应用不是只有一段矩阵乘。一个真实系统里会同时出现数据预处理、请求调度、检索、业务逻辑、张量计算、KV cache 管理、后处理、网络通信和存储访问。

这些任务不适合全部交给同一种硬件。CPU 适合控制、调度和系统服务；GPU 适合大批量张量计算；NPU 适合在匹配的模型和部署场景下提供高能效推理。存储、网络和互连还要负责把数据送到合适位置。

> 图24-2 异构协作为什么会成为常态
> AI 系统里同时存在控制逻辑、并行数值计算和专用执行路径，因此高效架构通常是让不同硬件在各自合适的位置协作。

异构协作的难点不只是“把几种芯片放在一起”，还包括任务切分、数据格式、内存共享、调度策略、驱动稳定性和性能分析。协作做不好，数据在设备之间来回搬运，收益可能很快被通信和转换成本吃掉。

---

## 9. “快很多倍”需要放回上下文

宣传里的性能比较如果不说明条件，参考价值很有限。至少要问清楚：任务类型是什么，模型结构是什么，输入大小和 batch 多大，使用什么精度，是否量化，软件栈是否成熟，数据是否已经在设备本地，比较的是吞吐、延迟还是能效。

CPU、GPU、NPU 的差异，很多时候不是“先进”和“落后”的差异，而是当前任务是否匹配当前硬件组织方式。

控制逻辑重、批量小、延迟敏感、不规则访问多的任务，CPU 可能更合适。大规模矩阵和张量计算、batch 足够、访存连续时，GPU 更容易发挥吞吐。固定模型、固定精度、算子匹配、功耗敏感的端侧推理场景，NPU 可能更划算。

离开上下文谈快慢，很容易把工程问题变成伪结论。

---

## 10. 从三类处理器走向系统分层

这一章把 CPU、GPU、NPU 放在一起比较，目的不是给出简单排名，而是让读者看到：硬件能力总是在工作负载、微架构、内存路径、软件栈和系统目标之间形成。

下一章会把视角再放大，进入计算机系统分层图。因为无论 CPU、GPU 还是 NPU，都不是孤立发挥作用。它们要被驱动调用，被操作系统管理，被编译器和 runtime 使用，被框架封装，最后才以应用功能呈现给用户。

只有把硬件差异放回系统分层中，才能理解一项能力为什么能被上层稳定使用，也能理解性能问题为什么经常出现在层与层之间。

---

## 本章收束

1. CPU、GPU、NPU 的差别不能按快慢简单排序，它们的资源组织方式不同。
2. CPU 擅长复杂控制、低时延响应、系统调度和不规则任务。
3. GPU 擅长大规模数据并行和吞吐型数值计算，但依赖足够并行度和合适访存。
4. NPU 更强调神经网络模式下的数据流、低精度计算和能效，但依赖算子、编译器和模型匹配。
5. 内存层级、数据路径和软件栈，常常决定峰值能力能否兑现。
6. 真实 AI 系统通常需要 CPU、GPU、NPU、存储和互连协作。
7. 任何“快很多倍”的说法，都需要回到任务、精度、batch、软件栈、吞吐、延迟和能效上下文。

CPU、GPU、NPU 不是三档速度，而是三种围绕不同工作负载组织计算、数据和控制的方式。
