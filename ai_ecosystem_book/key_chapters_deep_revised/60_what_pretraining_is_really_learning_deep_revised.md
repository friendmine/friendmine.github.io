# 第60章 预训练到底在学什么

## 先说结论

预训练不是把模型教成一个会聊天的助手。
它先把模型放进海量序列里，让它反复学习：在当前上下文之后，下一个 token 应该是什么。

这个目标看起来很窄，实际压力很大。
为了持续降低预测误差，模型需要从文本、代码、表格、说明文档、网页、论文和对话记录中抽取多层规律：词与词的搭配、句法组织、语义关系、任务格式、代码结构、事实共现、推理样式和人类表达世界的方式。

所以，预训练学到的不是一个可以直接交付给用户的“可靠人格”，而是一套可复用的内部表示。
这套表示可以支撑翻译、总结、问答、代码补全和多步组织，但它仍然带着数据偏差、事实滞后、幻觉风险和目标错位。
要把它变成对话模型，还需要下一章要讲的指令微调、偏好优化和系统层约束。

## 1. “只是预测下一个 token”这个说法太容易低估问题

大语言模型的预训练目标通常被概括为 next token prediction。
给定前面的上下文，模型预测接下来更可能出现的 token。

这个描述没有错，但它容易让人误以为模型只是在做浅层接龙。
问题在于，短句里的接龙和长文档、代码库、数学推导、法律条款、对话场景里的预测不是同一类难度。
要在这些场景中持续猜得更准，模型不能只依赖附近几个词。

例如，看见“如果 API 返回 401，客户端应该……”，下一个片段取决于 HTTP 语义、身份认证流程、错误处理习惯、文档写作格式，甚至取决于上下文中这个系统采用的是 token、cookie 还是签名请求。
表面上仍然是预测 token，内部却需要调动一组结构性判断。

这就是预训练有力量的地方：目标函数简单，但它把大量复杂结构压到同一个优化问题里。

---

## 2. 模型为了降低损失，会学习多层次的可预测结构

如果模型只记住局部搭配，它能处理一部分常见短语，却很难处理长距离依赖和复杂任务。
训练数据一旦足够大、上下文足够长，降低损失就会迫使模型吸收更丰富的规律。

这些规律大致可以分成几个层次：

- token 与词片段的组合习惯
- 语法、句法和标点结构
- 实体、属性、关系和语义角色
- 段落、列表、标题、代码块等文档组织方式
- 问答、翻译、总结、解释、规划等任务模板
- 人类在语言中反复表达的事实共现和经验模式

这些层次不是彼此隔离的。
一句技术说明里的下一个 token，可能同时受到函数签名、变量命名、错误信息、上下文约束和读者预期影响。
预训练就是在这种复合压力下，让模型把不同层次的可预测结构压进参数和中间表示。

## 3. 语言不是世界本身，但它携带了世界的投影

预训练模型没有直接生活在物理世界里。
它接触到的是数据：文本、代码、图像说明、网页片段、论坛讨论、书籍、论文和各种人工记录。

这意味着它学到的“世界知识”有一个中间层。
模型不是直接观察杯子会掉落、数据库会超时、公司会重组、政策会变化；它是在学习人类如何用语言记录这些现象。
语言不是世界本身，却是人类经验、制度、知识和行为模式的高密度投影。

因此，模型可以表现出常识、事实记忆、代码经验和问题求解样式。
但这些能力的来源仍然是数据分布中的表达痕迹，而不是经过验证的现场感知。
当数据过时、语料有偏、上下文不完整，模型就可能把流畅表达误当作可信答案。

这一点对出版级论述很关键：预训练可以吸收世界的语言投影，却不能因此被描述成“拥有世界本身”。

## 4. 参数不是资料库，而是被训练出的压缩结构

很多人会把大模型想象成一个巨大的资料库。
这个比喻只能解释一小部分现象。
模型确实会记住一些训练样本、固定表达和高频事实，但更有价值的部分，是它把重复出现的结构压缩成可以迁移的表示。

如果模型只会背诵，它面对稍微改写的问题、陌生的代码上下文、不同格式的任务说明时就会迅速失效。
而强预训练模型之所以能在新上下文里工作，是因为它学到的不是单条答案，而是大量可组合的关系：

- 什么样的句子结构通常表达因果
- 什么样的代码块通常承担初始化、校验或异常处理
- 什么样的提问格式通常期待定义、比较、步骤或判断
- 什么样的上下文暗示后续需要补例子、给约束或收束结论

参数承载的就是这些被压缩后的规律。
激活状态则在具体上下文中临时组织这些规律，让模型为当前输入生成下一步预测。

## 5. 表示空间让“词”变成上下文里的角色

预训练学到的并不只是词表中的静态含义。
在 Transformer 中，同一个 token 会因为上下文不同而获得不同表示。

“bank”在金融文本里接近账户、贷款、利率；在河流描述里接近岸边、泥土、水位。
“模型”在机器学习章节里指参数化系统，在产品文档里可能指型号，在组织管理语境里又可能指运营范式。

这种上下文化表示很要紧。
模型不是先查一个固定词典，再机械拼接答案；它是在每一层里根据上下文重新组织 token 之间的关系。
第59章讲过的注意力、MLP、残差连接和归一化机制，正是在这个过程中协同工作：注意力负责在上下文中选择相关信息，MLP负责对局部表示做非线性变换，残差和归一化让多层结构可以稳定叠加。

所以，预训练学到的是一种动态表示能力。
它让模型可以在不同上下文里重新解释同一个词、同一个变量、同一个任务指令。

## 6. 许多任务可以被改写成条件延续

预训练目标之所以能迁移到很多任务，一个原因是：大量任务都可以放进“给定条件之后继续生成”的框架。

问答是给定问题后生成回答。
翻译是给定源语言后生成目标语言。
总结是给定原文后生成压缩表达。
代码补全是给定上下文后生成下一段程序。
解释、改写、分类、抽取信息，也都可以通过合适的提示格式转化为条件生成。

这并不是说所有任务都被等价地解决了。
它只是说明，序列建模给了模型一个统一入口：不同任务可以共享同一套表示、同一套上下文建模能力、同一套生成机制。

于是，预训练模型会表现出一种很有用的特征：只要提示把任务边界、输入材料和期望输出形式表达清楚，模型就能把已经学到的结构调动起来。
这也是 prompt、few-shot 示例和上下文学习能够发挥作用的基础。

## 7. 规模会放大能力，但不会取消边界

随着数据量、参数量、上下文长度和训练算力提升，模型通常会表现出更好的语言建模能力。
这种提升会带来一系列可观察变化：长依赖处理更稳，格式遵循更清楚，代码和数学样式更像样，多步组织也更连贯。

但规模不是魔法。
它放大的是训练目标和数据分布中可以被学习的结构，也会放大其中的噪声、偏差和错误关联。
如果语料里大量存在过时事实、片面叙述、错误代码、低质量问答或带偏见的表达，模型也会把其中一部分吸收进去。

因此，谈预训练能力时，需要把三个问题分开：

- 模型是否在训练中形成了某种能力迹象
- 这种能力在评测集上是否稳定可复现
- 这种能力在真实用户场景里是否可交付、可约束、可追责

很多误判来自把第一层当成第三层。
基础模型可能已经具备不少可调用结构，但它距离工程产品仍有一段路。

## 8. 预训练的边界：它学会了续写，不等于学会了负责

预训练模型的输出目标，是让后续 token 在训练分布下更合理。
这个目标和人类对助手的期待并不重合。

用户期待的是：听懂意图，遵守指令，承认不确定，回避不该做的请求，区分事实和推测，必要时提问澄清，并在约束内完成任务。
这些行为并不会从 next token prediction 中自动稳定出现。

预训练的典型边界包括：

- 事实可能过时，也可能来自错误或低质量语料
- 模型会倾向于生成看起来连贯的内容，即使证据不足
- 它学到的是分布中的表达模式，不是现实中的验证流程
- 它可以模仿指令格式，但未必稳定服从用户意图
- 它可能把相似场景混在一起，给出过度自信的答案

这就是为什么“会说”不等于“可信”，“会补全”不等于“会负责”，“像专家口吻”不等于“专家判断”。
预训练提供能力底座，但可靠行为需要后续训练和系统设计。

## 9. 基础模型和对话模型之间，还隔着行为塑形

经过预训练的模型通常可以称为基础模型或基座模型。
它具备广泛的语言与任务结构，但它的默认行为仍然更接近“根据上下文继续生成”。

一个好用的聊天助手需要在此基础上继续被塑形。
常见路径包括指令微调、人工偏好数据、强化学习或偏好优化、安全策略、工具接口、检索增强和系统提示。
这些环节并不是给模型额外贴一层包装，而是在改变它面对用户请求时的行为分布。

例如，基础模型可能会顺着一个错误前提继续写下去；对话模型则应该指出前提有问题。
基础模型可能会模仿危险指令的文本格式；对齐后的模型则应该拒绝或转向安全替代方案。
基础模型可能会生成百科式段落；产品化助手则需要按照用户任务组织答案。

理解这条分界线，才能避免两个相反误区：
一方面，不能低估预训练，因为后续能力建立在它压缩出的结构上；
另一方面，也不能把预训练模型直接当成可靠助手，因为可靠性来自基座能力、行为训练和系统工程的组合。

## 10. 本章小结

1. 预训练的表面目标是 next token prediction，但这个目标会迫使模型学习多层次结构。
2. 模型学到的世界知识来自语言和数据记录中的投影，不等于经过验证的现实本身。
3. 参数不是简单资料库，而是大量可重复规律被压缩后的能力载体。
4. 上下文化表示让同一个 token 能在不同语境中承担不同角色。
5. 许多任务可以被改写成条件延续，因此预训练能力能够迁移到问答、翻译、总结、代码等场景。
6. 规模能放大可学习结构，也会继承数据和目标带来的边界。
7. 基础模型要成为对话模型，还需要指令微调、偏好优化和系统层约束。

核心判断：

> 预训练学到的不是一个现成助手，而是一套从海量序列中压缩出来的可复用表示；它解释了模型能力的来源，也划出了可靠应用还需要继续建设的边界。
