# 第92章 不同场景下的 Harness 形态

## 先说结论

Harness 没有统一外观，因为它不是一种固定界面，而是围绕任务对象、动作空间、风险结构、状态管理和交付物不断变形的组织结构。

场景变了，系统要承接的对象、权限、反馈速度和失败代价都会变。
因此，不存在一个能直接套进所有行业的通用 Harness 骨架。

![插图 第92章 四种 Harness 工作台]({{ site.baseurl }}/ai_ecosystem_book/images/chapter_092_harness_forms_across_scenarios.png)

> 插图 第92章 四种 Harness 工作台

第91章区分了 Agent 和 Harness：Agent 更像执行智能，Harness 更像组织智能。
本章继续回答另一个问题：既然 Harness 是组织结构，为什么不同场景下它会长成完全不同的样子。

---

## 1. 只从界面看 Harness，会立刻陷入表面化

很多人谈 Harness，第一反应是界面：
聊天框、侧边栏、工作区、控制台、插件面板。

这些当然重要。
但它们只是表层结果。

真正决定 Harness 形态的，是更深的五个问题：

- 当前任务对象是什么
- 系统可以采取哪些动作
- 状态如何被保存和更新
- 风险需要怎样被控制
- 最终交付物是什么

同样都是 AI 产品，通用聊天助手、编程助手、研究工作台、企业流程系统、机器人控制台，看起来会非常不同。
原因不是设计风格不同，而是它们要承接的任务本来就不同。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/92_harness_forms_across_scenarios_deep_revised__mermaid_01.png)

> 图92-1 不同场景下的 Harness 形态对比图
> 表面上它们都像带 AI 的工作系统，但决定形态的是任务对象、动作空间、风险结构、状态管理和交付物。场景一变，系统骨架就会跟着重写。

很多团队的问题不是没有做 Harness，而是下意识沿用了上一代产品模板。
该做成工作台的地方做成了聊天框；
该做成控制台的地方做成了轻量面板；
该有审批和回放的地方，只写成“让模型谨慎一点”。

模板一旦借错，后面所有局部优化都会变成在错误骨架上缝补。

---

## 2. 通用聊天 Harness 是轻协作结构，不是所有任务的默认终点

通用聊天场景里，系统围绕会话组织。
用户输入问题，模型输出回答，历史上下文以消息流串联。

这类 Harness 的重点通常是：

- 快速提问
- 会话连续性
- 轻量记忆
- 轻量工具接入
- 文本输出

它的优点是轻、快、适应面广。
它很适合验证用户是否愿意把问题交给 AI，也适合承接短回路协作。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/92_harness_forms_across_scenarios_deep_revised__mermaid_02.png)

> 图92-2 通用聊天 Harness 的基本组织方式
> 通用聊天 Harness 的优势在于轻和快，但它会把任务状态压缩进消息流。只要任务变长、变重、需要显式工作区，这种结构就会开始吃力。

它的限制也很清楚。
消息流不是承载复杂任务状态的最佳结构。

当任务开始依赖：

- 文件
- 多步骤计划
- 审批
- 长期项目状态
- 多人协作
- artifact 交付

聊天框就会开始吃力。

聊天式 Harness 适合作为入口或起步形态，但不应该被误认为所有 AI 产品的终点形态。

---

## 3. 编程 Harness 更像开发工作台，而不是聊天壳

编程场景会很快暴露聊天范式的局限。

开发者处理的不是一段回答，而是：

- 文件
- 目录
- diff
- 终端
- 测试结果
- 依赖环境
- 版本控制
- 回滚路径

因此，编程 Harness 的核心不是“说得多好”，而是能否尊重软件工程的工作对象。

它至少要支持：

- 正确读取项目结构
- 理解文件之间的关系
- 生成可审阅 diff
- 运行命令和测试
- 解释失败日志
- 保留变更记录
- 支持回滚
- 让开发者接管

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/92_harness_forms_across_scenarios_deep_revised__mermaid_03.png)

> 图92-3 编程 Harness 的核心工作对象
> 编程场景不能停在聊天壳，因为它处理的是文件、命令、diff、测试和回滚。这些对象进入中心后，Harness 的本体自然变成开发工作台。

从采用角度看，编程 Harness 的成败不取决于模型说得像不像高手。
关键是改完之后，开发者敢不敢接进现有工程链。

如果系统不能稳定处理 diff、测试、版本控制和回滚，它再会解释代码，也很难成为真正可用的开发工作台。

---

## 4. 办公和企业流程 Harness 的重心，是协同、权限和责任

办公和企业流程场景面对的对象通常是：

- 文档
- 表格
- 邮件
- 工单
- 审批链
- 角色权限
- 跨部门协作
- 业务系统状态

这类 Harness 的关键会明显转向：

- 谁能看到什么
- 谁能修改什么
- 哪个节点需要谁批准
- 结果如何进入现有流程
- 每次变更如何留痕
- 责任如何回溯

在这里，单纯让模型更会表达帮助有限。
难点是让系统嵌进组织已有的责任结构。

企业场景里，价值更高的往往不是“聪明回答”，而是“责任对齐”。
系统能不能穿过部门边界、角色边界、审批边界和审计边界，决定了它是一个演示工具，还是一个能被组织接纳的工作系统。

这类 Harness 一旦设计失配，组织的第一反应通常不是“效果差一点”，而是“先别让它碰真实流程”。

---

## 5. 研究 Harness 更像知识工作台

研究场景既不是纯聊天，也不是标准流程。
它需要在开放探索和严谨记录之间保持平衡。

研究型 Harness 往往更重视：

- 资料检索
- 引用追踪
- 笔记沉淀
- 假设管理
- 证据链组织
- 实验过程记录
- 结论版本
- 不确定性标注

它的价值不只是给出一个答案，而是帮助使用者构造一条可复查的认知路径。

因此，研究 Harness 对几个能力特别敏感：

- 来源是否透明
- 证据是否可回看
- 引用是否可验证
- 假设和结论是否分开
- 中间材料是否能复用

如果研究 Harness 只输出一段漂亮总结，却不能保留证据链、引用和推理路径，它就很难支撑严肃研究。

研究场景的核心不是快，而是可追溯。

---

## 6. 工业、EDA 和机器人场景，会把 Harness 推向控制台逻辑

一旦 Agent 接触物理设备、制造流程、EDA 工具链或高价值工程系统，Harness 形态会再次改变。

原因很直接：
错误代价不再是回答不够好，而可能是停线、损耗、越权、错误设计、设备风险或安全事故。

这类场景里，Harness 必须更强调：

- 严格权限分级
- 仿真和预演
- dry-run
- 强制审批
- 实时状态反馈
- 安全停机
- 回退路径
- 事故记录

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/92_harness_forms_across_scenarios_deep_revised__mermaid_04.png)

> 图92-4 高风险场景为什么会把 Harness 推向控制台
> 一旦错误代价显著升高，系统优先级会从流畅交互转向强控制、强反馈和强制动。控制台逻辑会自然压过聊天逻辑。

在这些场景里，顺滑不是最高优先级。
系统控制力、可验证性、可中止性和责任边界更重要。

这类 Harness 更像控制台，而不是聊天产品。

---

## 7. 教育、医疗、法律和金融场景，会把 Harness 推向安全与解释

还有一类场景，动作未必总是直接控制设备，但责任风险很高。

例如：

- 教育
- 医疗
- 法律
- 金融
- 政务
- 合规咨询

这些场景的 Harness 形态会被安全、解释、证据和责任强烈塑造。

系统需要回答：

- 是否应该回答
- 是否需要拒答
- 是否需要提示不确定性
- 是否需要引用来源
- 是否需要人工复核
- 是否需要保留咨询记录
- 是否有监管或合规边界

这类场景不能只追求“回答更像专家”。
它还要保证回答的责任边界清楚。

因此，安全型 Harness 往往会更强调证据展示、限制性提示、人工复核、风险分级和留痕。

---

## 8. 场景不同，但背后的五类结构问题相同

不同场景下的 Harness 外形差异很大。
但背后的结构问题始终相似。

第一，入口。
人如何把任务交给系统，任务是否被结构化。

第二，上下文。
系统如何装配背景、材料、约束、状态和权限过滤后的信息。

第三，动作。
系统可以调用哪些工具，对哪些对象产生影响。

第四，控制。
哪些动作需要审批、预演、限制、回退和审计。

第五，交付。
结果以什么形式进入后续工作。

这五类问题让 Harness 成为一个通用概念。
但每个场景对这五类问题的回答都不同，所以 Harness 又不会拥有统一外观。

它的通用性在结构层，差异性在场景层。

---

## 9. 场景差异会重写状态结构、动作空间、控制强度和交付形式

如果把差异压得更具体，可以看四类系统结构。

第一是状态结构。
编程场景围绕文件、diff、测试和仓库状态；
研究场景围绕资料、笔记、引用和实验过程；
企业场景围绕流程节点、角色和单据状态；
工业场景围绕设备状态、实时反馈和安全边界。

第二是动作空间。
编程 Harness 的动作是读写文件、跑命令、改代码；
研究 Harness 的动作是检索、比对、摘要、组织证据；
企业 Harness 的动作是填单、审批、同步系统；
工业 Harness 的动作可能直接触发高代价控制。

第三是控制强度。
风险越高，控制层越不能后补。
聊天和研究类系统可以更开放，企业、医疗、工业类系统则需要更强调审批、留痕、隔离和回退。

第四是交付形式。
有的场景以文本答复为主，有的以代码 diff 为主，有的以工单、报告、审批记录、控制指令或可执行配置为主。

一旦交付形式不同，整个 Harness 的界面、状态管理和工作流都会被重塑。

## 10. 工程现实：常见失败，是拿错了上一代模板

现实里，很多团队的问题并不是没有做 Harness，而是把错误模板套到了错误场景上。

典型例子包括：

- 用通用聊天框承载复杂项目协作
- 用轻量工具面板承载高风险工业控制
- 用文档流系统承载强实时反馈任务
- 用个人助手骨架承载企业审批流程
- 用生成式问答骨架承载严肃研究证据链

这些失配的共同点是：
系统表面上拥有 AI，结构上却仍停留在上一代产品形态里。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/92_harness_forms_across_scenarios_deep_revised__mermaid_05.png)

> 图92-5 场景失配的典型来源
> 很多失败并不是功能不够，而是产品骨架一开始就借错了模板。拿错骨架，后期会用大量补丁去修一个结构性问题。

Harness 设计的难点，是从任务本体出发，而不是从流行产品长相出发。

每一种工作都有自己的承重结构。
谁先看见那套承重结构，谁就更可能做出贴近任务本体的系统。

## 11. 第92章到第93章：形态之后，是协作流程

本章讨论不同场景下 Harness 为什么会长成不同形态。
但形态还不是终点。

无论是聊天、编程、研究、企业流程还是工业控制，只要系统开始与人协作，就会进入一条真实工作链：

- 人如何提出目标
- 系统如何澄清约束
- 上下文如何被装配
- 工具和权限如何被设置
- 过程如何被观察
- 哪些节点需要审批
- 结果如何验收
- 失败如何接管
- 经验如何沉淀

这就是第93章要继续拆开的内容。
不同场景决定 Harness 的形态；
真实协作流程决定 Harness 怎样被使用。

## 12. 容易误判的地方

### 误解一：不同场景下的 Harness 只是换个皮肤

变化的是任务对象、状态结构、动作空间、控制强度和交付方式，不只是视觉外观。

### 误解二：聊天是最通用的入口，所以也应是最通用的骨架

聊天适合入口和轻协作。
复杂任务通常需要任务面板、工作区、控制台或专门工作台。

### 误解三：行业场景只是功能更多，不涉及底层结构变化

很多行业场景会直接改变核心设计原则，尤其是在权限、实时性、风险和交付物上。

### 误解四：控制越少，用户体验越顺

在高风险场景里，缺少控制不是顺滑，而是把失败成本推迟到事故之后。

### 误解五：先做一个通用 Harness，再慢慢适配行业就够了

可以先做通用入口，但关键行业场景必须尽早重写状态、动作、控制和交付结构。

## 13. 本章小结

1. Harness 不是固定界面，而是围绕任务本体变形的组织结构。
2. 通用聊天、编程、企业流程、研究、工业控制和高责任咨询场景，会把 Harness 推向不同形态。
3. 不同形态背后，都要回答入口、上下文、动作、控制和交付五类结构问题。
4. 场景差异会重写状态结构、动作空间、控制强度和交付形式。
5. 好的 Harness 设计从任务现实出发，而不是从流行模板出发。

核心判断：

> Harness 没有标准长相，只有一个不变原则：它必须长成适合承接那类真实工作的样子。
