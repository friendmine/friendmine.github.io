# 第92章 不同场景下的 Harness 形态

## 先说结论

Harness 没有统一外观，因为它不是一种固定界面，而是一套围绕真实任务不断变形的组织结构。  
场景变了，它的重点、风险和交付方式也会跟着变，所以不存在一个能在所有行业里直接套用的万能骨架。

## 1. 只把 Harness 理解成“一个 AI 产品长什么样”，会立刻陷入表面化

很多人谈不同场景下的 Harness，最先想到的是 UI 风格：聊天框、面板、侧边栏、工作区。  
这些当然重要，但它们只是表层结果。

真正决定 Harness 形态的，不是视觉长相，而是四件更深的事：

- 当前任务如何被表示
- 系统可以对什么对象行动
- 哪些风险必须被控制
- 最终要交付的到底是什么

所以，同样都是 AI 产品，编程助手、研究助手、企业流程助手和机器人控制系统，看起来会完全不同。  
因为它们面对的任务对象，本来就不是同一种东西。

这也是很多团队一开始最容易犯的结构性错误。  
他们并不是没有做 Harness，而是下意识沿用了上一代最熟悉的产品模板。  
结果表面上功能越来越多，骨架却始终不对：该有控制台的地方做成了聊天框，该有工作区的地方做成了消息流，该有审批节点的地方做成了“让模型自己小心一点”。  
场景一旦错判，后面所有局部优化都会越来越像在错误骨架上缝补。

> 图92-1 不同场景下的 Harness 形态对比图  
> 表面上它们都像“带 AI 的工作系统”，但真正决定形态的，是任务对象、风险结构、实时性要求和最终交付物。场景一变，系统骨架就会跟着重写。

## 2. 通用聊天 Harness 的本质，是以对话为中心的轻协作结构

通用聊天场景里，系统通常围绕会话组织。  
用户输入问题，模型输出回答，历史上下文以消息流形式串联起来。  
在这种环境里，Harness 的重点通常是：

- 会话连续性
- 轻量工具接入
- 基本记忆
- 快速输出文本结果

它的优点是上手快，适应面广。  
但它也天然受限，因为“消息流”并不是最适合承载复杂任务状态的结构。  
一旦任务开始依赖文件、审批、长期项目状态和 artifact 交付，仅靠聊天框就会显得越来越吃力。

也正因为如此，聊天式 Harness 往往最适合作为起步形态，而不是最终形态。  
它非常适合验证用户愿不愿意把任务交给系统，也适合承接轻量问题与短回路协作；  
但一旦任务需要跨天持续、多人接力、artifact 流转或复杂审批，消息流就会开始承担本不属于它的状态压力。  
很多产品不是不会做事，而是一直停留在一个只适合“开始对话”的骨架里。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/92_harness_forms_across_scenarios_deep_revised__mermaid_02.png)

> 图92-2 通用聊天 Harness 的基本组织方式  
> 这类 Harness 的优势在于轻和快，但它天然把任务状态压缩进消息流里。只要任务一旦变长、变重、变得需要显式工作区，这种结构就会开始吃力。

## 3. 编程 Harness 更像智能开发工作台，而不是聊天壳

编程场景会立刻暴露出聊天范式的局限。  
开发者真正要处理的是文件、目录、diff、终端、测试结果、版本控制和回滚，而不是一段段彼此脱离环境的回答。

因此，编程 Harness 的核心通常不是“说得多好”，而是：

- 能否正确看见项目结构
- 能否稳定执行命令
- 能否把修改呈现为可审阅的 diff
- 能否在失败时迅速回放和回退

这时，聊天只是一种协商入口。  
系统的真正本体更接近 IDE、终端和审查工具被智能重新组织后的工作台。

从采用角度看，这类 Harness 成败往往不取决于“模型有多聪明”，而取决于它是否尊重开发者原本的工作对象。  
如果系统不能稳定处理 diff、测试、回滚和版本控制，那么它再会解释代码，也很难成为真正的开发工作台。  
编程场景最看重的不是“说得像不像高手”，而是“改完之后敢不敢接进现有工程链”。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/92_harness_forms_across_scenarios_deep_revised__mermaid_03.png)

> 图92-3 编程 Harness 的核心工作对象  
> 编程场景之所以不能停在聊天壳，是因为它真正处理的是文件、命令、diff、测试和回滚。这些对象一旦进入中心，Harness 的本体就会自然变成工作台。

---

## 4. 办公和企业流程 Harness 的重心，在于协同与责任

办公自动化和企业流程类系统，面对的不是代码和命令，而是文档、表格、审批链、角色权限和跨部门协作。  
这类场景下，Harness 的关键就会明显转移：

- 谁可以看到什么
- 哪个节点需要谁批准
- 结果如何进入现有流程
- 每一次变更如何被审计

在这里，单纯把模型做得更“会说”帮助有限。  
真正的难点，是让系统能被嵌进组织已有的责任结构里。  
因此，这类 Harness 往往比个人助手更强调流程托底和权限治理。

也就是说，企业场景里最值钱的往往不是聪明回答，而是责任对齐。  
系统能不能顺利穿过部门边界、角色边界、审批边界和审计边界，决定了它到底是一个“看起来很先进的工具”，还是一个真正能被组织接纳的工作系统。  
这类 Harness 一旦设计失配，组织的第一反应通常不是“效果差一点”，而是“先别让它碰真实流程”。

---

## 5. 研究 Harness 更像知识工作台

研究场景既不像纯聊天，也不像标准流程。  
它需要在开放探索和严谨记录之间维持平衡。

一个研究型 Harness 往往更重视：

- 资料检索与引用
- 笔记与摘要沉淀
- 实验过程追踪
- 假设与结论之间的关系保留

这类系统的价值，不只在于给出一个“答案”，而在于帮助使用者构造一条可复查的认知路径。  
因此，它对上下文结构、知识来源透明度和 artifact 组织方式，会提出与聊天产品完全不同的要求。

---

## 6. 工业、EDA 和机器人场景，会把 Harness 推向控制台逻辑

一旦 Agent 接触物理设备、制造流程或高价值工程系统，Harness 的形态会再次改变。  
因为此时错误代价不再是“回答不够好”，而可能是停线、损耗、越权或安全事故。

在这些场景里，Harness 必须更强调：

- 严格权限分级
- 仿真和预演
- 强制审批
- 实时状态反馈
- 安全停机与回退

换句话说，它更像控制台，而不是聊天产品。  
漂亮交互不再是优先级最高的事，系统控制力才是。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/92_harness_forms_across_scenarios_deep_revised__mermaid_04.png)

> 图92-4 高风险场景为什么会把 Harness 推向控制台  
> 一旦错误代价显著升高，系统优先级就会从“流畅交互”转向“强控制、强反馈、强制动”。这时，控制台逻辑会自然压过聊天逻辑。

---

## 7. 场景虽然不同，但背后的结构问题始终相同

看到这里，很容易以为不同场景下的 Harness 完全不可比较。  
其实不是。  
它们的外形差异很大，但背后的结构问题始终相同：

- 人如何进入系统
- 系统如何组织上下文
- 动作如何被执行和限制
- 过程如何被记录和回放
- 结果如何成为可继续工作的产物

也正因为如此，Harness 才既是一个通用概念，又不会退化成空泛口号。  
它的通用性在结构层，它的差异性在场景层。

## 8. 工程现实：最常见的失败，是拿错了上一代模板

现实里，很多团队的问题并不是没有做 Harness，而是把错误的模板套到了错误的场景上。  
例如：

- 用通用聊天框去承载复杂项目协作
- 用轻量工具面板去承载高风险工业控制
- 用文档流系统去做需要强实时反馈的任务

这些失配的共同点是：系统表面上拥有 AI，结构上却还停留在上一代产品形态里。  
Harness 设计真正困难的地方，就在于你必须从任务本体出发，而不是从流行产品长相出发。

归根到底，不同场景下的 Harness 之所以会长得很不一样，不是因为设计师偏好不同，而是因为每一种工作都有自己的承重结构。  
谁先看见那套承重结构，谁就更可能做出真正贴任务本体的系统；  
谁只是把流行外观贴上去，谁后面就会花大量时间和成本去修一开始就借错的模板。

如果把这个判断再压实一点，可以看到场景差异会直接重写四类系统结构：

- `状态结构`
  编程场景围绕文件、diff、测试和仓库状态；研究场景围绕资料、笔记、引用和实验过程；企业场景围绕流程节点、角色和单据状态；工业场景则围绕设备状态、实时反馈和安全边界。

- `动作空间`
  编程 Harness 的动作是读写文件、跑命令、改代码；研究 Harness 的动作是检索、比对、摘要、组织证据；企业 Harness 的动作是填单、审批、同步系统；工业 Harness 的动作则可能直接触发高代价控制。

- `控制强度`
  风险越高，控制层越不能后补。  
  聊天和研究类系统可以更开放，企业和工业类系统则必须更强调审批、留痕、隔离和回退。

- `交付形式`
  有的场景以文本答复为主，有的场景以代码 diff 为主，有的场景以工单、报告、审批记录或控制指令为主。  
  一旦交付形式不同，整个 Harness 的界面、状态管理和工作流都会被重新塑造。

所以，场景差异不是“加几个功能模块”这么简单，而是会从底层改写系统究竟该长成什么样。

## 9. 容易误判的地方

### 误解一：不同场景下的 Harness 只是换个皮肤

真正变化的是状态结构、动作空间、控制强度和交付方式，不只是视觉外观。

### 误解二：聊天一定是最通用的入口

聊天很通用，但未必最适合承载复杂协作。很多场景最终需要任务面板、工作区和控制台。

### 误解三：行业场景只是功能更多，不涉及底层结构变化

很多行业场景会直接改变 Harness 的核心设计原则，尤其是在权限、风险和实时性上。

## 11. 本章小结

1. Harness 不是一种固定界面，而是一套围绕任务本体不断变形的组织结构。  
2. 通用聊天、编程、办公、研究、工业和机器人场景，会把 Harness 推向完全不同的形态。  
3. 这些形态虽然不同，但都要回答入口、上下文、动作、控制和交付五类核心问题。  
4. 好的 Harness 设计必须从任务现实出发，而不是从流行模板出发。  

核心判断：

> Harness 没有标准长相，只有一个不变原则：它必须长成最适合承接那类真实工作的样子。
