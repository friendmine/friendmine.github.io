# 第101章 Agent 与 Harness 的评测

## 先说结论

执行型 AI 的评测，不能只看最终答案是否正确，还要评估路径、工具调用、状态管理、人工接管、权限边界、产物质量和整套系统是否值得这样运行。

单轮模型评测更像判断“答得对不对”。
Agent 与 Harness 的评测更像判断“一套自动化机制能不能被组织托付”。

![插图 第101章 执行型系统评测靶场]({{ site.baseurl }}/ai_ecosystem_book/images/chapter_101_evaluating_agents_and_harnesses.png)

> 插图 第101章 执行型系统评测靶场

第100章讨论模型评测，重点是判断模型候选能否进入下一步工程流程。
本章讨论 Agent 与 Harness 的评测，重点是判断一个会规划、会调用工具、会改变外部状态的执行型系统，能不能进入真实业务。

---

## 1. Agent 评测的对象不是一句回答，而是一条执行轨迹

普通模型评测通常围绕输入和输出展开。
题目给进去，答案拿出来，再判断是否正确、是否有用、是否安全。

Agent 不一样。
它的价值存在于一条执行轨迹里。

这条轨迹可能包括：

- 任务理解
- 计划拆解
- 上下文检索
- 工具选择
- 参数构造
- 外部系统调用
- 中间结果检查
- 错误恢复
- 人工确认
- 最终产物交付

因此，Agent 评测不能只问“最后有没有完成”。
还要问：

- 它是怎样完成的
- 中间有没有明显绕路
- 有没有越权或误操作风险
- 工具调用是否可靠
- 遇到失败后是否能恢复
- 成本和时延是否可接受
- 人类是否能看懂并接管

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/101_evaluating_agents_and_harnesses_deep_revised__mermaid_01.png)

> 图101-1 Agent 与 Harness 的多维评测图
> 执行型系统不能只靠“答对率”判断，而要同时看任务完成、路径质量、工具成功率、人工接管、产物交付、成本时延和风险控制。

同样一个正确结果，可能来自两条完全不同的路径。
一条路径稳定、经济、可解释；另一条路径反复试错、频繁重试、依赖偶然成功。
如果评测只看结果，两者会被误判为同等质量。

这正是 Agent 评测和普通模型评测的分界线：
它评的不是一个答案，而是一套行为。

---

## 2. Harness 评测更进一步：它评的是智能行为所在的环境

Agent 是执行者。
Harness 是把执行者放进现实任务环境的外壳。

一个成熟 Harness 通常会负责：

- 装配上下文
- 暴露工具
- 管理权限
- 记录 trace
- 呈现中间状态
- 设置审批节点
- 保存 artifact
- 支持人工接管
- 提供回放与审计

所以 Harness 评测比 Agent 评测更宽。
它不只问 Agent 会不会做事，还问这套环境有没有把 Agent 放进正确边界里。

例如，一个 Agent 能够修改文件，不代表 Harness 合格。
评测还要看：

- 修改前是否展示了 diff
- 高风险操作是否需要确认
- 失败后是否能恢复工作区
- 用户是否能理解当前状态
- 操作记录是否可回放
- 权限是否按最小必要原则授予

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/101_evaluating_agents_and_harnesses_deep_revised__mermaid_02.png)

> 图101-2 为什么工具成功率需要单独评测
> 执行型系统的很多失败并不来自模型不会想，而来自工具协议、权限边界、参数构造和外部系统状态。把这些层拆开，问题才不会被笼统归因成“Agent 不稳定”。

很多团队会遇到一种错觉：
Agent 在单个任务上表现不错，但放进产品后体验仍然脆弱。
原因往往不是模型突然变差，而是 Harness 没有把上下文、工具、权限、接管和回放组织成一个可靠环境。

---

## 3. 任务完成率重要，但必须和路径质量一起看

任务完成率是执行型评测里的核心指标。
但它不能单独存在。

一个系统完成了任务，仍然可能不适合上线：

- 它用了过多步骤
- 它频繁调用高成本模型
- 它多次误用工具后才碰巧成功
- 它需要大量人工确认
- 它生成了难以复核的中间状态
- 它对同类任务表现不稳定

因此，任务完成率至少要和这些指标一起读：

- 路径长度
- 工具调用次数
- 工具调用成功率
- 重试次数
- 失败恢复率
- 人工介入率
- 接管耗时
- 端到端时延
- 单任务成本
- 产物返工率

如果一个版本任务完成率略高，但成本翻倍、人工接管增加、路径波动变大，那么它未必是更好的版本。
执行型评测的目标不是证明系统“能做成”，而是判断系统“是否值得这样做成”。

这也是第101章与第102章的连接点。
一旦执行型系统准备进入真实流量，任务完成率、路径成本、推理成本和服务延迟就会共同决定产品是否可持续。

---

## 4. Trace 是执行型系统评测的基本证据

Agent 评测离不开 trace。

Trace 不是普通日志的同义词。
它应该足够完整地记录系统为什么走到某一步：

- 用户原始请求
- 系统收到的上下文
- 中间计划
- 每次工具调用的名称、参数和结果
- 每次模型调用的输入、输出和版本
- 权限判断
- 人工确认节点
- 错误与恢复动作
- 最终 artifact
- 成本、时延和资源消耗

没有 trace，评测很容易停留在主观印象。
系统失败了，团队只能猜：是模型理解错了，检索错了，工具错了，权限错了，还是用户意图本身不清楚。

有了 trace，评测才可以回答更细的问题：

- 失败发生在第几步
- 是计划错误还是执行错误
- 是工具协议问题还是参数构造问题
- 是上下文缺失还是上下文过载
- 是模型能力不足还是 Harness 边界不清
- 这个失败能否用规则、工具修复、提示词、模型升级或产品交互解决

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/101_evaluating_agents_and_harnesses_deep_revised__mermaid_03.png)

> 图101-3 接管成本为什么应进入正式指标体系
> 对生产系统来说，trace 不只是排查材料，也是人工接管材料。人能不能快速理解当前状态，直接决定系统在关键时刻能否被接住。

出版上要把这点说透：
执行型 AI 的评测证据不是一段漂亮输出，而是一条可解释、可复现、可审计的行为链。

## 5. 工具调用要评契约，而不是只评成功或失败

Agent 一旦调用工具，就进入了软件工程世界。

工具评测不能只看“有没有调用成功”。
它还要看工具契约是否被正确遵守：

- 参数类型是否正确
- 必填字段是否完整
- 权限范围是否匹配
- 返回结果是否被正确解析
- 错误码是否被区分处理
- 幂等性是否被尊重
- 重试是否有上限
- 高风险动作是否有确认

这类问题经常被错误地归因到模型。
例如，Agent 调用 CRM 更新客户资料失败，表面看是 Agent 没做好，实际可能是工具 schema 设计含糊、错误信息不可读、权限边界不明确或 Harness 没有提供足够上下文。

工具评测应该拆成几类指标：

- `call validity`：调用格式是否有效
- `argument accuracy`：参数是否符合任务意图
- `execution success`：外部系统是否执行成功
- `result interpretation`：返回结果是否被正确理解
- `recovery quality`：失败后是否采取合理补救
- `side-effect safety`：外部状态变化是否安全可控

工具越多，评测越不能只靠端到端完成率。
端到端分数会把太多问题糊在一起，让团队看不见真正的薄弱层。

---

## 6. 人在环不是兜底口号，而是一组可量化指标

很多执行型系统都会说“必要时由人接管”。
但如果接管过程没有被评测，这句话就只是安慰。

人在环评测至少要回答：

- 系统什么时候请求人介入
- 请求介入的时机是否太早或太晚
- 人能否快速理解当前状态
- 接管前需要补多少上下文
- 接管后能否继续推进
- 人的确认是否被系统正确执行
- 接管记录是否能进入回放集

接管成本不是体验细节，而是生产成本。
如果每次接管都需要人重新阅读十几页上下文、判断 Agent 做过什么、猜测工具执行到哪里，那么自动化收益会被迅速吞掉。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/101_evaluating_agents_and_harnesses_deep_revised__mermaid_04.png)

> 图101-4 Agent 与 Harness 的评测栈
> 成熟评测不会只靠一组离线题目。它需要把离线样本、真实回放、人工接管和线上表现连起来，才能判断系统既“能不能做”，也“敢不敢放量”。

这也解释了为什么 Harness 评测一定要包含界面、状态呈现和 artifact 质量。
如果人看不懂系统正在做什么，就谈不上真正接管。

---

## 7. 评测集要分层：离线样本、真实回放和线上信号各有职责

执行型系统不能只靠一组静态题目评测。

更合理的做法，是把评测集分成三层：

- 离线评测集
- 回放评测集
- 线上评测集

离线评测集适合做快速版本比较。
它成本低、可重复、适合日常回归。
但它容易变得过熟，也很难覆盖真实用户的混乱表达和组织流程。

回放评测集来自真实 trace。
它记录系统曾经失败、卡住、误判、误用工具或需要人工接管的场景。
它的价值在于让团队不重复犯同样的错。

线上评测集来自真实流量。
它观察真实分布下的任务完成率、人工介入率、用户修正率、重试率、成本、时延和投诉信号。
它最接近现实，但风险也最高，所以必须配合灰度、限流和回滚。

少任何一层，评测都会偏。
只看离线样本，会高估稳定性。
只看线上信号，会把风险转嫁给用户。
只看回放样本，又可能过度修复历史问题，忽略新分布。

## 8. 真实工作分布比演示样例重要得多

许多 Agent 项目不是没有评测，而是评测太像 Demo。

Demo 样例通常有几个特点：

- 任务边界清楚
- 工具状态干净
- 用户表达配合
- 失败代价低
- 路径短
- 产物容易判断

真实工作分布恰恰相反。
用户会表达含糊，权限会受限，外部系统会报错，数据会冲突，任务会中途改变，产物需要符合组织标准。

因此，成熟评测集应该覆盖：

- 常见任务
- 高频失败模式
- 长尾边界场景
- 高风险动作
- 多工具协同
- 跨步骤状态依赖
- 权限不足场景
- 需要人工确认的场景
- 产物质量难以自动判断的场景

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/101_evaluating_agents_and_harnesses_deep_revised__mermaid_05.png)

> 图101-5 演示分布与真实工作分布的差别
> 评测集如果太像演示，就会持续高估系统成熟度。只有把真实失败模式、长尾场景和高风险节点放进评测里，评测才会服务发布决策。

出版编辑上，这一节的重点不是批评 Demo，而是明确 Demo 的边界。
Demo 可以证明“可能性”，评测要证明“可托付性”。

## 9. 产物质量要单独评，因为完成任务不等于交付成果

Agent 和 Harness 往往会产生 artifact：
代码、文档、表格、邮件、报告、配置、工单、设计稿、数据分析结果。

这些产物不能只看“有没有生成”。
还要评：

- 是否完整
- 是否可执行或可打开
- 是否符合格式规范
- 是否满足用户意图
- 是否引用了正确数据
- 是否有明显遗漏
- 是否便于人继续编辑
- 是否引入安全或合规风险

例如，一个代码 Agent 最后说“已修复”，但没有测试、没有解释变更、没有处理边界条件，这不是合格交付。
一个文档 Agent 输出了一份报告，但数据来源混乱、结构不适合发布，也不是合格交付。

产物评测通常需要多种方法组合：

- 规则检查
- 单元测试或构建测试
- 格式校验
- 人工 rubrics
- 对比历史优秀样例
- 用户后续修改量
- 发布后错误反馈

执行型 AI 的最终价值往往落在产物上。
如果不单独评产物质量，系统很容易在“流程完成”的指标里掩盖“交付不可用”的事实。

---

## 10. 发布门槛要同时包含能力、成本、风险和可接管性

执行型系统的发布门槛不能只是一句“通过评测”。

更稳妥的发布门槛应当同时包含：

- 关键任务完成率达到目标
- 高风险任务没有不可接受退化
- 工具调用错误率低于阈值
- 人工介入率可接受
- 接管耗时可接受
- 产物返工率可接受
- 单任务成本可接受
- 端到端时延可接受
- 权限与审计链路完整
- 回滚方案明确

这类门槛会让团队不再只追求“模型更聪明”，而是追求“系统更可靠”。

它也能避免一种危险发布：
新版本在少数样例上表现更聪明，但工具误用增加、成本上升、人工接管更困难。
如果发布门槛只看能力分数，这种版本很容易被放出去。
如果发布门槛看完整系统指标，它就会被拦住。

## 11. 工程现实：Agent 评测真正难的，是持续维护失败记忆

执行型系统的评测不是一次性工程。
它需要持续维护。

每一次线上失败，都应该被问清楚：

- 这是新问题，还是旧问题复发
- 是否有 trace
- 是否能回放
- 是否能加入回归集
- 是否需要修改工具契约
- 是否需要修改 Harness 交互
- 是否需要调整权限或审批
- 是否需要修改模型、提示词或路由策略

成熟团队会维护“最不想再犯的一组失败集”。
这些样本未必漂亮，但它们非常有价值，因为它们定义了系统真实的脆弱边界。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/101_evaluating_agents_and_harnesses_deep_revised__mermaid_06.png)

> 图101-6 为什么执行型评测不能做得像 Demo
> 评测如果只负责证明系统“看起来很强”，就会错过它脆弱的地方。成熟团队更愿意用评测提前暴露脆弱性，并把失败记忆转化成回归资产。

这也是执行型评测最重要的组织能力：
不是一次性证明系统好，而是让系统持续不重复犯关键错误。

## 12. 容易误判的地方

### 误解一：只要任务完成了，路径怎么走并不重要

路径决定成本、风险、稳定性和可解释性。
对生产系统来说，路径本身就是质量的一部分。

### 误解二：Agent 评测和普通模型评测差别不大

Agent 评的是多步行为、工具调用、状态变化和人机协作。
它比单轮答案评测多了完整执行链路。

### 误解三：工具失败也是模型能力差

未必。
工具 schema、错误信息、权限边界、上下文装配和外部系统状态，都可能是失败源。

### 误解四：人在环只要有入口就够了

不够。
人能否理解状态、快速接管、继续推进、留下可回放记录，才是真正的接管能力。

### 误解五：评测集越干净越好

执行型系统需要“脏样本”。
真实任务里的含糊表达、权限失败、工具异常和长尾边界，正是评测最有价值的部分。

## 13. 本章小结

1. Agent 评测的对象不是一句回答，而是一条包含计划、工具、状态、错误恢复和产物交付的执行轨迹。
2. Harness 评测评的是环境：上下文装配、权限、接管、回放、审计和 artifact 交付都在评测范围内。
3. 任务完成率必须与路径质量、工具成功率、人工介入率、接管成本、产物质量、时延和成本一起看。
4. Trace、回放集和失败集是执行型系统评测的核心资产。
5. 成熟发布门槛要同时覆盖能力、成本、风险和可接管性。

核心判断：
