# 第101章 Agent 与 Harness 的评测

## 先说结论

执行型 AI 的评测，不能只问答案对不对，还必须追问它是怎样做成的、用了什么代价、过程是否稳定、人在什么时候需要介入，以及整套系统究竟值不值得这样运行。

![插图 第101章 执行型系统评测靶场]({{ site.baseurl }}/ai_ecosystem_book/images/chapter_101_evaluating_agents_and_harnesses.png)

> 插图 第101章 执行型系统评测靶场

对执行型系统来说，评测同时也是发布门槛。  
如果团队只维护漂亮样例，不维护失败集、回放集和高风险路径，那么系统越复杂，评测就越容易失真。

---

## 1. 传统模型评测为什么到了 Agent 时代就不够了

对单轮模型来说，看输出质量往往已经能覆盖大半问题。  
但 Agent 和 Harness 所面对的对象已经不是单次回答，而是多步执行、多工具调用、状态管理和人机协作组成的系统。

因此，仅仅评估“最终答案是否正确”，会遗漏太多关键事实。  
同样一个正确结果，可能来自两种完全不同的路径：

- 一条路径稳定、经济、可解释
- 另一条路径高成本、充满偶然、几乎不可复现

如果评测看不见这两者的差异，团队就很容易把碰运气的成功误判为系统能力。

---

## 2. 路径质量是执行型系统最重要、也最容易被忽视的指标之一

Agent 与普通模型最大的区别，在于它的价值很大一部分存在于路径里。  
它不只是产出一个结果，而是在一连串状态变化中把任务推向结果。

因此，路径评测至少要关心：

- 步骤是否合理
- 是否存在明显绕路
- 是否反复试错
- 是否依赖高风险动作
- 中途有没有偏航又被拉回

这些东西听起来不像传统 AI 指标，却直接决定系统稳定性、可解释性和成本结构。  
路径一旦不稳，结果再漂亮，也难以真正进入生产。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/101_evaluating_agents_and_harnesses_deep_revised__mermaid_01.png)

> 图101-1 Agent 与 Harness 的多维评测图  
> 执行型系统不能只靠“答对率”判断，而必须同时看任务是否完成、过程中人介入了多少、回放是否能解释行为、最终产物是否真的可交付。

---

## 3. 工具成功率必须被单独拆出来看，因为很多问题根本不在模型层

在执行型系统里，失败往往并不来自模型不会想，而是来自它与外部世界连接的那一层不够稳。  
工具协议不一致、参数格式错误、权限被拒绝、返回结果解析失败，都会直接影响任务推进。

因此，工具相关指标必须被单独评估，例如：

- 工具调用成功率
- 失败恢复率
- 参数有效率
- 错误分类分布

只有把这部分单独拆开，团队才知道问题到底在模型层、工具层，还是两者接口层。  
否则，很多原本可以被精准定位的问题，最后都会被含糊地归结成“Agent 不够稳”。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/101_evaluating_agents_and_harnesses_deep_revised__mermaid_02.png)

> 图101-2 为什么工具成功率必须单独评测  
> 执行型系统的很多失败并不来自模型不会想，而来自连接世界的那一层不够稳。把模型、工具和接口拆开评，团队才不会把所有问题都糊成一句“Agent 不稳定”。

---

## 4. 任务完成率当然重要，但它绝不能孤立存在

一个系统任务完成率高，并不自动等于它成熟。  
因为完成任务的方式本身也需要被评估。

例如，同样达到完成率，系统可能存在巨大差异：

- 一个版本路径更短、时延更低
- 一个版本过度依赖人工救场
- 一个版本频繁使用高成本模型
- 一个版本在边缘场景里极不稳定

因此，任务完成率必须与成本、时延、路径长度、人工干预比例等指标联动看。  
否则，团队很容易为了“完成率好看”而默默接受一套并不划算的系统。

这也说明系统评测和模型评测不是一套东西。  
模型评测更偏向能力判断：答得准不准、推理强不强、表达稳不稳。  
系统评测则必须进一步回答：

- 它是不是值得这样运行
- 它有没有把组织流程和工具链一起拖进高成本区
- 它在出错时人能不能接得住

也就是说，模型评测主要判断“这脑子行不行”，系统评测则在判断“这整套机制能不能进现实”。

执行型系统还有一个很容易被忽略的门槛：  
有些路径虽然最后完成了任务，但中间已经消耗了过多人工确认、过多工具重试和过多高价模型调用。  
这种系统在 demo 里看不出问题，一旦进入真实流量，往往会先死于成本和协作摩擦。  
所以执行型评测必须天然带着“发布门槛”意识，而不只是“成功案例统计”意识。

---

## 5. 接管成本和系统稳定性，必须进入评测体系

如果我们承认现实目标是可接管自动化，那么评测就不能只看系统自己做得怎么样，还必须看人类在关键时刻接得住接不住。

这类评测会关注：

- 人多久能看懂当前状态
- 接管前需要补多少上下文
- 接管后继续推进是否顺滑
- 同类任务跨版本行为是否稳定

这些指标看起来不像传统机器学习指标，但它们对生产价值极其关键。  
一个接管成本过高的系统，即使平时表现不错，也很难被组织真正信任。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/101_evaluating_agents_and_harnesses_deep_revised__mermaid_03.png)

> 图101-3 接管成本为什么应进入正式指标体系  
> 对生产系统来说，接管成本不是体验细节，而是现实成本。一套系统如果平时表现不错、关键时刻却接不住，它就很难真正被组织托付。

---

## 6. Harness 评测比 Agent 评测更复杂，因为你评的已经不只是执行，而是环境

Agent 评测聚焦的是执行闭环本身。  
Harness 评测则会进一步涉及：

- 上下文装配是否有效
- 审批节点是否合理
- 界面是否支持接管
- artifact 是否真正可交付
- 权限和回放体系是否成立

也就是说，Harness 评测更接近产品系统评测。  
它既看智能行为，也看这套智能被放进了怎样的协作环境里。  
这也是为什么很多团队会发现：Agent 看起来不错，但真正上线的整套体验仍然很脆弱。

Harness 评测因此天然更像环境评测。  
你评的不只是一个 Agent 会不会行动，还在评这套环境有没有把行动放进正确的边界、正确的上下文和正确的协作结构里。  
这一步一旦缺失，团队就很容易把“单个任务偶尔成功”误判成“整套系统已经成熟”。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/101_evaluating_agents_and_harnesses_deep_revised__mermaid_04.png)

> 图101-4 Agent 与 Harness 的评测栈  
> 真正成熟的评测，不会只靠一组离线题目。它必须把离线样本、真实回放和线上表现连起来，才能判断系统既“能不能做”，也“值不值得这样做”。

如果把这三层再拆清楚，会更容易理解为什么执行型系统评测比普通模型评测更难。

- `离线评测集`
  适合看基本能力是否成立、规则是否明显失效、版本之间有没有基础退化。  
  它回答的是“这套系统在受控条件下能不能站住”。

- `回放评测集`
  适合看历史失败模式有没有被真正修掉、同类 trace 在新版本上会不会再次偏航、artifact 质量是否更稳定。  
  它回答的是“这套系统有没有从真实错误里学到东西”。

- `线上评测集`
  适合看真实用户分布下的干预率、接管成本、路径波动和发布风险。  
  它回答的是“这套系统进了现实之后，到底敢不敢持续放量运行”。

少任何一层，评测都容易失真。  
只看离线，会高估稳定性；只看线上，会把代价压到用户身上；只看回放，又容易错过新分布下的问题。  
三层连起来，团队才真正拥有“发布、回滚、修复、再发布”的判断基础。

---

## 7. 真正成熟的执行型评测，必须贴近真实工作分布

几个演示任务的成功，说明不了太多问题。  
执行型系统一旦进入真实环境，就会面对大量长尾情况和复杂约束。

因此，评测集不能只由“最好展示的样例”构成。  
它还应该覆盖：

- 常见失败模式
- 多步复杂任务
- 高风险动作节点
- 人在环场景
- 跨角色、多工具协作场景

只有贴近真实工作分布，评测才有资格指导产品决策。  
否则，它更像一组用来鼓舞团队士气的样板题。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/101_evaluating_agents_and_harnesses_deep_revised__mermaid_05.png)

> 图101-5 演示分布与真实工作分布的差别  
> 很多团队并不是完全没有评测，而是评测集太像 demo。只有把真实失败模式、长尾场景和高风险节点放进评测里，评测才真正服务发布决策。

---

## 8. 工程现实：很多团队不是没有评测，而是评得太像 Demo

现实里，许多团队并非不做 Agent 评测，而是把评测做得太像演示。  
他们会挑选一组熟悉、顺利、容易成功的任务，最后得出“系统效果不错”的结论。  
一旦进入真实使用，问题立刻暴露：

- 路径波动大
- 工具成功率不稳
- 人工接管成本高
- 用户场景差异极大

这说明执行型 AI 的评测，本质上不是展示能力，而是暴露脆弱性。  
谁越早把脆弱性纳入评测，谁的系统越可能真正成熟。

真正成熟的团队，甚至会专门维护“最不想再犯的一组失败集”。  
这些失败集未必漂亮，却比那些顺滑样例更有价值，因为它们定义了系统目前最真实的脆弱边界。  
执行型评测一旦失去这层“坏样本记忆”，团队就很容易在一轮轮看起来成功的发布里重复踩回旧坑。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/101_evaluating_agents_and_harnesses_deep_revised__mermaid_06.png)

> 图101-6 为什么执行型评测不能做得像 Demo  
> 评测如果只负责证明系统“看起来很强”，就会错过它真正脆弱的地方。成熟团队更愿意用评测去提前暴露脆弱性，而不是遮住它。

---

## 9. 容易误判的地方

### 误解一：只要任务完成了，路径怎么走并不重要

路径决定成本、风险、稳定性和可解释性，因此绝对重要。

### 误解二：Agent 评测和普通模型评测差别不大

执行链路、工具系统、状态管理和人在环，使问题复杂得多。

### 误解三：接管成本不是技术指标，没必要量化

对生产系统来说，接管成本就是现实成本，完全值得进入指标体系。

## 11. 本章小结

1. Agent 与 Harness 的评测必须超出单轮答案质量，进入路径、工具、接管和环境层。  
2. 路径质量、工具成功率、任务完成率、成本、接管成本和稳定性，都应成为核心指标。  
3. Harness 评测比 Agent 评测更接近完整产品系统评测。  
4. 真正有价值的执行型评测，必须贴近真实工作分布，而不是贴近演示分布。  

核心判断：

> 对执行型 AI 来说，真正重要的不只是能不能做成，而是怎样做成、做得稳不稳、出了问题人能不能接住，以及整套系统究竟值不值得这样运行。
