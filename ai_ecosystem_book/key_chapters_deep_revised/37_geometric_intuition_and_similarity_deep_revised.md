# 第37章 几何直觉与相似度

上一章讲了向量、矩阵和张量：对象进入模型之前，要先被写成可计算的数字结构。第37章继续追问下一步：对象已经进入向量空间之后，模型如何判断它们之间的关系。

很多 AI 能力表面上像是在理解语义、意图、图像内容或用户偏好，底层却经常要先落成空间问题：谁和谁更近，谁和谁方向更像，哪些对象聚成一片，哪些特征主要沿某个方向变化。几何直觉与相似度，就是理解这些系统行为的入口。

这不能简单理解成“相似度等于理解”。更准确的说法是：相似度提供了一种可计算的关系判断，模型和工程系统会把这种判断用于检索、推荐、匹配、对齐、注意力分配和上下文选择。它有能力，也有边界。

## 1. 对象进入向量空间后，问题会变成几何问题

一个词、一段文档、一张图片的一块区域、一段代码、一个用户请求，只要被编码成向量，就会被放进某个空间。模型随后面对的不是原始对象本身，而是这些对象在空间中的坐标和关系。

这时，许多问题会改写成几何问题：

- 两个对象是否接近
- 两个向量方向是否一致
- 某组对象是否聚在同一区域
- 某种属性是否沿某个方向增强
- 一个查询应该匹配空间中的哪些邻近点

这种转写让复杂对象可以被计算系统处理。语言中的“相近语义”、推荐里的“相似兴趣”、视觉里的“相似局部结构”、代码里的“相似功能”，都可以在表示空间中被转化为距离、方向、投影和邻域关系。

## 2. 相似度是当前空间里的关系判断

“相似”在日常语言里带有很强的直觉含义，容易让人以为两个对象接近，就说明它们在现实中也属于同一类。模型里的相似度要更具体：它描述的是在当前模型、当前训练目标、当前表示空间下，两个对象的向量关系。

两个对象相近，可能意味着它们语义接近，也可能意味着它们经常出现在相似上下文中，或者在某类任务里会触发相似行为。它还可能来自数据采样偏差、标签定义、训练目标设计或空间压缩带来的混淆。

因此，相似度并不是对现实对象的终极判断。它更像一种计算关系，回答的是：“在这套表示空间里，这两个对象是否足够邻近或方向足够一致？”

这一区分很关键。很多检索、推荐和 RAG 失败，并不是向量相似度公式写错了，而是空间里学到的“邻近”与业务中需要的“相关”不是同一件事。用户要找能解决当前故障的文档，系统却返回措辞相似但版本过旧的材料，就是典型例子。

## 3. 点积、距离和余弦，是空间判断的基础语言

要让空间关系进入系统，需要把“近不近”“像不像”写成可计算的指标。点积、距离和余弦相似度就是基础工具。

欧氏距离关注两点之间相隔多远。点积关注两个向量是否在相近方向上共同增长。余弦相似度会弱化长度影响，更强调方向一致性。在 embedding 检索、向量数据库、推荐系统、attention 和多模态匹配中，这些指标都很常用。

常见形式可以写成：

```text
dot(x, y) = x · y = Σ_i x_i y_i
||x - y||_2 = sqrt(Σ_i (x_i - y_i)^2)
cos(x, y) = (x · y) / (||x|| ||y||)
```

选择哪一种指标，不只是数学偏好，也会改变系统行为。归一化之后使用余弦，和直接使用点积，可能让排序结果发生变化；保留长度信息，或只关注方向信息，也会得到不同解释。工程系统里所谓“相关性”，往往就落在这些度量选择和归一化策略上。

## 4. 投影和子空间让表示结构可以被分析

向量空间不是一团不可解释的数字云。许多模型表示内部，会形成相对稳定的方向、区域和子空间。投影和子空间的价值，就在于帮助我们观察这些结构。

同一个向量里，可能混合了主题、风格、情绪、语法、身份线索、任务信息和上下文位置。投影可以用来观察某个向量在特定方向上的成分，子空间可以用来描述一组特征共同占据的结构区域。

最小的投影可以写成：

```text
proj_u(x) = (x · u / ||u||^2) u
```

它表达的是：如果我们关心方向 `u`，对象 `x` 在这个方向上有多少分量。很多模型解释工作都会借助类似直觉，观察某些属性是否可以被线性分离，某类能力是否集中在某些方向上，或者某种偏差是否沿空间中的特定方向放大。

当然，能被投影分析，并不意味着表示已经被人类语言清楚拆开。高维空间中的方向常常是多义的，子空间之间也可能纠缠。投影提供的是分析入口，不是简单答案。

## 5. 高维空间会削弱低维直觉

人类熟悉的几何经验大多来自二维和三维世界。模型表示空间往往有几百维、几千维，甚至更高。进入这种空间后，很多低维直觉会变得不可靠。

在高维空间里，距离分布可能变得更集中；最近邻未必像低维图像中那样“很近”；方向差异可能比距离差异更有解释力；局部聚类也可能受数据密度、索引结构和度量方式影响。

这会直接影响工程系统。向量数据库返回的最近邻，未必就是用户所需内容；RAG 检索到的段落，可能只是表述相近，却缺少问题所需的必要约束；推荐系统找到的相似用户，可能在统计行为上接近，却在当前场景下需求不同。

所以，高维几何不能直接套用日常直觉。它需要和训练目标、数据分布、度量方式、索引误差、业务定义一起理解。

## 6. embedding 的价值在于塑造可运算的邻近关系

embedding 常被讲成“把词或对象查表变成向量”。这个说法没有错，但容易低估它的意义。embedding 更重要的作用，是把对象之间的关系塑造成空间结构。

在训练过程中，相似上下文里的词会被推向更接近的位置；功能相近的代码片段可能进入相邻区域；视觉上或语义上相关的图像块会形成可比较的表示；用户行为、商品属性和内容主题也可以被放进同一个空间。

于是，许多原本需要规则描述的关系，开始以邻近、方向和区域的方式存在。系统可以用这些空间关系做检索、聚类、推荐、去重、对齐和迁移。

但 embedding 空间不是静止资产。模型版本变化、训练数据变化、归一化策略变化、业务标签变化，都可能让空间结构漂移。一个线上系统如果依赖 embedding，就需要管理版本、重建索引、校准召回、评估漂移，并监控空间关系是否仍然服务当前任务。

## 7. RAG 和向量检索暴露了相似度的工程边界

RAG 是理解相似度边界的好例子。用户提出问题后，系统会把查询变成向量，再去文档库中找相似片段，把检索结果交给模型生成回答。

这条链看起来直接，实际每一环都可能出错。查询向量可能没有表达出用户要寻找的约束；文档切分可能打断关键上下文；相似度排序可能偏向措辞相近的段落；近似最近邻索引可能牺牲一部分精确性；召回片段可能缺少时间、版本、权限或场景条件。

因此，“数据库里有答案”不等于“系统能找到答案”。答案是否能被找到，取决于文档如何切分、embedding 如何训练、索引如何构建、相似度如何定义、召回多少片段、是否重排、是否做过滤，以及生成模型如何使用这些上下文。

这也是本章需要强调的工程判断：相似度是入口，还不是完整解决方案。可靠的检索系统通常要把向量召回、关键词匹配、结构化过滤、重排序、权限控制、时效性判断和答案验证组合起来。

## 8. attention 也可以从相似度角度理解

相似度不只出现在检索系统里，也出现在模型内部。Transformer 的 attention 机制，就可以从“查询和键之间的相似度”来理解。

在自注意力中，每个位置会产生 query、key 和 value。系统计算 query 与 key 的匹配程度，再用这个匹配程度决定从哪些 value 中汇聚信息。简化地看：

```text
Attention(Q, K, V) = softmax(QK^T / sqrt(d_k)) V
```

这里的 `QK^T` 就是在计算一组相似度或匹配分数。softmax 把这些分数转成权重，随后再加权汇聚 `V`。这说明相似度不仅用于外部检索，也参与模型内部的信息路由。

当然，attention 里的相似度不是简单语义相似。它还受位置、上下文、层数、头的分工和训练目标影响。不同 attention head 可能关注句法关系、局部依赖、远距离引用、格式结构或任务相关线索。把 attention 简化成“找相似词”会丢掉很多细节，但从相似度和加权汇聚的角度入手，能帮助读者理解它为什么有效。

## 9. 这一章如何接到概率与不确定性

第37章讨论的是对象之间的空间关系。下一章会把问题推进到概率：在这些表示和关系的基础上，模型如何面对不确定性，如何给不同结果分配可能性。

二者是连续的。表示空间告诉系统“哪些对象更接近、哪些方向更相关”；概率分布进一步告诉系统“在当前条件下，哪些输出更可能”。检索、推荐、分类、语言建模和生成系统，都同时依赖这两层：先有表示和关系，再有排序、置信度、采样和决策。

这也是中册逐步展开的主线：对象先被编码成向量、矩阵和张量；这些对象在空间中形成几何结构；随后，模型在这些结构之上经营概率和不确定性。

---

## 10. 本章收束

对象进入向量空间后，许多问题都会改写成距离、方向、投影和邻域关系。点积、距离和余弦相似度，是检索、推荐、attention 和多模态匹配中的基础度量；投影和子空间则提供了观察表示结构的入口。

但这些几何关系不能被误读成现实对象的终极定义。高维空间会削弱低维直觉，最近邻和真实相关性之间经常存在落差。embedding 的价值在于塑造可运算的邻近关系，同时也要求系统管理版本、漂移和任务适配。RAG 和 attention 只是两个典型场景：前者暴露相似度的工程边界，后者说明相似度也会参与模型内部的信息路由。
