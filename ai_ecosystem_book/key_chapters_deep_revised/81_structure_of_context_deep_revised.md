# 第81章 Context 的组成结构

## 先说结论

Context 不是一团长文本，而是一种有限认知带宽下的结构化资源分配。

但一旦这个精心组装的长上下文“塞过了头”，模型就可能出现明显的注意力衰减：前面重要内容被忽略，后面新进来的局部信息却被过度放大。
![插图 第81章 上下文装配板]({{ site.baseurl }}/ai_ecosystem_book/images/chapter_081_structure_of_context.png)

> 插图 第81章 上下文装配板

上下文不是一坨输入文本，而是由多种职责不同的信息层拼装出来的工作环境。真正成熟的上下文设计，重点往往不在“更长”，而在“更有秩序”。

---

## 1. 只有把 Context 拆开看，它才可能被真正设计

如果把上下文理解成“模型看到的一长段字”，就很难系统化设计它。  
真正成熟的做法，是把 Context 拆成功能层，看清楚谁负责给方向，谁负责给状态，谁负责给约束，谁负责给环境信息。

这样系统才能知道该补哪一层，而不是一味加字数。  
所以 Context 的第一步不是堆信息，而是做结构分工。

这一步如果不先做，后面几乎所有优化都会变形。  
团队会不断往窗口里塞更多内容，却说不清到底哪一层负责方向、哪一层负责状态、哪一层负责环境、哪一层负责长期背景。  
结果看起来像信息越来越丰富，实际却是在把不同职责的东西混成一团。  
Context 一旦失去分层，后面所有“加信息”的努力都很容易变成加噪声。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/81_structure_of_context_deep_revised__mermaid_01.png)

> 图81-1 Context 的分层组成  
> 一个成熟上下文并不是“把一切都塞进去”，而是把规则、任务、环境、记忆和外部知识这几类职责不同的信息，装配成模型当前真正可见的工作环境。

## 2. 指令上下文负责方向，决定系统到底要去哪里

指令上下文定义任务目标与行为边界。  
它回答的是：

- 要做什么
- 不要做什么
- 结果要长成什么样
- 在什么边界内行动

没有这层，即使信息再多，模型也可能从一开始就跑错方向。  
所以指令上下文的职责不是补资料，而是立航向。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/81_structure_of_context_deep_revised__mermaid_02.png)

> 图81-2 指令上下文负责方向，决定系统到底要去哪里

## 3. 用户上下文和任务上下文必须分开，因为它们服务的是两种完全不同的连续性

用户上下文强调的是“这个人是谁、偏好是什么、历史关系如何”；  
任务上下文强调的是“当前这件事进展到哪里、还缺什么、优先级如何”。

两者经常同时存在，但职责完全不同。  
一个偏长期关系连续性，一个偏当前工作连续性。  
如果把它们混在一起，记忆和状态管理就会迅速变乱。

现实里这类混淆非常常见。  
例如某个用户在上一个项目里的长期偏好，被系统错误抬升成当前任务的强约束；  
或者当前任务里的临时决定，又被误写成了面向未来所有任务的稳定背景。  
一旦长期关系连续性和当前工作连续性没有拆开，系统就会越来越像“记住了很多”，但在关键时刻总拿错东西。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/81_structure_of_context_deep_revised__mermaid_03.png)

> 图81-3 用户上下文和任务上下文必须分开，因为它们服务于不同连续性

## 4. 环境上下文和工作空间上下文，是执行型系统真正踩着的地面

很多任务并不只发生在语言层，还发生在具体环境里。  
例如：

- 当前有哪些文件
- 代码仓库处于什么状态
- 哪些网页或文档已打开
- 哪些资源可访问
- 当前屏幕或系统界面是什么

这些内容都属于环境或工作空间上下文。  
没有它们，Agent 往往只能空想，因为它根本不知道自己正站在什么现实地面上。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/81_structure_of_context_deep_revised__mermaid_04.png)

> 图81-4 环境和工作空间上下文，是执行型系统真正踩着的现实地面

## 5. 工具状态与历史上下文，让系统知道“刚才发生了什么，现在接下来该做什么”

工具不是静态图鉴，它们也有当前状态：

- 哪些工具可用
- 上一步调用了什么
- 返回了什么结果
- 哪些动作已经完成

再加上历史上下文，系统才能真正理解“刚才发生了什么、现在接下来该做什么”。  
这对多步任务尤其关键。  
否则，模型很容易重复做事、漏掉关键约束，或者在已经失败过的路径上再次尝试。

> 图81-5 工具状态与历史上下文，让系统知道“刚才发生了什么”

## 6. 外部知识上下文是动态供给层，不是永久常驻层

除了内部历史和当前状态，很多任务还需要外部知识：

- 文档
- 检索结果
- 数据库记录
- 企业知识库

这些内容通常不是永久放在窗口里，而是按需检索、压缩、装配进来。  
这使上下文不再只是静态拼接，而开始像一套动态供给系统。

所以 Context 的组成结构，本身就暗示了一种信息流：  
有些层常驻，有些层按需进入，有些层只在特定阶段重要。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/81_structure_of_context_deep_revised__mermaid_06.png)

> 图81-6 外部知识上下文是动态供给层，而不是永久常驻层

## 7. 工程里最难的，不是层够不够多，而是层与层如何协同

现实系统里，上下文最难的地方通常不是“有哪些层”，而是这些层如何协同：

- 顺序怎样排
- 哪些层固定常驻
- 哪些层按需动态进入
- 发生冲突时谁优先
- 哪些信息必须压缩后才能进入

这决定了系统最终是否稳定。  
层次再丰富，如果协同糟糕，结果只会是更长的噪声。

真正难的不是能不能列出很多层，而是冲突时谁说了算。  
系统提示、当前任务、环境状态、检索材料和历史记忆很少会永远一致。  
一旦没有优先级和让位机制，每一层都像真的，模型就会在一个“处处有信息、处处都不完全对”的混合地面上继续推理。  
这类失败最麻烦，因为它不像完全缺信息那样显眼，反而更像系统一直在认真做错事。

这里有两个特别常见、也特别容易被低估的工程问题：

第一个是`上下文污染`（Context Pollution）。  
所谓污染，并不是几个错别字，而是那些本该被清理掉的过时结果、失效目标和无关材料，长期占据了有限的高优先级窗口，把真正重要的信号挤出了局。  
一旦发生这种情况，模型看到的“现实”就已经被改写，后面的推理再认真，也可能只是沿着错误前提一路推进。

第二个是`上下文竞争`。  
窗口总是有限的，但规则、任务、记忆、检索和环境状态都想进入。  
当这些层同时争抢有限位置时，真正的问题就变成：

- 谁该常驻
- 谁该压缩
- 谁该后置
- 冲突时谁优先

这时候最常见的失败不是某一层单独完全错误，而是优先级关系错了。  
例如系统提示里仍保留着旧流程，任务层已经切到新需求，记忆层又注入了上一个项目的偏好，结果模型会在一个“每层都像是真的”混合上下文上继续推理。  
这就是典型的上下文污染。  
污染一旦发生，后面即使每一步推理都很认真，也只是在错误地面上越走越远。

这就是为什么上下文设计最终会变成资源分配问题。  
Context 不是无限容器，而是一块昂贵而有限的认知带宽。  
谁占据这块带宽，谁就更可能决定模型下一步怎么想、怎么做。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/81_structure_of_context_deep_revised__mermaid_07.png)

> 图81-7 上下文装配中的常驻层、动态层与按需层  
> 并不是所有信息都应该平权进入窗口。真正成熟的上下文装配，必须先区分哪些规则必须常驻，哪些状态应当动态刷新，哪些知识只在特定时刻按需进入。

换句话说，上下文装配不是“把信息往里扔”，而是持续做优先级管理。  
通常更稳的原则是：

- 规则和当前任务目标优先级最高，应该尽量稳定常驻
- 最近状态和关键工具结果需要持续刷新，但不能无上限膨胀
- 检索材料和背景记忆应按需进入，并在完成阶段使命后及时退出

如果没有这层优先级意识，系统就很容易在窗口里出现一种表面丰富、实则失焦的状态：  
最关键的东西没有被固定住，最临时的东西却不断挤占注意力。  
这正是很多上下文系统“字很多、信息也不少，但模型总抓不住重点”的根源。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/81_structure_of_context_deep_revised__mermaid_08.png)

> 图81-8 Context 结构真正难的，不是层数够不够多，而是层与层如何协同和争抢有限带宽

## 8. 容易误判的地方

### 误解一：上下文结构越复杂越好

过多层次也会带来噪声和维护成本，关键是分工清楚而不是层数堆叠。

### 误解二：用户上下文和任务上下文差不多

一个偏长期关系，一个偏当前状态，职责完全不同。

### 误解三：环境信息只是附属材料

对执行型系统来说，环境上下文经常决定成败。

## 10. 本章小结

1. Context 必须被拆成职责不同的信息层，才能真正被设计。  
2. 指令上下文负责方向，用户与任务上下文分别服务不同连续性。  
3. 环境和工作空间上下文是执行型系统的现实地面。  
4. 工具状态与历史上下文让系统知道“刚刚发生了什么”。  
5. 外部知识上下文通常是动态供给层，而不是永久常驻层。  

核心判断：

> Context 不是一段长文本，而是模型开展工作的整个信息场景，而真正的工程难度在于这些信息层如何被有秩序地拼装起来。
