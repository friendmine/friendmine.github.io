# 第109章 训练集群与推理集群

## 先说结论

训练集群和推理集群看起来都在“跑模型”，但它们服务的是两套不同的系统目标：前者在制造能力，后者在交付能力。

训练集群更像一座连续生产模型能力的工厂，关注总吞吐、扩展效率、长时间稳定和实验推进。
推理集群更像一套在线服务系统，关注首 token 延迟、并发、服务等级、缓存管理和单位请求成本。

![插图 第109章 两种工厂布局对照图]({{ site.baseurl }}/ai_ecosystem_book/images/chapter_109_training_clusters_and_inference_clusters.png)

> 插图 第109章 两种工厂布局对照图

第108章讲精度变革，本质上已经把模型计算从“数学格式”拉向“系统资源配置”。
本章继续往下走：同一套 AI 硬件，在训练和推理两侧会被组织成完全不同的基础设施制度。

---

## 1. 如果不区分训练与推理，就会把基础设施误解成“更多机器”

很多人刚接触算力基础设施时，会把问题想成线性扩展：
模型更大，就堆更多卡；用户更多，就再堆更多机器。

这个理解太粗。
训练和推理虽然都在运行模型，但目标函数不同。

训练关心的是：

- 总吞吐
- 扩展效率
- 长时间稳定运行
- 大规模并行效率
- checkpoint 与故障恢复
- 实验迭代速度

推理关心的是：

- 首 token 延迟
- 每 token 生成速度
- 并发吞吐
- 尾延迟稳定性
- KV cache 和显存占用
- 单次请求成本
- 服务等级和可用性

同样是“跑模型”，训练在回答“怎样把能力制造出来”，推理在回答“怎样把能力稳定交付出去”。
这两类问题如果混在一起，系统设计很快就会错位。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/109_training_clusters_and_inference_clusters_deep_revised__mermaid_01.png)

> 图109-1 为什么不能把训练与推理都理解成“多堆机器”
> 训练和推理虽然都在用集群，但它们优化的不是同一个目标，因此不能用同一套基础设施直觉去看。

训练侧可以接受更长作业，只要长期吞吐和扩展效率足够好。
推理侧则必须面对用户请求、排队、缓存、尾延迟和成本线。
这就是两套系统世界的分界。

## 2. 先把两类集群放在一张对照图里看

如果只靠文字描述，读者很容易误以为训练和推理只是规模不同。
更准确的理解，是把它们放在同一张目标对照里。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/109_training_clusters_and_inference_clusters_deep_revised__mermaid_02.png)

> 图109-2 训练集群与推理集群的目标差异
> 同样是“跑模型”，训练在制造能力，推理在交付能力，二者面对的核心约束明显不同。

训练集群的典型问题是：
这次训练能不能在预算时间内完成，能不能稳定跑过数天甚至数周，扩到更多卡后效率是否还能接受，故障后能不能从 checkpoint 恢复。

推理集群的典型问题是：
用户多久看到第一个 token，长短请求能不能混跑，高峰期能不能抗住，P95/P99 延迟是否稳定，每百万 token 成本是否可接受。

这两类问题不是同一套指标的大小差异。
它们从一开始就在优化不同东西。

---

## 3. 训练集群的关键，是把巨大实验拆成可扩展的工业流程

大模型训练很难在单机单卡上完成。
训练集群的任务，是把一个巨大实验拆成可以在成百上千甚至更多加速器上持续推进的工业流程。

常见并行方式包括：

- 数据并行：不同设备处理不同 batch，再同步梯度
- 张量并行：把同一层计算切到多个设备
- 流水并行：把不同层或阶段分布到不同设备
- 序列并行：围绕长序列拆分激活和中间状态
- ZeRO / optimizer sharding：把参数、梯度和优化器状态拆开管理

这些方法表面上是在分任务，实质上是在和显存容量、通信开销、同步效率和故障概率做长期拉锯。

从目标函数直觉上看，训练集群更接近追求：

$$
\text{总训练吞吐} \uparrow,\quad \text{扩展效率} \uparrow,\quad \text{失败恢复成本} \downarrow
$$

训练通常可以接受较高单步时延。
只要整体吞吐更高，作业能稳定推进，实验能按计划完成，系统就是有价值的。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/109_training_clusters_and_inference_clusters_deep_revised__mermaid_03.png)

> 图109-3 训练集群更像制造能力的工厂
> 训练集群关心的不是单步快不快，而是总吞吐和扩展效率能不能把一场巨大实验持续推进下去。这使它更像制造系统，而不是即时响应系统。

这也是为什么训练集群会特别关注网络拓扑、all-reduce、参数同步、checkpoint、作业调度和故障隔离。
它不是把机器连起来就结束，而是要让这些机器像一条长期运行的生产线一样工作。

---

## 4. 训练集群真正怕的，是扩展效率和稳定性同时下降

训练规模扩大后，系统难点会迅速变化。
单卡性能固然重要，但大规模训练更怕这些问题：

- 通信时间吃掉计算收益
- 数据加载跟不上计算
- 某些节点拖慢整体同步
- checkpoint 写入和恢复成本过高
- 长作业中硬件故障概率上升
- 调度系统无法合理安排大作业
- 实验中断后复现和恢复困难

训练集群的关键指标，不只是“单卡快不快”，而是“更多卡加入后是否仍然划算”。

一个粗略直觉是：

`有效训练效率 ≈ 理论计算能力 × 利用率 × 扩展效率 × 稳定运行时间`

只要其中一项明显下降，真实训练效率就会被拉低。
很多训练集群不是输在没有足够算力，而是输在算力无法长期、稳定、协调地被使用。

这也解释了为什么 checkpoint 和容错不是边角问题。
当训练作业持续数天或数周时，任何一次故障都可能浪费大量计算。
成熟训练基础设施必须把失败当成常态，而不是异常。

---

## 5. MoE 和专家并行，让训练集群进入选择性计算时代

随着 MoE 等架构出现，训练集群又多了一层复杂度。
MoE 的吸引力在于：总参数可以很大，但每次只激活部分专家。

这带来了规模化机会，也带来新的系统难题：

- 路由是否稳定
- 专家负载是否均衡
- token 分发是否造成通信热点
- 稀疏激活是否真的节省了系统成本
- 专家并行和数据并行如何组合
- 训练过程中是否出现少数专家过载或退化

MoE 让训练集群不再只是同步大矩阵计算平台。
它更像一个需要实时分拣、路由和负载均衡的选择性计算工厂。

如果路由不均衡，有些设备会忙，有些设备会等。
如果通信不可控，稀疏计算节省下来的 FLOPs 可能被网络成本吃掉。
如果工程工具不成熟，MoE 的理论收益会很难兑现。

所以，专家并行不是单纯模型架构问题。
它会直接改写训练集群的通信、调度和监控方式。

---

## 6. 推理集群更像在线服务系统，而不是训练工厂

推理集群面对的是持续在线请求。
这里的问题不再是“把模型训完”，而是“把能力稳定交付给用户”。

推理集群关心：

- TTFT：用户多久看到第一个 token
- TPOT：后续 token 生成速度
- P95 / P99 尾延迟
- 并发吞吐
- 请求排队时间
- KV cache 占用
- 服务可用性
- 单请求或单 token 成本

从目标函数直觉上，它更接近同时平衡：

$$
\text{延迟} \downarrow,\quad \text{并发} \uparrow,\quad \text{成本} \downarrow,\quad \text{稳定性} \uparrow
$$

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/109_training_clusters_and_inference_clusters_deep_revised__mermaid_04.png)

> 图109-4 推理集群更像交付能力的在线系统
> 推理集群不是在“做实验”，而是在持续面对真实请求。它需要把排队、缓存、调度和服务等级一起组织好，因此更接近在线服务系统。

训练侧可以用较大的 batch 追求长期吞吐。
推理侧不能简单照搬。
在线请求有用户等待，有服务等级，有峰谷流量，有长短请求混合，也有不同价值请求之间的优先级。

这就是为什么推理集群更像服务系统。
它不仅要算，还要排队、隔离、降级、缓存、路由和计费。

---

## 7. 推理集群的核心资源，不只是 GPU，而是缓存、队列和服务等级

推理服务里，硬件当然重要，但真正紧张的常常是组合资源。

关键资源包括：

- 显存：容纳模型权重、激活和 KV cache
- 带宽：读取参数和缓存
- 队列：安排不同请求进入执行
- batch 槽位：决定吞吐和时延折中
- cache 空间：决定长上下文和并发能力
- 网络：支撑多卡和跨节点推理
- 服务等级：决定谁优先、谁降级、谁等待

这意味着推理集群的调度不是普通任务调度。
它要同时处理长请求、短请求、高价值请求、后台任务、批量任务、交互式请求和不同模型路由。

常见机制包括：

- continuous batching
- 动态 batching
- prefix / prompt cache
- KV cache 分页或分块管理
- 小模型和大模型路由
- speculative decoding
- 请求优先级和限流
- 降级与 fallback 策略

这些机制都在回答同一个问题：
怎样把有限硬件组织成稳定在线服务，而不是只在 benchmark 里跑出漂亮数字。

## 8. 同一个模型，在训练和推理两侧会被不同目标重写

训练阶段，为了提升总吞吐和扩展效率，可以接受更大的 batch、更重的同步、更长的单步时间。
推理阶段，用户体验会直接受到首 token 延迟、排队时间和尾延迟影响。

因此，同一个模型在两类集群上会被不同方式优化：

- 训练侧重视数值稳定、梯度同步、激活重计算、optimizer state 和 checkpoint
- 推理侧重视量化、KV cache、请求调度、低延迟 kernel、模型路由和单位成本
- 训练侧关注作业完成时间和实验迭代
- 推理侧关注在线服务质量和长期运营成本

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/109_training_clusters_and_inference_clusters_deep_revised__mermaid_05.png)

> 图109-5 同一个模型在两类集群上的优化重心
> 训练和推理不是同一个问题的两个规模版本，而是两个目标函数不同的系统世界。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/109_training_clusters_and_inference_clusters_deep_revised__mermaid_06.png)

> 图109-6 两类集群的资源组织目标
> 训练集群首先在组织“怎样把更多机器一起高效干活”，推理集群首先在组织“怎样把有限机器更稳定地服务给更多真实请求”。看清这层差异，很多基础设施决策才不会混线。

这也解释了一个常见现象：
训练侧看起来非常顺手的架构，到了推理侧可能因为 KV cache、decode 吞吐、通信和调度成本变得很难服务。
反过来，一个为低成本推理做了强约束的模型，也可能在训练阶段牺牲能力上限或实验灵活性。

## 9. 同一批硬件，在两侧会长出两种使用哲学

同样是 GPU 或 AI 加速器，训练和推理会用出不同性格。

训练侧更像大作业系统：

- 作业时间长
- 资源占用集中
- 同步和通信规律较明确
- 更看重集群整体效率
- 更愿意为总吞吐牺牲单步时延

推理侧更像在线服务系统：

- 请求持续到来
- 流量峰谷明显
- 长短请求混合
- 服务等级不同
- 需要快速扩缩容和降级
- 更看重尾延迟和单位成本

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/109_training_clusters_and_inference_clusters_deep_revised__mermaid_07.png)

> 图109-7 同一批硬件在两侧会被按不同方式使用
> 训练和推理不是两种规模，而是两种系统世界。同一硬件会因为目标函数不同，长出两种使用哲学。

如果团队把两侧混成一种基础设施，就容易出现错配：
训练侧的调度策略搬到推理侧，会让在线服务不够敏捷；
推理侧的成本约束反推到训练侧，可能压缩实验空间；
训练侧追求大 batch，推理侧却要处理低延迟和混合请求。

成熟团队需要把两侧连接起来，但不能把它们混成一个问题。

## 10. 集群组织能力会反过来塑造模型路线

模型不是脱离基础设施独立存在的。
如果一种结构难以并行、难以部署、难以稳定服务，它的现实价值就会被削弱。

集群能力会反向影响：

- 模型规模选择
- 精度策略
- MoE 与稠密模型取舍
- 上下文长度设计
- 是否使用量化或蒸馏
- 是否做小模型 / 大模型路由
- 端侧、边缘、云侧如何分工

这就是为什么模型团队和基础设施团队不能只是上下游关系。
它们更像共同设计关系。

模型是否现实可行，很大程度上取决于它能否被训练集群制造出来，又能否被推理集群长期交付出去。

如果只追训练阶段能力，可能得到一个线上成本过高、延迟过差的模型。
如果只追推理成本，可能过早限制模型表达能力。
真正成熟的路线，是在训练可行性、推理可服务性和产品价值之间做系统折中。

---

## 11. 工程现实：两类集群的痛点像两套不同字典

现实里，训练集群常见痛点包括：

- 通信瓶颈
- checkpoint 过慢
- 故障恢复成本高
- 扩展效率下降
- 数据管线跟不上
- 大作业调度困难
- 资源碎片导致利用率下降

推理集群更常见的痛点包括：

- KV cache 占用过高
- 长短请求互相拖累
- TTFT 和尾延迟抖动
- 高峰流量击穿
- continuous batching 策略不稳
- 多模型混部导致资源碎片
- 单 token 成本压不下来

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/109_training_clusters_and_inference_clusters_deep_revised__mermaid_08.png)

> 图109-8 两类集群的痛点不是同一本字典
> 很多团队会自然地从“训练做得不错”推到“服务应该也问题不大”。一拆痛点就会发现，两类集群面对的是不同的系统病症。

这两套痛点放在一起看，就能理解为什么“训练能力强”不自动等于“服务能力强”。
训练集群解决的是长作业制造能力。
推理集群解决的是在线交付能力。
它们都重要，但不是同一种工程。

## 12. 容易误判的地方

### 误解一：训练和推理都是“多卡跑模型”

这太粗糙。
两者面对的目标函数、优化策略、失败形态和运维制度都不同。

### 误解二：训练集群越强，推理问题自然不大

训练强只能说明制造能力强。
上线后，推理才是持续成本、服务质量和用户体验的中心。

### 误解三：并行方式只是基础设施细节

并行策略会直接影响模型是否能训练、是否能服务、是否能在成本线上长期成立。

### 误解四：推理只是训练完成后的部署环节

推理是完整在线系统。
它有自己的缓存、队列、服务等级、扩缩容、降级、观测和成本治理问题。

## 13. 本章小结

1. 训练集群和推理集群服务于两套不同目标，而不是同一问题的两个规模版本。
2. 训练更重总吞吐、扩展效率、长时间稳定、checkpoint 和实验推进。
3. 推理更重首 token 延迟、并发、KV cache、尾延迟、服务等级和单位成本。
4. 同一模型、同一硬件，在训练和推理两侧会被不同目标重新组织。
5. 集群组织能力会反过来塑造模型路线，并自然引向下一章的大模型推理瓶颈。

核心判断：

> 同样是跑模型，训练集群是在制造能力，推理集群是在交付能力；前者决定模型能不能被造出来，后者决定模型能不能被长期、稳定、划算地服务出去。
