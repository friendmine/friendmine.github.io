# 第103章 模型服务化

## 先说结论

决定一家公司能不能把模型变成生意的，往往不是模型本身，而是它有没有把模型改造成可承诺、可调度、可隔离、可观测、可计费的服务能力。

服务层一旦成熟，它还会反过来塑造模型路线本身：
什么模型值得保留，什么能力值得卖，什么延迟和价格会被定义成产品标准。

![插图 第103章 配额、优先级、隔离中控图]({{ site.baseurl }}/ai_ecosystem_book/images/chapter_103_model_serving.png)

> 插图 第103章 配额、优先级、隔离中控图

第102章讨论推理优化，解决的是模型怎样跑得快、跑得起。
本章讨论服务化，解决的是这份能力怎样被多人、多系统、多业务长期稳定调用。

---

## 1. “能推理”和“能服务”之间，隔着完整的平台工程

本地把模型跑通，是一道门槛。
但只要系统进入产品，问题就会立刻改变。

你不再只面对一次模型调用，而要面对：

- 大量并发请求
- 不同租户和不同优先级
- 长短请求混合
- 峰值流量冲击
- 成本和配额控制
- 版本兼容
- 故障恢复
- 安全与审计
- 上层业务的稳定接入

这时，系统要回答的已经不是“模型会不会推理”，而是：
这套模型能力能不能被组织成长期可承诺的在线服务。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/103_model_serving_deep_revised__mermaid_01.png)

> 图103-1 模型服务化是一条责任链
> 用户接入的是 API，但平台要兑现的是完整责任链：谁能进来、被分到什么资源、请求落到哪种模型、异常时怎么处理、指标如何被持续观察。

服务化不是“部署完成”的同义词。
它是 AI 能力开始被平台化管理的起点。

一个强模型放在弱平台里，最后交付出来的往往不是强服务，而是不稳定、不可预测、不可承诺的调用体验。

---

## 2. API 是入口，不是服务化的全部骨架

很多团队一谈模型服务化，就先想到 API。
这个直觉没错。
API 的确是模型能力进入平台世界的第一道门。

API 负责把能力标准化：

- 如何鉴权
- 如何提交请求
- 如何返回结果
- 如何表达错误
- 如何兼容版本
- 如何让上层系统稳定接入

但如果把服务化理解成“暴露一个接口”，那就太浅了。

平台还需要回答：

- 请求进入后如何排队
- 不同用户如何限额
- 不同模型如何路由
- 高峰时谁优先
- 失败时如何降级
- 版本如何灰度
- 成本如何计量
- 指标如何观测

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/103_model_serving_deep_revised__mermaid_02.png)

> 图103-2 API 为什么只是服务化入口
> 很多团队会把“已经有接口”误当成“已经服务化”。API 只解决接入，在线能力还需要被完整的平台骨架接住。

API 像门面。
真正决定服务能否长期成立的，是门面背后的认证、调度、隔离、容量、版本、观测和运营体系。

---

## 3. SLA 和 SLO 把模型能力变成可承诺能力

模型能力在 Demo 里可以被描述成“效果很好”。
服务能力在生产里必须被描述成“在什么边界内可以稳定做到”。

这就是 SLA 和 SLO 的意义。

SLA 可以理解成对外服务等级承诺。
它回答客户或上层业务可以期待什么，例如可用率、响应时间、故障恢复时间和版本变更规则。

SLO 更偏内部目标。
它帮助团队把服务承诺拆成可追踪、可运营的工程指标。

对模型服务来说，常见承诺可能包括：

- 可用率
- 首 token 延迟
- P95 / P99 延迟
- 每秒 token 吞吐
- 错误率
- 限流和降级规则
- 版本兼容窗口
- 故障恢复时间

SLA 的价值，在于迫使平台从“通常能用”升级成“在约定边界内持续能用”。

一旦进入企业和生产场景，用户买的就不只是模型能力，而是这份能力的可预期性。
没有可预期性，再强的模型也很难成为核心业务依赖。

---

## 4. 多租户和调度，把模型能力重新塑造成平台能力

真实平台很少只服务一个用户。
一旦进入多租户环境，问题就不再只是模型够不够强，而是这套强能力如何在共享资源上被公平、稳定、可预测地分配。

多租户的难点包括：

- 不同租户服务等级不同
- 不同请求资源消耗不同
- 高价值请求和低价值请求竞争资源
- 长上下文请求挤占显存和 KV cache
- 某类任务可能拖垮整个队列
- 峰值流量可能击穿容量边界

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/103_model_serving_deep_revised__mermaid_03.png)

> 图103-3 多租户调度怎样把模型能力变成平台能力
> 一旦进入多租户环境，平台卖的就不再只是模型本身，而是它能不能被稳定、公平、可预测地分配给不同用户和不同任务。

调度系统要处理：

- 队列优先级
- 并发上限
- batch 组织
- 模型路由
- cache 分配
- 抢占和超时
- 高峰期降级
- 不同租户之间的隔离

一个强模型如果放在糟糕调度里，也会迅速变成糟糕服务。
用户感受到的不是模型论文指标，而是排队、延迟、失败率、降级和稳定性。

---

## 5. 限流、配额、隔离和容灾，是服务化底线

AI 请求通常比普通 Web 请求更重，也更不稳定。
一次长上下文、多工具、多轮推理请求，可能占用远高于普通接口调用的资源。

因此，平台必须具备四类底线能力。

容量规划回答：平台大致能承受什么级别的流量、上下文长度、模型组合和请求结构。

限流和配额回答：高峰来临时谁可以进，谁需要等，谁不能无限占用共享资源。

隔离回答：局部问题能不能被困在局部，而不是拖垮整个平台。

容灾回答：节点、机房、模型版本或工具链出问题时，系统是否有退路。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/103_model_serving_deep_revised__mermaid_04.png)

> 图103-4 容量、限流、隔离与 SLA 的关系图
> 服务化不是几个孤立组件的拼装，而是一条连续能力链。没有容量规划，限流会变成临时救火；没有隔离，SLA 很容易被少数异常流量拖垮。

这些能力不是高级选项。
它们是服务化的底线。

没有限流，局部突发会变成全局事故。
没有隔离，一个租户或一类任务可能拖垮所有人。
没有容灾，平台只是在赌一切永远正常。

---

## 6. 从“能被调用”到“能被承诺”，中间是资源分配制度

很多团队一开始会误以为自己已经有在线服务。
一旦真实流量进来，才发现自己拥有的只是“能被调用的模型”，还不是“能被长期承诺的服务能力”。

真正的服务化，会把技术机制变成资源分配制度：

- 谁能获得更高配额
- 谁能使用更强模型
- 谁能享受更低延迟
- 谁在高峰期被优先保障
- 谁只能接受降级路径
- 哪些请求值得保留长上下文
- 哪些请求应该路由到便宜模型

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/103_model_serving_deep_revised__mermaid_05.png)

> 图103-5 从“能被调用”到“能被承诺”
> 服务化困难的地方，不在首次能接住请求，而在于高峰、故障和多租户冲突出现时，平台还能不能继续兑现稳定服务。

服务化说到底，不只是把模型放在线上。
它是把模型能力放进一套明确的资源分配机制里。

当平台进入商业环境，这套机制会直接变成价值排序：
高价值客户、关键流程、低风险后台任务、探索性请求，不可能永远使用同一条服务路径。

成熟平台必须知道自己在保护什么、牺牲什么、承诺什么。

---

## 7. 版本、灰度和兼容性决定服务能否安全演进

模型服务不是一次部署后永久不变。
模型会升级，prompt 会改，工具会变，路由策略会调整，安全策略会收紧。

每一次变化都可能影响线上业务。

服务化平台必须具备：

- 模型版本管理
- API 版本管理
- prompt 和上下文策略版本管理
- 灰度发布
- A/B 测试
- shadow traffic
- 快速回滚
- 兼容性窗口
- 变更审计

模型服务的版本风险比普通接口更复杂。
同一个 API 可能没有变，但模型行为变了。
工具调用成功率可能没有明显下降，但答案质量变了。
成本可能变了，尾延迟可能变了，某类租户体验可能变了。

因此，版本治理不是第104章才需要的线上观测问题。
它在服务化阶段就必须进入平台设计。
没有灰度和回滚，平台就只能在真实用户身上冒险发布。

---

## 8. 服务层会反过来重塑模型选择和产品设计

并不是能力更强的模型就更适合服务化。
当并发、延迟、成本、稳定性和服务等级同时成为约束，很多模型选择都会被重新评估。

现实中常见做法包括：

- 大小模型分层路由
- 高价值请求使用强模型
- 低风险请求使用便宜模型
- 常见问题用缓存或模板化结果
- 长上下文能力按服务等级开放
- 工具链调用按风险和成本分级
- 多模态或 Agent 能力只在明确场景中开放

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/103_model_serving_deep_revised__mermaid_06.png)

> 图103-6 服务化如何反过来重塑模型和产品
> 服务层不是后面的运维细节。它会反向决定用什么模型、卖什么能力、给谁提供什么等级的服务，因此直接进入产品和商业决策核心。

服务层成熟的标志，往往不是“所有请求都走最强模型”，而是开始明确分层。

普通请求走便宜、稳定、快速的路径。
高价值请求才分配成本更高的模型、更长上下文或更重工具链。
系统过载时，还要知道先保护哪类服务，牺牲哪类体验。

服务化把模型能力重新翻译成资源和承诺问题。

---

## 9. 计费、成本和容量规划，让服务化进入商业现实

模型服务一旦进入真实流量，就会持续消耗资源。
成本不再是一次性训练预算，而是每个请求、每个 token、每个高峰时段不断发生的运营支出。

平台需要知道：

- 每类请求成本是多少
- 每个租户消耗多少资源
- 哪些模型路径最贵
- 长上下文和工具调用如何计费
- 缓存命中是否真的降低成本
- 峰值容量需要多少冗余
- 哪些服务等级值得保留

没有成本核算，服务化很容易变成“越成功越亏钱”。
请求量越大，亏损越快。
模型越强，单位成本越难忽视。

容量规划也不能只看平均流量。
服务系统要面对峰值、突发、地区差异、活动流量、批量任务和异常重试。
如果容量规划只看平稳日均，很容易在真实高峰下失守。

所以，服务化不是纯技术平台。
它也是成本平台和商业平台。

---

## 10. 工程现实：很多产品不是卡在模型不行，而是卡在平台能力不成立

现实里，常见服务化问题包括：

- 峰值流量打满资源
- 某类租户挤占全部吞吐
- 长请求拖慢短请求
- 版本切换带来兼容问题
- 局部故障导致整片服务回退
- 成本核算不清，无法定价
- 降级策略缺失，高峰期全线变差
- 缺少观测，问题发生后很难定位

这些问题和模型本身的“智力”关系并不直接，却足以决定产品能否长期活下去。

很多团队前期能靠优秀模型吃到红利。
但后期拉开差距的，往往不是谁的模型再强一点，而是谁能把服务做成可信赖的公共能力。

用户愿不愿意把核心流程接进来，看的不是一次好看的输出，而是平台在高峰、故障、升级、跨租户冲突时还能不能稳住。

模型服务化一旦走到这一步，竞争单位就已经从“模型质量”转向“平台可靠性和运营能力”。

## 11. 容易误判的地方

### 误解一：模型服务化就是部署一个在线接口

接口只是入口。
困难的是调度、隔离、限流、容灾、版本、SLA、成本和观测。

### 误解二：只要推理够快，服务化自然成立

速度重要，但稳定性、可运营性、资源治理和服务承诺同样不可缺。

### 误解三：服务层只是基础设施团队的工作，与模型和产品无关

服务层会直接反过来重塑模型策略、产品边界、商业定价和客户承诺。

### 误解四：所有请求都应该使用最强模型

服务化要求分层供给。
不同价值、风险、时延和成本约束的请求，应该走不同服务路径。

## 延伸阅读建议

- 推荐起点：服务治理、SLA/SLO、capacity planning、多租户隔离、灰度发布、降级设计，比“模型更强”更接近平台现实。

- 检索关键词：`SLA`、`SLO`、`multi-tenant serving`、`rate limit`、`quota`、`graceful degradation`、`capacity planning`、`canary release`、`shadow traffic`。

## 12. 本章小结

1. 模型服务化要把推理能力组织成可调度、可隔离、可承诺、可观测、可计费的在线能力系统。
2. API 是入口，但多租户调度、限流、隔离、容灾、版本治理和 SLA/SLO 才是服务化骨架。
3. 服务层会反过来塑造模型选择、产品设计、服务等级和商业策略。
4. 成熟平台不是让所有请求都用最强模型，而是按价值、风险、时延和成本分层供给。
5. 服务化之后，下一步核心问题就是第104章要讨论的线上观测与回归。

核心判断：

> 模型一旦进入服务化竞争，较量的就不再只是效果，而是谁能把能力变成长期可承诺、可治理、可计费、可持续演进的责任系统。
