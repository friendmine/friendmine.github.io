# 第86章 Agent 的工作模式

## 先说结论

Agent 的工作模式没有唯一正确答案。  
真正成熟的系统设计，不是把结构堆到最复杂，而是让行动方式与任务复杂度、风险和组织成本彼此匹配。

## 1. “要不要做 Agent”其实是一个过于粗糙的问题

很多讨论会把产品简单分成两类：有 Agent，或者没有 Agent。  
这个分法太粗，因为它掩盖了最重要的工程判断：系统究竟应当以什么方式行动。

同样叫 Agent，有的只是单体执行器，有的是被工作流包裹的局部智能，有的是带监督者的分层系统，还有的是多智能体协作。  
这些模式在成本、稳定性、解释性和治理要求上差异极大。

因此，真正的问题不是“我们要不要 Agent”，而是“这件事需要哪一种行动结构”。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/86_agent_working_modes_deep_revised__mermaid_01.png)

> 图86-1 Agent 工作模式对照图  
> 不同模式并不是按“先进程度”排队，而是在不同任务结构、风险水平和组织条件下，各自解决不同问题。

## 2. 单 Agent 模式的价值，在于把复杂度压到最低

单 Agent 不是落后方案。  
恰恰相反，它往往是最值得优先尝试的起点。

当任务边界比较清楚、工具数量不多、步骤链路有限时，一个单 Agent 就足以承担主要工作。  
它的优势非常现实：

- 路径更短
- 状态更集中
- 调试更容易
- 成本更可控

很多团队一上来就嫌单 Agent 不够“先进”，于是急着堆多角色、长链路和反思回环。  
但在高并发、强约束的现实系统里，结构每复杂一层，状态同步、调试、复盘和成本都会同步上涨。  
如果一个任务已经能用单 Agent 稳定闭环，那么继续加角色、加通信、加回环，往往不是能力升级，而是系统税升级。

如果把这些模式再和任务类型对应起来，会更容易做判断：

- `单 Agent` 更适合步骤链较短、工具数有限、错误可快速修复的任务
- `Workflow + Agent` 更适合流程骨架稳定、但某些节点需要灵活判断的任务
- `Supervisor` 更适合任务可拆分、阶段需要统一收敛和调度的场景
- `多 Agent` 更适合角色天然不同、且确实能并行分工的大任务
- `人在环` 更适合高代价动作、目标变化频繁、价值判断不能外包的任务

真正成熟的选型不是问“哪种最像未来”，而是问“哪种最符合当前任务的失误成本和组织负担”。

很多任务之所以不该一上来就上多 Agent，不是因为多 Agent 没有想象力，而是因为角色一多，通信和协调本身就会变成新问题。  
如果任务边界还不清楚、责任链还没拆稳、失败定位能力也没建立，多 Agent 往往只是把混乱复制成更多份。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/86_agent_working_modes_deep_revised__mermaid_02.png)

> 图86-2 单 Agent 适用条件图  
> 单 Agent 的价值，不是“最朴素”，而是它能在很多实际任务里，用最小系统税换来足够好的闭环。先把简单结构做到稳，往往比一开始就堆复杂角色更成熟。

## 3. Workflow + Agent 往往比“纯自由 Agent”更适合生产环境

不是所有任务都适合完全开放式决策。  
很多现实工作流程其实已经有稳定骨架，例如审批、生成、校验、提交、归档。  
在这种情况下，让 Agent 在所有环节自由发挥，并不一定是优势。

更常见也更有效的做法是：  
把稳定部分写成 workflow，把需要灵活判断的局部交给 Agent。

这种模式的关键好处在于，它把确定性和适应性分开了。  
流程负责托底，Agent 负责在具体节点做智能判断。  
这样一来，系统既不会完全僵硬，也不会因为过度自由而失控。

现实里很多生产系统最后都会停在这个平衡点。  
不是因为团队不懂更“先进”的模式，而是因为 workflow 能把责任边界、可观测性和复盘结构都固定住。  
真正高价值的灵活性，往往只存在于少数判断节点，而不是整条链都需要自由探索。  
谁能看清这一点，谁就更不容易把系统设计成一台昂贵而不可预测的自动装置。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/86_agent_working_modes_deep_revised__mermaid_03.png)

> 图86-3 Workflow + Agent 的典型组织方式  
> 真正适合生产环境的，常常不是“让 Agent 到处自由探索”，而是先把能稳定写死的部分固定下来，再把真正需要灵活性的节点交给智能层。

---

## 4. Supervisor 模式出现，是因为复杂任务需要显式调度

当任务被拆成多个子问题，或者系统里已经存在多个执行单元时，往往会出现一个更高层的需求：谁来分配任务，谁来汇总结果，谁来判断何时终止或重试。

这就是 supervisor 模式的由来。  
它相当于在 Agent 之上再加一层调度视角。

这类结构尤其适合：

- 子任务可拆分
- 结果需要整合
- 过程需要统一监控

但它的代价同样明显。  
一旦多出一层监督者，你就要额外处理层间通信、状态同步和责任边界。  
如果没有清晰收益，这层结构很容易从“调度器”变成“新的混乱源”。

很多 supervisor 失败，并不是因为“中心不够聪明”，而是因为中心层既想理解全局、又想处理细节，最后反而变成新的瓶颈。  
所以 supervisor 真正有价值的前提，是它只做必须集中处理的事：分配、汇总、终止、重试和冲突裁决。  
如果它开始过度侵入每个子任务细节，多一层调度通常只会多一层拖慢。

---

## 5. 多 Agent 不是更先进，而是更难管理

多 Agent 最容易让人产生未来感。  
不同角色并行工作、互相评论、彼此协作，听起来确实很强。  
但在工程上，它首先意味着更高的系统税。

多 Agent 带来的不只是并行，还包括：

- 更多通信开销
- 更复杂的状态一致性问题
- 更长的错误传播链
- 更困难的回放和调试

所以，多 Agent 并不天然更高级。  
只有在任务真的天然可拆，并且不同角色拥有明确边界时，它才值得。  
否则，所谓“多智能体协作”常常只是把单一系统的混乱复制成多份。

也正因为误解太多，多智能体协作值得被单独拉成下一章。  
它不是“工作模式总图”里的一个小分支，而是一个需要单独讨论的系统选择题。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/86_agent_working_modes_deep_revised__mermaid_04.png)

> 图86-4 多 Agent 的系统税  
> 多 Agent 的真正成本，不只是多几个模型调用，而是通信、同步、定位和复盘难度全面上升。只有当分工收益显著大于这些额外成本时，它才值得被引入。

---

## 6. 人在环，本身就是一种非常重要的工作模式

很多技术叙事喜欢把人在环讲成“自动化还不够彻底”的过渡状态。  
这个理解太轻。  
对于高价值、高风险、目标经常变化的任务来说，人在环不是过渡方案，而是成熟方案的一部分。

人在环的意义，不只是审批，更包括：

- 确认目标是否成立
- 在关键节点重新设定优先级
- 对高代价动作做授权
- 在系统偏航时及时接管

从系统设计上看，这不是“人帮机器补缺陷”，而是“把人的判断力放在最该出现的位置上”。

---

## 7. 真正的模式选择，取决于三件事

如果要给 Agent 工作模式建立一个实用判断框架，可以先看三件事：

- 任务结构是否稳定
- 错误代价是否高
- 问题是否天然可拆

任务稳定，就优先 workflow。  
错误代价高，就必须加控制和人在环。  
天然可拆，才值得考虑 supervisor 或多 Agent。  
这比从概念流行度出发做架构决策要靠谱得多。

把这个判断再压成一句更实用的话：  
如果单 Agent 加清晰 workflow 已经能把任务做稳，就没有必要为了“更先进”再引入多 Agent；  
因为一旦角色变多，通信、状态同步、责任归因和调试成本都会立刻上升。  
多 Agent 真正值得上的时刻，通常不是“听起来更智能”，而是任务本身已经天然分裂成多个必须并行、且彼此有清晰边界的子问题。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/86_agent_working_modes_deep_revised__mermaid_05.png)

> 图86-5 工作模式选择的最小判断框架

## 8. 工程现实：很多失败都来自“过度设计”

现实里，一些团队会在系统还没有证明基本闭环有效之前，就急着上多角色、多轮辩论、多层监督。  
结果往往是：

- 成本先涨上去
- 成功率并没有同步提升
- 问题更难解释
- 维护负担更重

这说明 Agent 模式设计的核心，不是追求复杂，而是压住无谓复杂度。  
系统只有在确实被复杂任务逼到那一步时，才值得演化成更复杂的形态。

从业者最需要抵抗的诱惑，就是把“看起来更像未来”错当成“更适合现在”。  
很多团队在最该做闭环验证的时候，先做了角色设计；  
在最该做失败恢复的时候，先做了多 Agent 讨论；  
在最该补控制层的时候，先扩张了行动空间。  
模式选型一旦脱离任务现实，就会让系统税先上涨，而真正的可交付性反而后退。

---

## 9. 容易误判的地方

### 误解一：多 Agent 一定比单 Agent 更强

没有清晰分工的多 Agent，通常只会更脆弱。

### 误解二：Workflow 不够智能，所以只是临时方案

恰恰相反，很多生产级系统正是靠 workflow 获得稳定性。

### 误解三：人在环代表系统还不成熟

在高风险任务里，人在环往往就是成熟系统的必要组成。

---

## 延伸阅读建议

- 推荐起点：single-agent、workflow + agent、supervisor、multi-agent 这些模式最好对照着看，而不是单独追某个框架。
- 继续延伸：如果你关心选型，建议继续看 orchestration cost、handoff design、supervisor bottleneck、human escalation。
- 检索关键词：`single agent`、`supervisor agent`、`multi-agent orchestration`、`handoff design`、`human escalation`。

---

## 11. 本章小结

1. Agent 设计不是“有没有”的问题，而是“采用哪一种行动结构”的问题。  
2. 单 Agent、workflow、supervisor、多 Agent 和人在环，分别适合不同任务条件。  
3. 模式选择要看任务结构、风险水平和可拆分性，而不是看哪种说法更先进。  
4. 复杂模式只有在确实解决问题时才有价值。  

核心判断：

> 最好的 Agent 工作模式，不是最花哨的那一种，而是最符合任务现实、能把复杂度压在必要范围内的那一种。
