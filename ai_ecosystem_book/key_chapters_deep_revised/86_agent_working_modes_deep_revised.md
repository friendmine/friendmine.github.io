# 第86章 Agent 的工作模式

## 先说结论

Agent 的工作模式没有唯一正确答案。

成熟的系统设计，不是把结构堆到最复杂，而是让行动方式与任务复杂度、风险水平、可观测性和组织成本彼此匹配。

单 Agent、Workflow + Agent、Supervisor、多 Agent、人在环，并不是一条从低级到高级的直线。它们是不同的行动结构，适合不同任务条件。

## 1. “要不要做 Agent”是一个过于粗糙的问题

很多讨论会把产品简单分成两类：有 Agent，或者没有 Agent。

这个分法太粗，因为它掩盖了更关键的工程判断：系统究竟应该以什么方式行动。

第85章已经讲过，一个能长期推进任务的 Agent 不是“一个更会说话的模型”，而是一套包含目标、状态、规划、工具、反馈和控制的小型工作系统。

既然 Agent 已经是系统对象，接下来就不能只问“模型够不够强”，而要问：

- 谁负责决定下一步；
- 哪些步骤可以固定成流程；
- 哪些节点需要模型判断；
- 哪些动作必须被审批；
- 出错时如何暂停、回滚和接管；
- 多个执行单元之间如何交接状态。

这些问题共同决定了 Agent 的工作模式。

同样叫 Agent，有的只是单体执行器；有的是被 workflow 包裹的局部智能；有的是带 supervisor 的分层系统；有的是多智能体协作；还有的是把人类判断明确嵌入流程的可接管系统。

它们在稳定性、成本、解释性、权限治理、调试难度和组织接受度上差异很大。

因此，更成熟的问题不是“我们要不要 Agent”，而是“这件事需要哪一种行动结构”。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/86_agent_working_modes_deep_revised__mermaid_01.png)

> 图86-1 Agent 工作模式对照图
> 不同模式并不是按“先进程度”排队，而是在不同任务结构、风险水平和组织条件下，各自解决不同问题。

## 2. 单 Agent 模式的价值，是把复杂度压到最低

单 Agent 不是落后方案。

恰恰相反，它往往是最值得优先尝试的起点。

当任务边界清楚、工具数量有限、步骤链路不长、错误可以快速修复时，一个单 Agent 就足以承担主要工作。

它的优势非常现实：

- 路径短；
- 状态集中；
- 调试容易；
- 成本可控；
- 失败定位更直接；
- 人类接管时更容易看懂当前状态。

很多团队一开始就嫌单 Agent 不够“先进”，急着堆多角色、多轮反思、多层监督和复杂编排。问题是，系统结构每复杂一层，状态同步、日志追踪、权限划分、失败复盘和运维成本都会同步上涨。

如果一个任务已经能用单 Agent 稳定闭环，继续加角色、加通信、加反思回环，往往不是能力升级，而是系统税升级。

单 Agent 特别适合三类任务。

第一类是短链任务。

例如整理一份文档、生成一个草稿、查询并汇总一组信息、修改一个范围明确的小功能。任务虽可能需要工具，但不需要复杂角色分工。

第二类是低风险任务。

即使输出不完美，也能很快人工修正，且不会直接影响生产、财务、法律或客户承诺。

第三类是探索性任务的早期阶段。

当团队还不确定目标、数据、工具和失败模式时，用单 Agent 先跑出闭环，比一开始设计复杂架构更稳。

单 Agent 的问题也很清楚。

一旦任务变长、状态变多、工具变多、风险变高，它会开始暴露上下文拥挤、计划漂移、错误累积和难以自检的问题。

但正确的升级顺序通常不是立刻变成多 Agent，而是先问：能否用 workflow 把稳定部分固定下来。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/86_agent_working_modes_deep_revised__mermaid_02.png)

> 图86-2 单 Agent 适用条件图
> 单 Agent 的价值，不是“最朴素”，而是它能在很多实际任务里，用最小系统税换来足够好的闭环。先把简单结构做到稳，往往比一开始就堆复杂角色更成熟。

## 3. Workflow + Agent 是生产环境里的常见稳态

不是每类任务都适合开放式决策。

很多现实流程本来就有稳定骨架，例如收集、生成、校验、审批、提交、归档。让 Agent 在每个环节都自由发挥，未必会更好，反而可能让系统难以解释和复盘。

更常见也更有效的做法是：

把确定部分写成 workflow，把需要灵活判断的局部交给 Agent。

这种模式的关键，是把确定性和适应性分开。

workflow 负责：

- 固定流程顺序；
- 管理输入输出；
- 记录状态变化；
- 设置权限边界；
- 处理重试、超时和失败路径；
- 保证关键节点可以观察和复盘。

Agent 负责：

- 在局部节点理解上下文；
- 生成候选方案；
- 判断非结构化输入；
- 调用合适工具；
- 对模糊任务做有限范围内的适应。

这种结构比“纯自由 Agent”更适合生产环境。

因为生产系统需要的不只是聪明，还需要可预期、可解释、可回滚、可审计。

举例来说，一个客服自动化系统未必应该让 Agent 自由决定整条流程。更稳的做法是：工单分类、风险分级、权限检查、发送渠道和归档由 workflow 固定；具体回复草稿、知识库检索、相似案例归纳交给 Agent；涉及退款、合同承诺或敏感投诉时再升级给人。

这不是保守，而是把智能放在真正需要智能的位置上。

现实里很多成熟系统最后都会停在这个平衡点。

不是因为团队不懂更复杂的模式，而是因为 workflow 能把责任边界、状态管理和失败恢复固定住，而 Agent 只在少数高价值节点提供弹性。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/86_agent_working_modes_deep_revised__mermaid_03.png)

> 图86-3 Workflow + Agent 的典型组织方式
> 适合生产环境的，常常不是“让 Agent 到处自由探索”，而是先把能稳定写死的部分固定下来，再把需要灵活性的节点交给智能层。

## 4. Supervisor 模式解决的是显式调度问题

当任务被拆成多个阶段，或者系统里已经存在多个执行单元时，往往会出现一个更高层的问题：谁来分配任务，谁来汇总结果，谁来判断何时重试、终止或升级。

这就是 supervisor 模式的由来。

它相当于在执行单元之上加一层调度视角。

Supervisor 主要适合以下场景：

- 子任务可以拆开，但仍需要统一收敛；
- 多个工具或多个执行器需要被调度；
- 过程需要全局监控和阶段验收；
- 冲突、失败和重试需要统一裁决；
- 人工接管需要从一个清晰入口进入系统。

好的 supervisor 不应该把所有事情都重新做一遍。

它应该负责分配、汇总、终止、重试、升级和冲突裁决，而不是陷入每个子任务的细节。

很多 supervisor 失败，并不是因为中心不够聪明，而是因为中心层既想看全局，又想处理细节，最后变成新的瓶颈。

它会不断打断执行者，不断改写计划，不断解释中间状态，却没有真正提高交付质量。

因此，Supervisor 模式成立的关键是克制。

它必须知道哪些判断应该集中，哪些执行应该下放。

如果任务只是短链、低风险、无须调度，用 supervisor 反而会让系统变慢。如果任务已经需要多角色协作、统一状态、冲突裁决和可接管入口，supervisor 才开始有价值。

## 5. 多 Agent 不是更先进，而是更难管理

多 Agent 容易让人产生未来感。

多个角色并行工作、互相评论、彼此协作，听起来确实很强。

但在工程上，它首先意味着更高的系统税。

多 Agent 带来的不只是并行，还包括：

- 更多通信开销；
- 更复杂的状态一致性问题；
- 更长的错误传播链；
- 更困难的回放和调试；
- 更复杂的权限治理；
- 更高的评测难度；
- 更不容易界定的责任边界。

所以，多 Agent 并不自动更高级。

只有在任务确实自然可拆，并且不同角色拥有明确边界时，它才值得。

否则，所谓“多智能体协作”常常只是把单一系统的混乱复制成多份。

一个实用判断是：

如果你无法写清楚每个 Agent 的输入、输出、权限、成功标准、失败处理和交接对象，就还不应该做多 Agent。

也正因为误解太多，多智能体协作值得被单独拉成下一章。

它不是工作模式总图里的一个小分支，而是一个需要单独讨论的系统选择题。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/86_agent_working_modes_deep_revised__mermaid_04.png)

> 图86-4 多 Agent 的系统税
> 多 Agent 的成本，不只是多几个模型调用，而是通信、同步、定位和复盘难度全面上升。只有当分工收益显著大于这些额外成本时，它才值得被引入。

## 6. 人在环本身就是一种工作模式

很多技术叙事喜欢把人在环讲成“自动化还不够完整”的过渡状态。

这个理解太轻。

对于高价值、高风险、目标经常变化的任务来说，人在环不是过渡方案，而是成熟系统的一部分。

人在环的意义，不只是审批。

它至少包括：

- 确认目标是否仍然成立；
- 在关键节点重新设定优先级；
- 对高代价动作做授权；
- 对关键输出做复核；
- 在系统偏航时及时接管；
- 对最终交付承担验收责任。

从系统设计上看，这不是“人帮机器补缺陷”，而是“把人的判断力放在最该出现的位置上”。

尤其是在以下场景里，人在环几乎不能被省略：

- 任务涉及法律、金融、医疗、安全或公共传播后果；
- 动作不可逆或补救成本很高；
- 目标本身可能变化；
- 需要在多个价值之间取舍；
- 系统执行结果会影响组织责任和外部信任。

成熟的自动化，不是让人完全消失，而是让人在关键处可见、可控、可接管。

这一点会在第88章继续展开。

## 7. 模式选择取决于五个变量

如果要给 Agent 工作模式建立一个实用判断框架，可以先看五个变量。

第一，任务结构是否稳定。

结构稳定，优先 workflow；结构不稳定，才需要更多 Agent 判断。

第二，链路长度是否可控。

短链任务可以先用单 Agent；长链任务要考虑状态管理、阶段验收和失败恢复。

第三，错误代价是否高。

错误代价越高，越需要权限控制、审计、审批、回滚和人在环。

第四，任务是否自然可拆。

只有自然可拆，才值得考虑 supervisor 或多 Agent。为了追求形式而拆分，通常会制造通信噪声。

第五，组织是否能承受系统税。

复杂模式不只消耗 token，还消耗调试能力、监控能力、评测能力、治理能力和人类注意力。

把这些变量合起来，可以得到一个简单判断：

- 任务短、低风险、工具少：优先单 Agent；
- 流程稳定、局部需要判断：优先 Workflow + Agent；
- 任务可拆、需要统一调度：考虑 Supervisor；
- 角色自然不同、可以并行且可移交：才考虑多 Agent；
- 动作高风险、不可逆或涉及价值判断：必须设计人在环。

这比从概念流行度出发做架构决策要可靠得多。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/86_agent_working_modes_deep_revised__mermaid_05.png)

> 图86-5 工作模式选择的最小判断框架
> 这张图的作用，是把“选型”从流行概念拉回工程条件。成熟系统通常不是先问哪种更先进，而是先问任务结构、代价和可拆分性到底长什么样。

## 8. 模式不是一次性选择，而是演化路径

很多团队把架构选择想成一次性决定：一开始就选好单 Agent、workflow、supervisor 或多 Agent，然后长期沿用。

现实通常不是这样。

更健康的演化路径往往是：

先用单 Agent 验证任务闭环；

再把稳定步骤固化成 workflow；

当任务被证明确实需要统一调度时，引入 supervisor；

当角色边界、状态交接和并行收益足够清楚时，再考虑多 Agent；

当动作开始影响真实世界和组织责任时，补上人在环。

这个顺序的价值，是让复杂度跟着证据增长。

每一次升级都应该回答一个问题：新结构解决了什么已经被观察到的真实问题？

如果答案只是“看起来更先进”，那就不是升级，而是过度设计。

模式演化也可以反向发生。

当系统发现多 Agent 并没有带来收益，应该退回 supervisor 或 workflow；当某个 Agent 节点行为已经足够稳定，也可以把它固化成传统流程或普通服务；当某个自动化动作风险过高，也应该把它改成人工审批或人工验收。

成熟系统不是越跑越复杂，而是根据证据不断把复杂度放回合适位置。

---

## 9. 工程现实：很多失败来自过度设计

现实里，一些团队会在系统还没有证明基本闭环有效之前，就急着上多角色、多轮辩论、多层监督。

结果往往是：

- 成本先涨上去；
- 成功率没有同步提升；
- 问题更难解释；
- 日志更难看懂；
- 责任更难追溯；
- 维护负担更重。

这说明 Agent 模式设计的核心，不是追求复杂，而是压住无谓复杂度。

系统只有在确实被复杂任务逼到那一步时，才值得演化成更复杂的形态。

从业者最需要抵抗的诱惑，是把“看起来更像未来”错当成“更适合现在”。

很多团队在最该做闭环验证的时候，先做了角色设计；

在最该做失败恢复的时候，先做了多 Agent 讨论；

在最该补控制层的时候，先扩张了行动空间。

模式选型一旦脱离任务现实，就会让系统税先上涨，而可交付性反而后退。

## 10. 如何评估一个工作模式是否合适

选型不是写在架构图上就结束了。

一个 Agent 工作模式是否合适，需要用运行结果证明。

至少要看七类指标。

第一，完成率。

系统是否能稳定完成目标，而不是只在演示样例里成功。

第二，失败可解释性。

失败时能否看出是目标理解、工具调用、状态管理、权限、模型判断还是流程设计出了问题。

第三，接管成本。

人类需要介入时，能否快速理解当前状态、已执行动作、剩余风险和下一步选择。

第四，单位成本。

包括 token、工具调用、外部 API、人工复核、延迟和运维成本。

第五，质量增益。

复杂模式是否真的带来了更高准确率、更少遗漏或更强鲁棒性。

第六，治理能力。

权限、审计、日志、审批、回滚和异常升级是否足够清楚。

第七，与更简单模式的比较。

复杂模式必须和认真设计过的简单模式比较，而不是和一个随便搭的基线比较。

如果多 Agent 只比一个粗糙单 Agent 好，却不比一个清晰 workflow 好，它就未必值得。

真正成熟的选型，必须经得起这种对照。

## 11. 容易误判的地方

### 误解一：多 Agent 比单 Agent 更强

不一定。

没有清晰分工和稳定状态交接的多 Agent，通常只会更脆弱。

### 误解二：Workflow 不够智能，所以只是临时方案

恰恰相反。

很多生产级系统正是靠 workflow 获得稳定性，再把少数需要判断的节点交给 Agent。

### 误解三：Supervisor 越强，系统越稳

不一定。

Supervisor 如果过度介入细节，会变成新的瓶颈和混乱源。

### 误解四：人在环代表系统还不成熟

在高风险任务里，人在环往往就是成熟系统的必要组成。

### 误解五：模式越复杂，越能体现技术水平

工程水平不体现在结构复杂，而体现在能否用足够简单的结构稳定完成任务，并在必要时才引入额外复杂度。

- 推荐起点：single-agent、workflow + agent、supervisor、multi-agent、human-in-the-loop 这些模式建议对照着看，而不是单独追某个框架。
- 继续延伸：如果关心选型，建议继续看 orchestration cost、handoff design、supervisor bottleneck、human escalation、workflow automation。
- 检索关键词：`single agent`、`workflow automation`、`supervisor agent`、`multi-agent orchestration`、`handoff design`、`human escalation`、`orchestration cost`。

## 12. 本章小结

1. Agent 设计不是“有没有”的问题，而是“采用哪一种行动结构”的问题。
2. 单 Agent、Workflow + Agent、Supervisor、多 Agent 和人在环，分别适合不同任务条件。
3. 单 Agent 的价值是低系统税；Workflow + Agent 的价值是把确定性和适应性分开；Supervisor 的价值是显式调度；多 Agent 的价值是多角色分工；人在环的价值是责任、审批和接管。
4. 模式选择要看任务结构、链路长度、错误代价、可拆分性和组织能否承受系统税。
5. 复杂模式只有在确实解决已观察到的问题时才有价值。
6. 一个模式是否合适，必须和更简单的基线比较，而不是只看架构图是否漂亮。

核心判断：

> 好的 Agent 工作模式，不是最花哨的那一种，而是符合任务现实、能把复杂度压在必要范围内，并且让系统在失败时仍然可解释、可接管、可修复的那一种。
