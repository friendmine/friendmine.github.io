# 第83章 检索、压缩与上下文装配

## 先说结论

上下文工程的核心难题，不是“有没有信息”，而是怎样在有限窗口和有限成本里，把最该看的信息以最合适的形态送进模型。
检索、压缩和装配的质量，往往直接决定了模型后面能否把已有知识真正用起来。

很多系统的失败并不是“完全没查到”，而是“查到了，但重排错了、摘要歪了、装配顺序乱了”，最后让真正关键的信息没能以任务可消费的形式进入这一轮上下文。

---

## 1. 有知识不等于模型能用上知识，因为窗口永远是有限的

系统也许拥有海量资料：

- 知识库
- 文档库
- 历史记录
- 用户记忆

但模型当前能看到的上下文始终有限。  
因此，真正的问题不是“有没有资料”，而是：

- 现在该取哪部分
- 取回来后怎么压缩
- 以什么顺序放进去
- 哪些信息该常驻，哪些该动态进入

这就是上下文装配真正开始的地方。

很多团队恰恰会在这里产生错觉：知识库已经很全，为什么系统还是不稳。  
原因通常不是“知识不够多”，而是“知识没有以任务可消费的形式进入这一轮窗口”。  
拥有材料和让模型用好材料，是两件完全不同的事。  
一旦窗口有限、成本有限、任务还在推进，信息供给就会从存储问题变成调度问题。

> 图83-1 检索、压缩与上下文装配流程图  
> 真正有效的上下文供给，不是“检索到就结束”，而是从召回、重排、压缩、裁剪一路走到最终装配，最后形成模型此刻真正能使用的任务包。

---

## 2. RAG 真正解决的，不是“外挂一个知识库”，而是构建一条按需知识供给链

RAG 的价值经常被说得过浅。  
它不是简单给模型外挂知识库，而是在建立一条知识供给机制：

- 当前问题来了
- 系统先判断需要什么材料
- 再从外部库中按需取回
- 最后把材料以可用形式交给模型

这样，模型就不必把所有知识都背进参数里。  
它开始可以在外部知识和内部推理之间形成分工。

但这条供给链一旦做得粗糙，系统也会迅速露出另一面：  
明明“查到了”，结果还是答不对。  
因为很多失败并不是知识库没有答案，而是供给链在中间某一段失真了。  
召回拿错材料、重排把“有点像”的排在最前、摘要把关键约束压没、装配顺序又让状态信息被噪声覆盖，最后用户只会看到同一个结果：系统像是看过资料，却没真正看懂。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/83_retrieval_compression_and_context_assembly_deep_revised__mermaid_02.png)

> 图83-2 RAG 作为知识供给链  
> RAG 不是“有库就行”，而是把外部知识组织成一条可被当前任务消费的供给链。真正重要的不是拥有多少材料，而是材料能否以对的形式进入模型这一轮工作。

---

## 3. chunking、embedding 检索和 rerank，其实是在共同回答“怎样先粗找，再精挑”

一条典型检索链路之所以不只是“搜一下”，是因为它往往包含不同阶段：

- chunking 把大文档切成可检索单元
- embedding 检索快速找出相关候选
- rerank 再在候选中重新排序

这三步分别在解决不同层次的问题：  
先让文档可被切进系统，再让搜索足够快，最后让结果尽量相关。  
少了其中任意一层，系统都很容易出现“找到了，但找得不对”或“根本找不到”的问题。

如果只保留一个最小直觉式，可以把重排理解成：

`排序分数 ≈ 语义相关性 + 任务匹配度 + 新鲜度 / 权重修正`

这个式子不是严格算法定义，而是帮你抓住工程直觉：  
一个片段值不值得被排到前面，不只取决于“像不像你问的东西”，还取决于它是否真的匹配当前任务、是否足够新、是否带有更高优先级。  
很多检索系统的问题，不是完全找不到，而是把“有点像”排在了“真正该看”的前面。

---

## 4. 压缩并不是附属工作，而是知识进入窗口前的第二次建模

即使检索到了很多相关材料，也不可能全部原样塞进窗口。  
于是，摘要、压缩、状态打包、关键信息提炼就变得不可避免。

这一步非常容易被低估。  
很多团队会把切片（Chunking）和压缩理解成纯粹的字符裁剪，但在结构复杂、依赖紧密的材料里，这种处理很容易直接切断关键关系：  
压缩和切片从来不只是“把文字变短”。  
它本质上是在重新组织知识进入模型的方式。  
如果切片把因果、约束和上下文关系一起切断，模型就不可能从残缺材料里自动恢复出原来的结构。  
结果往往不是“信息少一点”，而是“信息已经变形”，后续判断自然也会跟着偏掉。

这也是为什么很多 RAG 系统的失败非常“委屈”。  
它们不是完全没找对，而是“差一点点”。  
而在生产里，差一点点通常就已经足以让后续行动跑偏。  
所以压缩和摘要最需要警惕的，不是字数不够短，而是把真正决定任务成败的结构信息一起压掉了。

这里要把几类动作的角色差异讲清：

- `召回` 解决的是“先把可能有用的东西找出来”
- `重排` 解决的是“在候选里谁更该靠前”
- `摘要` 解决的是“把长材料压成更适合当前任务的短材料”
- `裁剪` 解决的是“在窗口有限时，最终哪些信息能留下来”

它们听起来都在“处理信息”，但职责并不相同。  
如果把这些动作混成一步，系统往往会在某个环节偷懒：  
要么召回太粗，后面没有好材料可选；  
要么摘要太猛，把关键约束一起压掉；  
要么裁剪太晚，结果让真正重要的信息最后没位置了。

> 这四步看上去都在“处理信息”，但其实分别对应不同问题。把它们拆开，读者才会真正理解为什么很多 RAG 系统明明“搜到了”，却还是不好用。

## 5. state packing 的价值，在于让模型看到的不只是知识片段，还有执行现场

对执行型系统来说，上下文里不仅有知识，还有任务状态和环境状态。  
如果这些状态组织混乱，模型即使拿到了知识，也不知道自己现在做到哪一步。

state packing 的意义就在这里：  
把执行现场打包成一个可读、可继续、可交接的上下文对象。  
它让模型看到的不是散碎素材，而是“任务此刻的结构化现场”。

这件事对于 Agent 尤其关键，因为 Agent 的难点从来不只是知道什么，而是知道当前处于哪一轮、下一步该接哪一段、哪些动作已经发生过。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/83_retrieval_compression_and_context_assembly_deep_revised__mermaid_04.png)

> 图83-4 state packing 把“资料”变成“任务包”  
> 只有把知识、目标、状态和约束一起打包，模型看到的才是可继续推进的现场，而不是一堆彼此没有位置关系的片段。

---

## 6. context window 再大，也不会消灭上下文工程

窗口变大当然有帮助，但问题不会因此自然消失。  
因为：

- 成本仍然会上升
- 噪声会跟着一起进入
- 重点信息仍需排序
- 动态状态仍需更新

所以更大的窗口不是上下文工程的终结，而是把问题带进更高维度。  
你依旧必须决定什么更重要、什么该压缩、什么该后置、什么必须被固定保留。

窗口一大，另一个问题反而更明显：噪声也会被一并放大。  
如果没有优先级和状态装配意识，系统并不会因为“都放进去了”就自动变聪明。  
它只会在更长、更贵、更多互相竞争的信息上继续推理。  
所以大窗口带来的从来不是“上下文工程结束”，而是“粗糙装配的代价更贵了”。

## 7. 工程里最常见的失败，不是完全没找到，而是“找到了但没被正确送进去”
现实系统中，检索和压缩最常见的失败不是完全空手而回，而是：

- 找到了，但排序不对
- 放进去了，但太长太乱
- 摘要丢了关键约束
- 状态信息被知识噪声淹没

这说明上下文装配本身就是系统能力，而不是简单中间件。  
它决定模型手里拿到的，到底是一份可工作的任务包，还是一堆彼此打架的材料。

## 8. 容易误判的地方

### 误解一：RAG 做好了，问题基本就解决了

检索只是第一步，后面的压缩和装配同样关键。

### 误解二：摘要就是把内容缩短

真正好的摘要要保留任务最相关的结构信息，而不是只做字数裁剪。

### 误解三：窗口大了，检索就不重要了

大窗口不等于高质量信息选择。

## 10. 本章小结

1. 知识丰富不等于模型能真正用上知识，关键在于供给与装配。  
2. RAG 的本质是一条按需知识供给链。  
3. chunking、embedding 检索与 rerank 共同构成“先粗找、再精挑”的流程。  
4. 压缩是知识进入窗口前的第二次建模。  
5. state packing 让模型看到的不只是资料，还有任务现场。  

核心判断：

> 真正决定模型能不能用好知识的，不是你有多少资料，而是你能不能把最该看的那一点在对的时候、以对的形态，摆到它眼前。
