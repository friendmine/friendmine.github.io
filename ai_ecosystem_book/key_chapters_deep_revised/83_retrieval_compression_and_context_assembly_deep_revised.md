# 第83章 检索、压缩与上下文装配

## 先说结论

上下文工程的难题，不是“有没有信息”，而是怎样在有限窗口和有限成本里，把最该看的信息以合适的形态送进模型。

检索、重排、压缩、裁剪、状态打包和最终装配，决定了模型此刻拿到的是一份可工作的任务包，还是一堆彼此打架的材料。

很多系统的失败并不是“没有查到”，而是“查到了，但重排错了、摘要歪了、装配顺序乱了、状态信息被噪声淹没了”，最后让关键信息没能以任务可消费的形式进入这一轮上下文。

---

## 1. 有知识不等于模型能用上知识

第82章讨论了短期记忆、长期记忆和工作记忆。

记忆解决的是“系统能否保存和取回信息”，但取回之后还有一个更难的问题：这些信息能否以正确形态进入当前上下文。

系统也许拥有海量资料：

- 知识库；
- 文档库；
- 历史记录；
- 用户记忆；
- 项目状态；
- 工具返回结果；
- 组织规则和外部事实。

但模型当前能看到的上下文始终有限。

即使上下文窗口变大，成本、注意力、排序、噪声和任务相关性仍然有限。

因此，问题不是“有没有资料”，而是：

- 现在该取哪部分；
- 取回来后怎样重排；
- 哪些内容应该压缩；
- 哪些信息必须原样保留；
- 以什么顺序放进去；
- 哪些信息该常驻，哪些该动态进入；
- 哪些过期信息必须排除。

这就是上下文装配开始的地方。

很多团队会在这里产生错觉：知识库已经很全，为什么系统还是不稳。

原因通常不是知识不够多，而是知识没有以任务可消费的形式进入这一轮窗口。

拥有材料和让模型用好材料，是两件不同的事。

一旦窗口有限、成本有限、任务还在推进，信息供给就会从存储问题变成调度问题。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/83_retrieval_compression_and_context_assembly_deep_revised__mermaid_01.png)

> 图83-1 检索、压缩与上下文装配流程图
> 有效的上下文供给，不是“检索到就结束”，而是从召回、重排、压缩、裁剪一路走到最后装配，最后形成模型此刻能使用的任务包。

## 2. RAG 不是外挂知识库，而是按需知识供给链

RAG 的价值经常被说得过浅。

它不是简单给模型外挂一个知识库，而是在建立一条按需知识供给链。

这条链通常包含：

- 理解当前任务需要什么材料；
- 改写或扩展查询；
- 从外部库中召回候选；
- 对候选进行重排；
- 对长材料做压缩和摘要；
- 把证据、状态和约束装配成上下文；
- 让模型基于这份上下文生成、判断或行动。

这样，模型就不必把所有知识都背进参数里。

它开始可以在外部知识和内部推理之间形成分工。

但这条供给链一旦粗糙，系统也会迅速露出另一面：明明查到了，结果还是答不对。

因为很多失败并不是知识库没有答案，而是供给链在中间某一段失真了。

召回拿错材料，重排把“有点像”的排在最前，摘要把关键约束压没，裁剪又把证据来源删掉，最后装配顺序让旧状态覆盖新状态。

用户看到的只有一个结果：系统像是看过资料，却没有看懂。

所以 RAG 不能只按“有没有接知识库”来判断成熟度。

更关键的是这条供给链是否能稳定回答三个问题：

- 该取什么；
- 该怎样压缩；
- 该怎样放进当前任务现场。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/83_retrieval_compression_and_context_assembly_deep_revised__mermaid_02.png)

> 图83-2 RAG 作为知识供给链
> RAG 不是“有库就行”，而是把外部知识组织成一条可被当前任务消费的供给链。重要的不是拥有多少材料，而是材料能否以对的形式进入模型这一轮工作。

## 3. 召回、重排和查询改写，要分开理解

一条典型检索链路之所以不只是“搜一下”，是因为它往往包含多个不同阶段。

第一步是 chunking。

大文档需要被切成可检索单元。但切片不是简单按字数切开。切片要尽量保留标题、段落、代码块、表格、因果链、前后约束和引用来源。

如果 chunk 切断了定义与例子、结论与证据、接口与调用方式，后续检索再强也只能拿到残缺材料。

第二步是查询改写。

用户的问题往往不是最适合检索的查询。

系统需要把自然语言请求改写成更接近资料结构的查询，有时还要扩展同义词、缩写、项目代号、实体名称和时间范围。

第三步是召回。

召回要尽量把可能有用的材料取回来。它追求的是不漏掉关键候选，而不是立刻精确排序。

第四步是重排。

重排负责在候选里挑出真正适合当前任务的材料。

这里看的不只是语义相似度，还包括任务匹配度、新鲜度、权威性、证据密度、来源可信度和当前状态相关性。

如果只保留一个工程直觉，可以把重排理解成：

`排序分数 ≈ 语义相关性 + 任务匹配度 + 来源可信度 + 新鲜度 + 状态相关性 - 噪声惩罚`

这个式子不是严格算法定义，而是提醒我们：一个片段值不值得被排到前面，不只取决于“像不像你问的东西”，还取决于它是不是当前任务真正该看的材料。

很多检索系统的问题，不是找不到，而是把“有点像”排在了“该看”的前面。

## 4. 压缩不是附属工作，而是第二次建模

即使检索到了很多相关材料，也不可能全部原样塞进窗口。

于是，摘要、压缩、状态打包和关键信息提炼就变得不可避免。

这一步非常容易被低估。

很多团队把压缩理解成纯粹的字数裁剪，但在结构复杂、依赖紧密的材料里，这种处理很容易切断关键关系。

压缩和切片不只是“把文字变短”。

它是在重新组织知识进入模型的方式。

压缩时最怕丢掉的，不一定是事实点本身，而是事实之间的结构关系：

- 因果关系；
- 例外条件；
- 适用范围；
- 时间顺序；
- 版本差异；
- 证据来源；
- 约束与后果；
- 结论的置信度。

如果压缩把这些关系一起压掉，模型就很难从残缺材料里自动恢复原来的结构。

结果往往不是“信息少一点”，而是“信息已经变形”。

后续判断自然也会跟着偏掉。

所以压缩最需要警惕的，不是字数不够短，而是把决定任务成败的结构信息一起压掉。

高质量压缩应当尽量保留：

- 当前任务直接需要的结论；
- 支撑结论的关键证据；
- 不确定和冲突之处；
- 不能突破的约束；
- 来源和时间；
- 下游使用这段信息时必须知道的前提。

这就是为什么可以把压缩看成知识进入窗口前的第二次建模。

它不是清理文本，而是在重新设计模型将要看到的世界。

---

## 5. 召回、重排、摘要、裁剪的职责不能混在一起

上下文工程里最常见的错误，是把所有信息处理都混成一步。

这会让系统在失败时很难定位问题。

四个动作必须分清。

`召回`解决的是：先把可能有用的东西找出来。

它追求覆盖率，宁可多拿一点候选，也不要漏掉关键材料。

`重排`解决的是：候选里谁更该靠前。

它追求任务相关性和证据质量，避免“语义有点像”的材料抢占窗口。

`摘要`解决的是：把长材料压成更适合当前任务的短材料。

它追求保留结构，不只是减少字数。

`裁剪`解决的是：窗口有限时，哪些信息能留下来。

它追求优先级控制，必须知道哪些内容可以删，哪些内容不能删。

这四步看起来都在处理信息，但对应的是完全不同的问题。

如果把它们混成一步，系统很容易在某个环节偷懒：

- 召回太粗，后面没有好材料可选；
- 重排太弱，噪声排在证据前面；
- 摘要太猛，关键约束被压掉；
- 裁剪太晚，重要状态最后没位置；
- 装配太乱，新旧信息彼此冲突。

很多 RAG 系统明明“搜到了”却不好用，根源就在这里。

> 这四步看上去都在“处理信息”，但其实分别对应不同问题。把它们拆开，读者才会理解为什么很多 RAG 系统明明“搜到了”，却还是不好用。

## 6. State packing 让模型看到任务现场

对执行型系统来说，上下文里不仅有知识，还有任务状态和环境状态。

如果这些状态组织混乱，模型即使拿到了知识，也不知道自己现在做到哪一步。

State packing 的意义就在这里：

把执行现场打包成一个可读、可继续、可交接的上下文对象。

它让模型看到的不是散碎素材，而是任务此刻的结构化现场。

一个好的 state pack 通常至少包含：

- 当前目标；
- 完成标准；
- 已完成步骤；
- 当前阻塞点；
- 已验证事实；
- 未验证假设；
- 关键约束；
- 可用工具；
- 已发生的外部动作；
- 下一步候选；
- 需要人工确认的风险点。

这件事对于 Agent 尤其关键。

Agent 的难点不只是知道什么，而是知道当前处于哪一轮、下一步该接哪一段、哪些动作已经发生过、哪些内容不能重复执行。

如果上下文里只有知识片段，没有任务现场，模型很容易出现两类问题。

第一，重复做已经完成的事。

第二，跳过应该验证的前置条件。

所以 state packing 不是文案整理，而是执行连续性的基础设施。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/83_retrieval_compression_and_context_assembly_deep_revised__mermaid_04.png)

> 图83-4 state packing 把“资料”变成“任务包”
> 只有把知识、目标、状态和约束一起打包，模型看到的才是可继续推进的现场，而不是一堆彼此没有位置关系的片段。

## 7. 装配顺序本身就是模型行为的一部分

很多人以为只要信息都放进窗口，顺序就不重要。

这是误解。

上下文窗口不是普通仓库。

模型看到什么、先看到什么、哪些内容更靠近任务指令、哪些内容被噪声包围，都会影响后续输出。

因此，context assembly 至少要处理五类信息的相对位置。

第一，系统规则和不可突破约束。

这类信息应该稳定、明确，并且不能被普通检索内容覆盖。

第二，当前任务目标和完成标准。

模型必须先知道这次要完成什么，否则检索材料越多，越可能分散注意力。

第三，当前状态。

包括已完成动作、正在处理的阶段、阻塞点和下一步候选。

第四，相关知识和证据。

它们应该围绕当前任务组织，而不是按数据库返回顺序堆进去。

第五，输出要求。

包括格式、引用方式、风险提示、是否需要保留不确定性。

装配不是简单拼接，而是在安排模型当前这一轮工作的注意力路线。

如果状态放得太靠后，模型可能忽略当前进度。

如果旧记忆放得太强，模型可能让过期信息覆盖新事实。

如果证据来源被裁掉，模型可能给出看似确定但无法追溯的结论。

如果约束和候选材料混在一起，模型可能把普通材料误当成规则。

所以，装配顺序本身就是模型行为的一部分。

## 8. Context window 再大，也不会消灭上下文工程

窗口变大当然有帮助。

但问题不会因此自然消失。

因为：

- 成本仍然会上升；
- 噪声会跟着一起进入；
- 重点信息仍需排序；
- 动态状态仍需更新；
- 冲突信息仍需裁决；
- 旧信息仍可能污染新判断。

更大的窗口不是上下文工程的终结，而是把问题带进更高维度。

你依旧需要决定什么更重要、什么该压缩、什么该后置、什么需要固定保留，什么必须被排除。

窗口一大，另一个问题反而更明显：噪声也会被一并放大。

如果没有优先级和状态装配意识，系统并不会因为“都放进去了”就自动变聪明。

它只会在更长、更贵、更多互相竞争的信息上继续推理。

所以大窗口带来的不是“上下文工程结束”，而是“粗糙装配的代价更贵了”。

## 9. 常见失败不是没找到，而是没正确送进去

现实系统中，检索和压缩常见的失败不是空手而回，而是：

- 找到了，但排序不对；
- 找到了，但来源不可信；
- 放进去了，但太长太乱；
- 摘要丢了关键约束；
- 裁剪删掉了证据出处；
- 状态信息被知识噪声淹没；
- 旧记忆覆盖了新事实；
- 冲突材料没有被标记；
- 模型拿到的是片段，不是任务现场。

这说明上下文装配本身就是系统能力，而不是简单中间件。

它决定模型手里拿到的，到底是一份可工作的任务包，还是一堆彼此打架的材料。

如果装配失败，后面再调模型、调提示词、调温度，都只是在错误输入上做补丁。

![Mermaid 图]({{ site.baseurl }}/ai_ecosystem_book/images/mermaid_png/83_retrieval_compression_and_context_assembly_deep_revised__mermaid_05.png)

> 图83-5 上下文装配的典型失败路径
> 很多失败不是“系统没有知识”，而是知识进来的顺序、密度和结构不对。理解这一点，才能把问题定位到上下文工程，而不是盲目归咎于模型本身。

## 10. 上下文工程也需要评测

很多团队只评测最终回答，却不评测上下文供给链。

这会导致一个问题：系统错了以后，团队不知道错在召回、重排、压缩、裁剪、装配，还是模型推理。

上下文工程至少要评测六件事。

第一，召回覆盖率。

关键材料是否进入候选集。如果候选里根本没有答案，后面所有步骤都无力补救。

第二，重排质量。

最该看的材料是否排在前面，噪声和弱相关材料是否被压下去。

第三，压缩保真度。

摘要是否保留了关键约束、证据、例外条件和不确定性。

第四，装配有效性。

最终上下文是否让模型容易完成当前任务，而不是只把材料堆在一起。

第五，引用和可追溯性。

模型给出的结论能否追溯到上下文中的证据来源。

第六，状态一致性。

当前任务进度、旧记忆、新证据和工具结果之间是否互相冲突。

这些评测最好不要只在最终答案上做。

更好的方式是保留每一轮供给链 trace：查询、召回候选、重排结果、压缩版本、最终装配包、模型输出。

这样系统失败时，团队才能定位到底是哪一段出了问题。

上下文工程没有 trace，就很难从经验变成工程。

## 11. 从上下文装配走向 Agent

本章看似在讲 RAG 和上下文窗口，实际上是在为下一章 Agent 铺地基。

因为一旦系统不只是回答，而是开始持续行动，上下文装配就会变得更关键。

Agent 每一轮都要根据当前上下文决定下一步动作。

如果上下文里目标不清、状态不明、旧信息污染新事实、证据缺失、约束被压掉，那么 Agent 不只是会答错，还会继续执行错。

也就是说，在回答系统里，上下文装配影响输出质量。

在 Agent 系统里，上下文装配会影响行动链安全。

这就是第84章要继续展开的问题：当模型从“回答”走向“行动”，上下文不再只是知识输入，而会变成下一步动作的地面。

---

## 12. 容易误判的地方

### 误解一：RAG 做好了，问题基本就解决了

不够。

检索只是第一步，后面的重排、压缩、裁剪、状态打包和装配同样关键。

### 误解二：摘要就是把内容缩短

不是。

好的摘要要保留任务相关的结构信息、约束、证据来源和不确定性，而不是只做字数裁剪。

### 误解三：窗口大了，检索就不重要了

大窗口不等于高质量信息选择。

窗口越大，粗糙装配的成本和噪声反而可能越高。

### 误解四：检索结果相关，就一定适合当前任务

不一定。

语义相关不等于任务相关。当前状态、时间、新鲜度、来源可信度和完成标准都会改变材料优先级。

### 误解五：上下文装配只是工程胶水

不是。

它直接决定模型这一轮“看见的世界”是什么，因此会影响判断、生成和后续行动。

## 13. 本章小结

1. 知识丰富不等于模型能用上知识，关键在于供给与装配。
2. RAG 可以理解为一条按需知识供给链，而不是外挂一个知识库。
3. Chunking、查询改写、召回与 rerank 共同构成“先粗找、再精挑”的流程。
4. 压缩是知识进入窗口前的第二次建模，必须保留结构、约束、证据和不确定性。
5. 召回、重排、摘要、裁剪职责不同，不能混成一个动作。
6. State packing 让模型看到的不只是资料，还有任务现场。
7. 装配顺序本身会影响模型行为，不能把上下文当成普通材料仓库。
8. 上下文工程需要单独评测，否则系统失败时无法定位是哪一段出了问题。
9. 当系统走向 Agent，上下文装配会从输出质量问题升级为行动安全问题。

核心判断：

> 决定模型能不能用好知识的，不是你有多少资料，而是你能不能把最该看的那一点，在对的时候、以对的形态、按对的顺序，摆到它眼前。
